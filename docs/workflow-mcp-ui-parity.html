Workflow Step Lifecycle — MCP / UI Parity Audit

Side-by-side comparison of every step lifecycle operation across MCP tools and UI server actions. Updated 2026-03-20.

🔍
Purpose: This document compares exactly what data each code path writes so you can verify MCP and UI produce identical outcomes. Intentional gaps (like MCP returning richer metadata) are noted separately from actionable gaps (where one path writes data the other doesn't).

State Transition Map

pending → in_progress MCP: claimNextStep UI: startWorkflowStep
in_progress → completed MCP: completeStep UI: completeWorkflowStep
in_progress → awaiting_approval MCP: completeStep UI: completeWorkflowStep (when human_check_required)
awaiting_approval → completed MCP: approveStep UI: approveWorkflowStep
in_progress → failed MCP: failStep UI: failWorkflowStep
awaiting_approval → failed MCP: failStep UI: failWorkflowStep (reject)
failed → pending UI: retryWorkflowStep (no MCP equivalent)
pending → skipped MCP: skipStep UI: skipWorkflowStep

1. Start / Claim Step AT PARITY

MCP: claimNextStep(task_id)

  • Step fields:
  • status = "in_progress"
  • started_at = now()
  • claimed_by = ctx.userId
  •  
  • Run update:
  • status = "running", current_step = step_order
  • Guard: .in("status", ["pending", "running", "failed"])
  •  
  • Comments: None
  •  
  • Concurrency: .eq("status", "pending").maybeSingle()
  •  
  • Extras:
  • Auto-selects next pending step by step_order
  • Returns rework_instructions, context, agents, instruction string

UI: startWorkflowStep(stepId)

  • Step fields:
  • status = "in_progress"
  • started_at = now()
  • claimed_by = user.id
  •  
  • Run update:
  • status = "running", current_step = step_order
  • Guard: .in("status", ["pending", "running", "failed"])
  •  
  • Comments: None
  •  
  • Concurrency: .eq("status", "pending").maybeSingle()
  •  
  • Extras:
  • Caller provides explicit stepId
  • Returns updated step row only
Data written: identical. The only difference is UX — MCP auto-selects the step and returns rich metadata (context chaining, rework instructions, agent list). The DB writes are the same.

2. Complete Step AT PARITY

MCP: completeStep(step_id, output?)

  • Step fields:
  • status = "completed" | "awaiting_approval"
  • completed_at = now() (only if completed)
  • output = value (only if provided)
  •  
  • Run update:
  • checkAndCompleteRun() if completed + run_id
  •  
  • Comments:
  • type = "output" if output provided + idea_id exists
  •  
  • Concurrency: .eq("status", "in_progress").maybeSingle()
  •  
  • Pre-fetch: human_check_required, idea_id, status

UI: completeWorkflowStep(stepId, output?)

  • Step fields:
  • status = "completed" | "awaiting_approval"
  • completed_at = now() (only if completed)
  • output = value (only if provided)
  •  
  • Run update:
  • checkAndCompleteRun() if completed + run_id
  •  
  • Comments:
  • type = "output" if output provided + idea_id exists
  •  
  • Concurrency: .eq("status", "in_progress").maybeSingle()
  •  
  • Pre-fetch: human_check_required, idea_id
Data written: identical. Both conditionally set output, create output comment, and check for run completion. MCP additionally returns run_complete boolean and an awaiting_approval stop message.

3. Fail Step AT PARITY

MCP: failStep(step_id, output?, reset_to_step_id?)

  • Step fields:
  • status = "failed"
  • completed_at = now()
  • output = value (if provided)
  •  
  • Run update:
  • "running" if cascade, "failed" if no cascade
  •  
  • Comments:
  • type = "failure" on failed step (if output provided)
  • type = "changes_requested" on cascade target (if stepsReset > 0)
  •  
  • Cascade reset (if reset_to_step_id):
  • All steps from target onward: status = "pending", output = null, started_at = null, completed_at = null, claimed_by = null
  •  
  • Concurrency: .in("status", ["in_progress", "awaiting_approval"]).maybeSingle()
  •  
  • Pre-fetch: id, run_id, step_order, idea_id

UI: failWorkflowStep(stepId, output?, resetToStepId?)

  • Step fields:
  • status = "failed"
  • completed_at = now()
  • output = value (if provided, else null)
  •  
  • Run update:
  • "running" if cascade, "failed" if no cascade
  •  
  • Comments:
  • type = "failure" on failed step (if output + idea_id)
  • type = "changes_requested" on cascade target (if targetStep found)
  •  
  • Cascade reset (if resetToStepId):
  • All steps from target onward: status = "pending", output = null, started_at = null, completed_at = null, claimed_by = null
  •  
  • Concurrency: .in("status", ["in_progress", "awaiting_approval"]).maybeSingle()
  •  
  • Pre-fetch: via update result .select("*")
Data written: functionally identical. Minor nuances:
  • MCP: changes_requested comment gated on stepsReset > 0; UI gated on targetStep found — equivalent in practice
  • MCP: failure comment uses step.idea_id (from pre-fetch); UI uses data.idea_id (from update result) — same value
  • MCP: output key absent from update if undefined; UI sets output: output ?? null explicitly — both produce same DB state

  • Comments: None
  •

UI: skipWorkflowStep(stepId, reason?)

  • Step fields:
  • status = "skipped"
  • completed_at = now()
  • output = reason ?? "Skipped -- not applicable to this task"
  •  
  • Run update:
  • checkAndCompleteRun() if run_id
  •  
  • Comments: None
  •  
  • Concurrency: .eq("status", "pending").maybeSingle()
Data written: identical. MCP has an extra app-level pre-check that the step is pending before the DB guard. Redundant but harmless.

5. Approve Step GAPS EXIST

MCP: approveStep(step_id, comment?)

  • Step fields:
  • status = "completed"
  • completed_at = now()
  • Does NOT update output
  •  
  • Run update:
  • checkAndCompleteRun() if run_id
  •  
  • Comments:
  • type = "approval" if comment provided + idea_id
  •  
  • Concurrency: App-level + .eq("status", "awaiting_approval").maybeSingle()
  •  
  • Extras:
  • Blocks bot callers (is_bot check)

UI: approveWorkflowStep(stepId, output?)

  • Step fields:
  • status = "completed"
  • completed_at = now()
  • output = value (if explicitly provided)
  •  
  • Run update:
  • checkAndCompleteRun() if run_id
  •  
  • Comments:
  • None
  •  
  • Concurrency: .eq("status", "awaiting_approval").maybeSingle()
  •  
  • Extras:
  • No bot caller guard
Gap MCP UI Severity
Approval comment Creates type = "approval" comment Does not create any comment Minor
Output handling Does NOT update step output Can overwrite step output if provided Minor
Bot guard Checks is_bot, rejects bots No is_bot check By design — UI is always a human

6. Retry Step GAPS EXIST

MCP: (no dedicated tool)

  • No standalone retry tool exists.
  •  
  • Retry-like behavior comes from the cascade in failStep which resets steps to pending, then claimNextStep picks them up.

UI: retryWorkflowStep(stepId)

  • Step fields:
  • status = "pending"
  • output = null
  • started_at = null
  • completed_at = null
  • claimed_by = null
  •  
  • Run update:
  • status = "running"
  • Guard: .eq("status", "failed")
  •  
  • Comments: None
  •  
  • Concurrency: .eq("status", "failed").single()
  • Note: uses .single() not .maybeSingle()
Gap MCP UI Severity
Dedicated retry tool Not available Available via retryWorkflowStep By design — agents use cascade
.single() vs .maybeSingle() N/A Uses .single() — throws different error type if row not found Minor — should be .maybeSingle() for consistency

7. Reset Workflow GAPS EXIST

MCP: resetWorkflow(task_id)

  • Step fields (all steps in run):
  • status = "pending"
  • output = null
  • started_at = null
  • completed_at = null
  • claimed_by = null
  •  
  • Run update:
  • status = "pending", current_step = null, completed_at = null
  •  
  • Run lookup: Finds active run via .not("status", "in", '("completed","failed")')

UI: resetWorkflow(runId)

  • Step fields (all steps in run):
  • status = "pending"
  • output = null
  • started_at = null
  • completed_at = null
  • claimed_by = null
  •  
  • Run update:
  • status = "pending", current_step = 0, completed_at = null
  •  
  • Run lookup: Caller provides explicit runId — no status guard
Gap MCP UI Severity
current_step reset value null 0 Minor — should align
Run status guard Only resets active (non-completed/failed) runs Resets any run by ID Minor — UI could reset a completed run

8. Remove Workflow AT PARITY

MCP: removeWorkflow(task_id)

  • Finds active run via status filter
  • Deletes workflow_runs row
  • Steps cascade via FK ON DELETE CASCADE

UI: removeWorkflow(runId)

  • Caller provides explicit runId
  • Deletes workflow_runs row
  • Steps cascade via FK ON DELETE CASCADE
Data written: identical. Same delete operation. MCP auto-finds the run; UI receives explicit ID. MCP only targets active runs; UI deletes any run (which is fine since user explicitly selected it).

Summary: Remaining Gaps

Operation Gap Impact Suggested Fix
Approve Step MCP creates type = "approval" comment; UI does not Minor Add approval comment creation to approveWorkflowStep server action
Approve Step MCP does not touch output; UI can overwrite it Minor Intentional — MCP preserves agent output, UI allows approver override. Document as by-design.
Retry Step Uses .single() instead of .maybeSingle() Minor Change to .maybeSingle() for consistent error handling
Reset Workflow current_step: MCP sets null, UI sets 0 Minor Align to null (more semantically correct for "no step")
Reset Workflow MCP guards against resetting completed/failed runs; UI does not By design UI user explicitly chose to reset — acceptable
Overall assessment: After the rework loop fix, MCP and UI are at parity for all data-writing operations (start, complete, fail, skip, remove). The remaining gaps are: 1 missing approval comment in the UI, 1 inconsistent .single() call, and 1 current_step value mismatch on reset. None affect the core workflow loop or rework detection.
