New Idea — concrete facts

  • 5 fields: Title, Project type, Description, Tags, GitHub URL, + Private switch
  • Description is REQUIRED (required on textarea)
  • Order: Title → Project type → Description → Tags → GitHub → Visibility
  • Enhance = full dialog: persona picker, prompt template, clarifying questions, streamed result, refine + undo, credit counter (BYOK/starter credits)
  • Visibility = Switch row labelled "Private idea" (off = public, implicit)
  • Submit: creates idea + applies kit → redirects to board. No AI board generation.
  • Container: shadcn Card, "Share Your Idea", max-w-2xl page

Onboarding — concrete facts

  • 3 fields: Project name, Description, Project type, + Visibility. No Tags, no GitHub URL.
  • Description is OPTIONAL ("optional — AI can help"); only 2 rows tall
  • Order: Name → Description → Project type → Visibility → Enhance → Submit
  • Enhance = one click, inline-replace: overwrites textarea, toast "Description enhanced!". No preview, no undo, no persona, no questions, no credits (badged "Free")
  • Visibility = segmented Public/Private pills, default Public, both shown
  • Submit ALSO auto-generates the board (AI tasks) → loading state → step 3 preview
  • Container: 580px Dialog, label "Project name" not "Title"

4 · Recommendation

Not chosen

(a) Share one component

Onboarding embeds <IdeaForm variant="onboarding"> (hides tags+GitHub, makes desc optional). Zero drift, but forces the heavy Card + full enhance dialog + credit UI into the 580px modal, and the modal needs board auto-generation the page form doesn't have. High refactor cost, fights the modal's leaner intent.

★ Recommended

(b) Harmonise patterns

Keep two surfaces, but unify the things that are pure drift: labels, visibility control, enhance UX, ordering. Cheap, low-risk, preserves the deliberately lean first-run modal and its auto-generate-board flow while killing the inconsistencies users actually notice.

Not chosen

(c) Keep leaner, minimal fixes

Only fix the very worst inconsistency. Leaves most drift in place — labels, enhance, visibility still diverge — so the next change re-opens the gap. Under-corrects.

Recommendation: (b) Harmonise patterns

Keep two forms, unify the patterns — don't force-share the component.

Why not (a). The two surfaces have genuinely different jobs. The page form is a deliberate, full-detail idea (required description, tags, GitHub, repeatable, no board generation). The onboarding step is a low-friction first-run form whose whole point is speed and a magic "Create & Generate Board" payoff. Cramming the real IdeaForm — Card chrome, the multi-phase enhance dialog with credits, tags, GitHub — into a 580px stepper modal would bloat first-run friction and still require an onboarding branch for board auto-generation. The shared component would be mostly if (variant === "onboarding") branches: the "never diverge" win is undercut by a component that's now two forms in a trench coat.

Why (b). Most of the differences in the map above are drift, not design. They can be removed without sharing JSX, by aligning four things:

  • Enhance UX is the big one. The page gives a previewable, undoable, persona-aware, credit-aware enhance; onboarding silently overwrites the textarea with a toast and no undo. That's a trust gap on a brand-new user's first AI interaction. Onboarding should adopt the preview + Undo affordance (it can stay single-click and free — just don't destroy the user's text irreversibly). Reuse EnhanceDialogShell in a lightweight mode, or at minimum keep an "Undo — restore original" link like the page form's handleUndo.
  • Visibility control: pick ONE pattern. The segmented Public/Private pills (onboarding) are clearer than the implicit "Private idea" Switch (off = public is non-obvious). Recommend standardising on the segmented pills in both.
  • Labels & copy: "Title" vs "Project name", "Share Your Idea" vs "What are you building?". Different microcopy is fine if intentional; the field label for the same concept should not differ. Align the name field label (pick "Project name") and the enhance button label.
  • Ordering: put Description immediately after the name in both (onboarding already does; the page interleaves Project type between them — acceptable since it feeds enhance context, so leave that one).

Concrete refactor. Extract the shared pieces, not the whole form: (1) a <VisibilitySelector> (segmented pills) used by both idea-form.tsx and onboarding-dialog.tsx; (2) a shared enhance entry that exposes preview+undo (lift the onboarding enhance onto EnhanceDialogShell like CreateEnhanceDialog already does, or give onboarding the same undo link). Drive name label and the kit-context strings from one constants source. No variant prop, no shared mega-component. Low effort, low risk, and the drift can't silently re-grow because the load-bearing controls now come from one place.

5 · Justified vs drift

✓ Should REMAIN different

  • Auto-generate board — onboarding's "Create & Generate Board" + loading/preview is the first-run magic moment; the page form deliberately doesn't.
  • Fewer fields — no Tags / no GitHub URL keeps first-run fast; both are easily added later from the idea page.
  • Optional, shorter description — first-run shouldn't gate on a written description, especially with enhance available.
  • Container — Card page vs stepper modal; different surfaces, correct.
  • Cancel affordance — "Cancel" vs "I'll do this later" matches each context.

✗ Pure drift — should GO

  • Two different enhance UXs — destructive overwrite vs preview+undo. Unify on preview+undo.
  • Two visibility controls — Switch vs pills. Standardise on segmented pills.
  • Name field label — "Title" vs "Project name" for the same field.
  • Credit visibility — page shows credit count, onboarding hides it. Pick one truth (onboarding's free enhance should still be honest about credits if it consumes them).
  • Enhance button label / styling — "Enhance with AI" pill vs full CTA card. Visual scale can differ; the wording shouldn't.
