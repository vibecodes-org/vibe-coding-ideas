Install-First Terminal — First-Run Flow

Redesign of the in-browser terminal dock so a first-timer with nothing installed is guided through a calm, one-time setup before any vibecodes:// deep link fires — never seeing the scary macOS “no application set” dialog. Returning users skip straight to connecting.

Designer: Compass (UX) Surface: board bottom terminal dock Replaces: auto-fire deep link on “In the browser” Palette matched to terminal-dock.tsx Mac-first · Windows deferred

§1The problem we are fixing

Built on the Requirements step (ProdOwner) and its 12 acceptance criteria — those criteria are the spec for every screen below (see the mapping table in §9). Today the board button “Launch Claude Code → In the browser” opens the dock and immediately fires the deep link. A first-timer with no helper installed gets macOS’s alarming dialog first, then a backwards “Nothing opened. Install the helper” fallback.

✕ Today (broken order)

Open dock → deep link fires blind → macOS “no application set / Search App Store” → confusing fallback. The user is punished for not having a thing they were never told to install.

⚠️
There is no application set to open the URL vibecodes://launch…

Search the App Store for an application that can open this document, or choose an existing application on your computer.

The dialog acceptance criterion #1 forbids for first-timers. We prevent it by never firing the link until the user has explicitly set up + pressed Connect.

✓ New (install-first order)

Open dock → calm numbered one-time setup (no link fired) → user downloads & installs → presses Connect → link fires once, pre-explained → connecting → success. The paired flag then lets returning users skip the setup entirely.

Core inference model
  • The web page cannot detect if the native helper is installed.
  • We infer readiness from an explicit action (pressing Connect) + a localStorage “paired-before” flag set on first success.
  • Unpaired → show setup. Paired → auto-connect. That single flag is the whole branch.

§2Flow & state map

Two entry points (unpaired vs paired) converge on one connect→result loop. Retry never re-shows the setup wall.

The deep link fires at exactly one edge — the Connect press on the unpaired path, and the auto-connect on the paired path (both are the user opening a session on purpose). It never fires on dock open for an unpaired browser (criterion #2).

Interaction & why
  • No link on open. Nothing fires until the user presses Connect in step 3 — directly satisfies criteria #1 and #2. The user is never ambushed by the macOS dialog.
  • Gestalt numbering. The 3 numbered badges + hairline dividers group the setup into an obvious ordered sequence (recognition over recall). Progressive disclosure keeps it to 3 choices, not a wizard (Hick’s Law; no gold-plated onboarding per Requirements).
  • Information scent. Button reads “Download for Mac (Apple Silicon)” and “Connect” — never “Submit”/“Download”. Arch is detected so the file always matches the machine.
  • Pre-warning near the trigger (Fitts). The “Open VibeCodes?” explanation sits immediately above Connect — satisfies criterion #4 so the OS prompt is never a surprise.
  • Recovery for the already-installed user: “Already installed it? Just click Connect.” lets a partway user skip straight to step 3.
  • WCAG: Connect is a full-width ≥44px target; all steps carry text labels + numerals, never colour alone; emerald on near-black passes 4.5:1.

Interaction & why
  • ~8s calm timeout (criterion #7). Header pill reads “Not connected yet” — provisional, not a dead end. No red, no alarm icon.
  • Forbidden words absent across both variants — verified copy: no “Nothing opened”, “Failed”, or “Error”. Tone is “not yet”, always with a next action.
  • Two clear actions, right precedence. Retry is the primary (sky, most likely fix); Download / Re-install the helper is the secondary. Both ≥44px, side by side, both labelled (no colour-only).
  • Retry loops to Connecting (§4), never back to the setup wall — a returning user is never demoted to “set up from scratch”.
  • Returning-user variant (criterion #4b): paired flag is set but the helper didn’t answer → “Couldn’t reach the helper on this Mac … re-install”. Same calm pattern, machine-aware wording, plus reassurance that local work is safe.

Interaction & why
  • No setup wall — the branch is purely the paired-before flag; a paired browser lands on Connecting, satisfying criterion #6.
  • The deep link fires here too, because auto-connect is the returning user deliberately reopening their session — still an explicit intent, not a blind dock-open fire.

  • Graceful degradation: if it doesn’t attach in ~8s, it falls through to the returning-user fallback in §6 — never back to setup.

§11Design-system & accessibility notes

Reuse & consistency
  • Colours, radii, and the header/body split are lifted from terminal-dock.tsx (#141417 header, #0c0c0e body, zinc-700 borders, emerald/amber/sky/rose status accents). Setup replaces only the centred StateOverlay “idle” content.
  • Status pills keep the existing icon + text + colour pattern (never colour alone). New pills: “One-time setup” (zinc), “Not connected yet” (zinc), “Coming soon” (zinc).
  • Buttons map to existing shadcn variants — primary = emerald (matches the live “Launch”), secondary/outline = zinc, fallback primary = sky (matches the existing helper-install accent).
WCAG 2.1 AA checklist
  • Contrast: emerald/amber/sky/zinc text on near-black all ≥4.5:1; pill borders ≥3:1 for UI components.
  • Targets: Connect, Retry, Download, Reinstall all ≥44×44px; full-width on mobile.
  • No colour-only signals: every state pairs an icon + text label with its colour.
  • No naked disabled control: the non-Mac download always carries its “unlocks when Windows ships” explanation.
  • Keyboard: steps are real focusable buttons/links in reading order; the OS-prompt warning is inline text (not a tooltip), so it’s never hover-gated.
  • No primary action behind hover; no essential info in modals — success is inline, setup is inline.
Scope guard (no gold-plating)
  • No animated onboarding, video, or progress-wizard bar — the numbered list is the whole affordance (per Requirements).
  • Windows is a single gated state, not a parallel flow.
  • The “Advanced — pair a remote machine” disclosure already in the dock is untouched and out of scope for this first-run redesign.
