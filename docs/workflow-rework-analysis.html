4. The Bug: claim_next_step Rework Detection Gap

Current Detection Logic

mcp-server/src/tools/workflows.ts:335-354

1
claim_next_step(task_id) picks up Step 4
Step 4 is the first pending step in the run
2
Check: step.output || step.comment_count > 0
step.output = null (wiped by cascade)
step.comment_count = 0 (comments are on Step 6, not Step 4)
3
Condition is FALSE → rework check skipped entirely
rework_instructions = null
No REWORK REQUIRED section in instruction string
4
Agent starts from scratch with zero rejection context
The agent has no idea why this step was reset. It re-implements everything instead of making the targeted fix the reviewer requested.
Root Cause: Comments live on the wrong step The rejection comment (changes_requested) is placed on Step 6 (the step that was rejected), but claim_next_step only looks for comments on Step 4 (the step being claimed). In a cascade scenario, these are different steps — the rework context never reaches the agent.

The disconnect Comments survive cascade (they're in a separate table). But claim_next_step only queries .eq("step_id", step.id) — it never looks at comments on other steps in the run. The rework context exists in the database, but the query doesn't find it.

7. Fix: Look for Failed Steps Later in the Run Proposed

The fix adds a second check in claim_next_step: if no rework context is found on the claimed step itself, look for failed steps later in the same run — that's where the cascade originated from.

1
Scenario A: Check the claimed step itself
Existing logic — works when step was directly failed & retried
2
Scenario B (NEW): Check for failed steps later in the run
Query task_workflow_steps in same run where status = "failed" and step_order > current
If found, fetch that step's output + failure/changes_requested comments
3
Build rework_instructions from the source step
previous_failure_output = failed step's output
changes_requested = comments on the failed step
4
Include in instruction string
The existing P2 code formats rework_instructions into the instruction string with REWORK REQUIRED warning + feedback content
Wait — will a failed step exist later in the run? In the UI path: Step 6 is set to "failed" first, then the cascade resets steps 4-6 back to "pending". Step 6 ends up as "pending", not "failed" — the cascade overwrites the failed status.

This means Scenario B won't work for the UI path either, because there's no failed step left to find. The cascade resets ALL steps from the target onward, including the step that was just failed.
Deeper problem: the failed step doesn't stay failed Both failWorkflowStep (server action) and failStep (MCP tool) first set the step to "failed", then the cascade update uses .gte("step_order", target) which includes the failed step itself (since it has a higher step_order than the target). The failed step gets overwritten back to "pending".

This means: after cascade, there is NO record of which step was rejected or why — except for comments on that step (which survive, but aren't queried by claim_next_step).

8. Implemented Fix — Option B: Cascade propagates changes_requested comment

Status: IMPLEMENTED — Both MCP failStep and server action failWorkflowStep now insert a changes_requested comment on the cascade target step during rejection. The existing claim_next_step rework detection picks it up without modification.

How it works

  1. Reviewer rejects Step 6 with message "Please fix the validation" and cascades to Step 4
  2. failStep / failWorkflowStep marks Step 6 as failed with completed_at
  3. A failure comment is created on Step 6 (preserves the rejection reason)
  4. Cascade resets steps 4, 5, 6 → pending (clears output, claimed_by, timestamps)
  5. NEW: A changes_requested comment is inserted on Step 4 (the cascade target)
  6. When an agent calls claim_next_step and claims Step 4, the existing rework detection finds the changes_requested comment on Step 4 and populates rework_instructions
  7. The instruction string includes "REWORK REQUIRED" with the reviewer's feedback

Additional MCP/UI parity fixes

  • MCP failStep: Fixed broken FK (idea_id instead of task_id), added completed_at
  • Server action failWorkflowStep: Now creates failure comment (was missing)
  • Server action completeWorkflowStep: Now creates output comment, conditional output field
  • Server action startWorkflowStep: Sets claimed_by, accepts failed run status
  • Server action retryWorkflowStep: Clears claimed_by
  • UI reject handler: Simplified to single failWorkflowStep call (no separate addStepComment)
