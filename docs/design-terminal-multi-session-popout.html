UX Design · Step 2 of 6 · In-App Terminal — Later Phase: Multi-Session + Pop-out

Many terminals, one honest dock — tabs, a sessions list, and pop-out

Compass · UX Designer This phase turns the P1 single-session terminal dock into a VS Code-style tabbed multi-session surface, adds a "My sessions" list with a one-click panic button, and lets any session be popped out into its own browser window. Everything here is UI on top of plumbing that already isolates sessions (one relay DO per session, worktree isolation on disk). Desktop-only (≥768px) per requirement NG2 — the dock remains absent on mobile, exactly as in P1. Feature-flag OFF = none of this exists B9 F6.

⚠ Gate reminder (EC1–EC5). This is a design-only deliverable. The Requirements step's blocking entry gate is not met (parent P1 task e31f4594 at 3/6 steps). Nothing in this document licenses implementation before the gate opens.

1 · Building on Requirements (step 1)

Binding input: the Requirements step (Product Owner). That document fixed the shape of this phase and this design implements it rather than re-deciding it: tabs, not panes (NG4); pop-out as a same-origin browser window re-attaching the same relay session via owner-verified preemption, not native handoff (D1, NG3); a server-enforced cap of 3 with a user-explainable refusal (C2 E1); a registry-backed "My sessions" list with End-all (C3); and every P1 security invariant unchanged (F1–F6). Where the Requirements doc left Q2 (board-scoped vs global tabs) to design, §3 answers it with a recommendation.

Why VS Code's grammar, exactly. Jakob's Law — every user who has seen VS Code's panel tabs (P1's stated model) already knows: active tab shares the panel background and gets a top accent line; background tabs are dimmer; × on hover-and-active; "+" at the end. We innovate on value (per-tab live status glyphs, which VS Code terminals also do) and nowhere on interaction pattern. Active-tab signalling is threefold — background colour, top accent bar, and bolder text — never colour alone.

3 · Q2 answered — board-scoped tabs, global "My sessions"

Recommendation: board-scoped tab strip + global sessions list RECOMMENDED

The dock's tabs show only this idea's sessions. The "My sessions" panel (§9) is global — every live session across all ideas, with idea/task labels. This matches the Requirements doc's own inclination in Q2 and is the honest choice for four reasons:

  • The dock's identity line stays truthful. The dock lives on a board; its sessions run in that idea's folder (or its worktrees). A tab for another idea's session would show a cwd that has nothing to do with the board behind it.
  • The launch bus is already board-scoped. vibecodes:terminal-browser-launch events come from this board's toolbar and task cards B7; routing them into a mixed global strip invites delivering a prompt to the wrong idea's session.
  • Hick's Law. A user with 3 sessions across 2 ideas sees at most the 2 relevant tabs here, not all 3 — fewer choices at the point of action.
  • Nothing is hidden. US5's "lost track" fear is answered globally by "My sessions", which is one click away from the strip and is the named path in the cap-refusal message. The collapsed bar's session button shows the global count, so off-board sessions are never invisible (§6).

Trade-off accepted: switching between two ideas' terminals means switching boards (or popping one out — §10 is the designed answer to exactly that need). A global strip was rejected because it couples every board's dock to every session's lifecycle and breaks the label scheme in B3.

4a · Overflow — many tabs, long titles

  • Shrink then scroll: tabs flex between 190px and 110px min-width; past that the strip scrolls horizontally (mouse wheel / trackpad / arrow-key focus walking scrolls the focused tab into view). A gradient fade at the clipped edge signals more tabs (Continuity — the eye follows the row off-edge).
  • Never wrap, never hide the "+": "+" and "My sessions" are pinned outside the scroll region — launch and oversight stay one click away at any tab count.
  • Truncation is safe because the tooltip and the session header both carry the full title + identity; with cap 3 per user, pathological tab counts are bounded anyway E1.

Attention without alarm. A background tab leaving connected changes its glyph shape and tints its label — icon + colour together, satisfying B5's "never colour alone". 6 · Collapsed bar — the worst-status summary

Collapsing the dock never ends anything B8. The collapsed bar must answer "are my agents okay?" without expanding B5: it shows a count-by-state summary, ordered worst-first, plus the global sessions button.

▣ Terminals ● 2 connected ↻ 1 reconnecting
Mixed state: "● 2 connected · ↻ 1 reconnecting" (AC 5). Chips appear only for states that exist — all healthy collapses to a single "● 3 connected". An error state adds a rose "▲ 1 needs attention" chip, which is also the bar's leftmost (worst-first ordering = information scent: the first thing you read is the thing to act on). Clicking any chip expands the dock and activates the worst-affected tab.

7 · Launch routing & the cap

7a · A launch arrives while tabs exist B7

Toolbar "In the browser", a task-card "Launch in browser terminal", or the strip's "+" all do the same thing: open a new tab — never replace a running session. The new tab becomes active immediately in connecting; the launch payload (prompt + cwd) is delivered to that tab only. Existing sessions' streams are untouched (AC 4). If the dock was collapsed, it expands with the new tab active.

7b · Cap reached C2 E1

The mint route refuses before creating anything. The "+" (and launch menu items) are not disabled — a dead control with no explanation is a dead end. Instead the refusal explains and routes (constraint: no disabled buttons without adjacent explanation):

⚠️
You already have 3 terminals running
That's the limit for now. End one to start another — view my sessions.
Cap refusal, shown as a toast anchored to whatever triggered the launch. The link opens the "My sessions" panel (§9) — the fix is one click from the failure (recognition over recall: we show the sessions rather than asking the user to remember where they are). Same copy and behaviour from the toolbar menu, the task-card menu, and "+". No session is minted; no tab appears (AC 9).

8 · First-run, empty, and flag-off

◻ No sessions yet (dock idle)

Identical to P1's resting state: the dock only exists after the user launches "In the browser". No tab strip renders with zero sessions — the strip appears with the first session and persists (even as a single tab + "+") so the surface is stable. First launch runs the unchanged P1 install/pair path, then lands as tab 1.

◻ Last session ended

Ending the final session returns the dock to the P1 idle/resting state B8 — the strip does not linger empty. Ended tabs before that stay visible (■) until closed, preserving their scrollback for reading.

⚑ Flag OFF B9 F6

Honest absence. No dock, no strip, no "My sessions", no pop-out routes, no new menu items. Users who never opted in see zero new UI anywhere in this phase.

◔ Loading (sessions list)

The "My sessions" panel loads from the registry: skeleton rows (label + identity-line bars), never a spinner-only void. Registry unreachable → inline error with Retry, and the panel still offers End for locally-known sessions.

9 · "My sessions" — the global list + panic button

Opened from the strip or collapsed bar button, or from the cap-refusal link. A popover panel (not a modal — it must not block the board; constraint: no modals for content that could be inline). Lists every live session across all ideas from the registry C3, each with the full identity line C4. Popped-out sessions appear once, marked ⧉ (AC 18).

10 · Pop-out — same session, different glass

Pop-out re-attaches the browser leg of the same relay session in a new same-origin window D1. Owner-verified preemption does the safety work: the moment the popped window attaches, the relay closes the dock's leg — one browser leg at all times F2 (AC 13). No tokens ever ride the URL D4 (AC 16).

Popped out

The placeholder never auto-reattaches while the popped window lives — that would preempt-loop D3 R3. Reattach happens two ways only: (1) the user clicks Bring back to dock (dock preempts the window; the window shows its own honest "This session went back to the dock" end-card with a "Close window" button), or (2) the popped window closes and signals via BroadcastChannel, after which the dock reattaches automatically. If the close-signal is missed (crashed window), the placeholder's Bring-back still works — it's the manual override for every failure of the automatic path.

12 · Full state index

● Tab connected

Quiet glyph; active tab shows P1 connected panel. Next: user works, pops out, or ends.

◌ Tab connecting / waiting-to-pair

Pulsing ring; P1 connecting/waiting overlays in body. New-tab default state after any launch route.

↻ Tab reconnecting (background)

Spinner glyph + amber label; collapsed bar adds "↻ n reconnecting". Grace window runs per-tab; recovery is silent, exhaustion → ■ ended with honest copy.

▲ Tab error

Triangle + rose tint; body shows the P1 error overlay for that errorKind. Only this tab is affected — siblings stream on (AC 5, 19).

■ Tab ended

Calm square; body shows P1 ended overlay (user/idle/max/reconnect-failed copy) + "Launch again". Tab stays until closed so output can be read.

⧉ Tab popped out

Violet glyph; placeholder body with Bring back. Window closing auto-returns the session to the dock.

⚠ Cap reached

Toast: "You already have 3 terminals running — end one to start another" + link to My sessions. No mint, no tab (AC 9).

⧉ Pop-up blocked

Toast explains; tab unchanged and still attached (D7).

◻ Empty / idle

P1 resting dock; sessions list empty state teaches the launch path.

⚑ Flag off

Nothing renders. Anywhere. B9 F6 (AC 8).

14 · Accessibility

  • WAI-ARIA tabs pattern: the strip is role="tablist" (aria-label "Terminal sessions"); tabs are role="tab" with aria-selected, roving tabindex. ←/→ move focus, Home/End jump, Enter/Space activate. Manual activation (focus ≠ select), because activating swaps a live xterm surface — arrow-browsing must not thrash sessions' rendering.
  • Close from keyboard: Delete on a focused tab triggers the same end-session confirm as ×. We deliberately do not bind Ctrl/Cmd+W (it closes the browser tab — catastrophic mis-affordance).
  • Status is announced, not just drawn: each tab carries visually-hidden state text ("reconnecting", "ended"…) appended to its accessible name; background-tab state changes post to a single polite aria-live region ("Migrate ingredient quantities: reconnecting") — one announcer, so three tabs can't shout over each other. Cap and popup-blocked toasts are role="alert".
  • Pop-out focus management: on pop-out, focus lands in the new window's terminal input; the dock placeholder's "Bring back to dock" is its first focusable. On bring-back/auto-return, focus goes to the reattached tab. The popped window's <title> is "Terminal · {label}" for window-switchers and screen readers.
  • Contrast (WCAG 2.1 AA): all state colours on the dark panels are the P1-audited pairs (emerald/amber/rose/violet 300-range on zinc-900 ≥ 4.5:1 for the 11–12.5px bold labels; glyphs are supplementary to text, and state is always also words in tooltip, header pill, and accessible name — never colour alone). Muted label text stays at or above zinc-400 on the strip; zinc-500 is reserved for the non-essential identity line, which is duplicated in tooltips.
  • Targets & visibility: tab rows 38px tall; × hit area ≥24×24 inside a ≥110px tab; all controls visible at rest (no hover-only affordances — × is always rendered, not hover-revealed). Focus rings: 2px accent, offset 2, on tabs, ×, +, and all panel buttons.
  • Desktop-only: per NG2 the dock (and therefore the strip, panel, pop-out) does not render under 768px; no mobile layouts are specified, matching P1.
