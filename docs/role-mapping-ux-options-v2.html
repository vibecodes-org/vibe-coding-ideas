Role-Agent Mapping — Extended Options

3 alternative approaches that rethink the Role Coverage panel beyond chips and tables.

Option D: Agent-Centric Grouped View

Flip the perspective: instead of listing roles, list agents and show which roles each one handles. Immediately shows which agent is overloaded (handling many roles) vs idle.

Pros

  • Shows workload distribution — Atlas handles 4 roles, others handle 1
  • Reveals idle agents (BA Bob has no matching roles)
  • Natural grouping — users think "what is each agent doing?"
  • Unmatched roles clearly separated at bottom

Cons

  • Takes more vertical space than chips
  • Partially duplicates the agent cards below
  • Harder to answer "who handles developer?" at a glance

Option E: Two-Column Connection View

Roles on the left, agents on the right, with visual lines connecting them. Like a matching exercise.

Pros

  • Visually intuitive — like a wiring diagram
  • Shows many-to-one relationships clearly (3 roles → Atlas)
  • Unmapped roles visually obvious (dashed line to nowhere)
  • Memorable and distinctive UX

Cons

  • Complex to implement (SVG line rendering, positioning)
  • Doesn't scale well with many roles (20+ roles = spaghetti)
  • Hard to make responsive on mobile
  • Over-engineered for the actual problem

Option F: Agent Cards with Role Count Badge

Keep the current chips but enhance the existing agent cards below with a "handles X roles" indicator. Click the agent card to see which roles it matches.

  • Need click/expand for details — not immediate

Summary of all 6 options:

A. Tooltip on hover — compact, least discoverable
B. Inline "role → agent" in chips — best balance of visibility and compactness
C. Expandable table — most detailed, heaviest
D. Agent-centric groups — shows workload distribution, new perspective
E. Connection lines — visually distinctive but over-engineered
F. Role count on agent cards — enhances existing cards, needs click for detail

My recommendation: B + tooltip for tier or D if workload visibility matters most.
