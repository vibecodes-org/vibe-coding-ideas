list_notifications — enrichment spec

Quick Win · UX Design (step 2/6) · task “MCP list_notifications is useless for acting on mentions”. This doc is the implementation spec for step 4. Additive only — no schema, no signature changes.

TL;DR. Enrich each notification so a calling agent can act without a second round-trip: add the raw FK ids (task_id, comment_id, discussion_id, reply_id), inline-join task {id,title} + discussion_title (both FKs exist on notifications), a never-null url deep-link built by importing the canonical buildNotificationUrl (zero drift), and a best-effort mention_context (the triggering board-comment excerpt) via one extra batched query. Total added cost: 1 extra query (the main select stays one query; joins are embeds). Budget FR6 (≤2) satisfied.

1 · Current state

listNotifications (mcp-server/src/tools/notifications.ts) selects only:

id, type, read, created_at,
actor:users!notifications_actor_id_fkey(id, full_name),
idea:ideas!notifications_idea_id_fkey(id, title)

Problem: no task_id/comment_id/discussion_id, no titles beyond the idea, no link. An agent that receives “you were mentioned” cannot open the task, read the comment, or even name the task without extra lookups. Everything below is additive — existing fields keep their shape.

3 · Query plan

3a · Main query (one query, joins are embeds)

All four embed FKs exist on notifications (verified in database.ts Relationships): notifications_actor_id_fkey, _idea_id_fkey, _task_id_fkey → board_tasks, _discussion_id_fkey → idea_discussions. So the task title and discussion title join inline — no extra query. New select string:

id, type, read, created_at,
idea_id, task_id, comment_id, discussion_id, reply_id,
actor:users!notifications_actor_id_fkey(id, full_name),
idea:ideas!notifications_idea_id_fkey(id, title),
task:board_tasks!notifications_task_id_fkey(id, title),
discussion:idea_discussions!notifications_discussion_id_fkey(title)

Map the discussion embed to the flat discussion_title in code: discussion_title: row.discussion?.title ?? null, then drop the nested discussion object from the emitted shape. A deleted task/discussion yields a null embed — the raw task_id/discussion_id survives, so the url still builds.

3b · Batched mention_context query the only extra query

Collect the (task_id, actor_id) pairs from notifications where type === "task_mention" and task_id != null. Skip the query entirely when there are zero such pairs. Then one query into board_task_comments:

ctx.supabase
  .from("board_task_comments")
  .select("task_id, author_id, content, created_at")
  .or(pairs.map(p => `and(task_id.eq.${p.task_id},author_id.eq.${p.actor_id})`).join(","))
  .order("created_at", { ascending: false })
  .limit(200)   // safety cap against a chatty author on one task

Client-side pairing: rows arrive newest-first; keep the first row seen per (task_id,author_id) key = most-recent comment by that author on that task. A mention notification is created the instant its comment posts, so the most-recent comment is the triggering one in ~all cases (hence best_effort:true).

.or(and()) vs .in×.in — decision: use .or() of and() pairs. Rationale: a page is ≤50 notifications and typically few are mentions, so the pair count is small and .or() is precise (fetches only the requested pairs). .in(task_ids)×.in(author_ids) would fetch the full cross-product (author A’s comments on task B even if A never mentioned anyone there) — wasteful and, per team memory, large .in() lists have misfired at scale. If pair count ever exceeds ~40, chunk the .or() — but that stays a single logical query for the budget.

Truncation: text = content.length > 200 ? content.slice(0,200) + "…" : content. source is the literal "board_task_comment"; best_effort is literal true.

Documented limitation: mentions originating from workflow_step_comments are not covered — those notifications also carry type:"task_mention" + task_id, but the triggering text lives in a different table. When no matching board_task_comment exists, mention_context is simply omitted (never wrong text). This is intentional scope for the Quick Win.

Budget

Main select = 1 query (unchanged count; titles are embeds). mention_context = 1 query, skipped when no mention pairs. Worst case 2 queries total, +1 vs today. FR6 (≤2 extra) satisfied.

4 · URL helper — placement & branches

Decision: import the canonical web helper directly. import { buildNotificationUrl } from "../../../src/lib/notification-url"; No mcp-server-local copy.

Rationale. FR3 demands routing identical to buildNotificationUrl, and drift here silently produces dead links. The cross-package import is already the established pattern — mention-notify.ts imports ../../../src/lib/logger the same way, and mcp-server/tsconfig.json maps @/* → ../src/*, so the helper’s type-only import type { Database } from "@/types/database" resolves under tsc and is erased at runtime (tsx/node). The mention feature copied logic into mcp-server/src/lib only because that logic is mention-specific parsing with no canonical web owner; URL routing is the opposite — one canonical owner, high drift cost. Importing wins decisively; a local copy would reintroduce exactly the drift FR3 forbids.

Signature (unchanged):

buildNotificationUrl({
  type, ideaId, commentId, taskId, discussionId, replyId, appUrl
}): string

Call it per-notification, passing the row’s raw FK ids and the module-level APP_BASE_URL (§5).

Branch table — every enum type → url

The helper is field-driven (FK presence first), then type-driven fallback. Table shows the URL for each notification_type given its typical FK payload; {b} = {appUrl}/ideas/{ideaId}.

typeFK carriedurl

Precedence inside the helper (do not reorder): !ideaId → taskId → discussionId → commentId → type fallbacks (task_mention→board, discussion*→discussions) → {b}. The table above is a consequence of that order, not a second source of truth.

5 · Env / config

ModeReadsResolves to
Remote (HTTP, Next.js route)process.env.NEXT_PUBLIC_APP_URLthe deployment’s app URL (present in Vercel env)
Local (stdio)process.env.NEXT_PUBLIC_APP_URLusually unset → fallback constant (correct for prod users; staging users may export it)

One resolution, one constant, read once at module load in notifications.ts:

const APP_BASE_URL =
  (process.env.NEXT_PUBLIC_APP_URL ?? "https://vibecodes.co.uk").replace(/\/+$/, "");

Trailing-slash strip keeps buildNotificationUrl’s {appUrl}/ideas/… concatenation clean. Same var name in both modes — each process resolves its own env, and the fallback constant is the FR4 default. No new env var is introduced.

Task deleted (task embed = null)task: null, but raw task_id survives → url still {b}/board?taskId=…. Not null.
Discussion deleted (embed = null)discussion_title: null; raw discussion_id survives → url still built.
NEXT_PUBLIC_APP_URL absentFallback https://vibecodes.co.uk. url never null.
Mention from workflow_step_commentsNo matching board comment → mention_context omitted (documented limitation §3b). Never emits wrong text.
Main query errorUnchanged — still throws Failed to list notifications. Enrichment does not alter the existing error path.

8 · Open questions

None blocking. Two intentional scope calls are already resolved in-doc and need only a nod at the human gate (step 3):

  • workflow_step_comments excluded from mention_context (§3b) — accepted limitation for the Quick Win; a follow-up card can add it.
  • “Most recent” = triggering comment heuristic (§3b) rather than time-correlating each notification’s created_at to a specific comment — chosen for simplicity; best_effort:true signals the approximation to callers.
