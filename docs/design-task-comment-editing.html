Task Comment Editing — UX Design Review

Feature Development workflow · Step 2: UX Design · Building on the Requirements step (step 1)

Component: src/components/board/task-comments-section.tsx · Server action: updateTaskComment in src/actions/board.ts · Scope: two backend defects + minimal UX deltas. This is not a redesign.

Verdict: the existing inline edit flow is sound and stays. Pencil → inline textarea → Check/X with Ctrl+Enter / Esc matches Linear/GitHub conventions (Jakob's Law), edits happen in place (no modal), and the action icons are always visible — not hover-gated — so mobile works. Critically, handleSaveEdit applies the local state update after await updateTaskComment() resolves. That means once the server action starts throwing on 0-row updates (Defect 2 fix), the existing catch path automatically delivers AC8: error toast, no false "saved" state, edit mode stays open with the user's draft intact. No client reordering is needed — only guards and small accessibility fixes, listed in §3.

1User flow — trigger to completion, including error recovery

Actor: a team member editing their own comment, or a comment authored by a bot they own (userBotIds). The flow below is the flow as it will behave after the two backend fixes; steps that change are annotated.

1
Trigger — click/tap the pencil icon
Pencil sits inline next to the timestamp on every comment the user can modify — own comments and owned-bot comments (AC5). Always rendered, never hover-only. Δ3, Δ4: needs aria-label + bigger hit area
2
Edit in place
Comment body swaps to a textarea pre-filled with current content, auto-focused. Check (save) and X (cancel) buttons directly below — primary action adjacent to the work (Fitts). Ctrl/⌘+Enter saves, Esc cancels; tooltips advertise both shortcuts (recognition over recall).
3
Guards before the network
Empty edit (AC7): trimmed-empty content never reaches the server — Check is disabled and a keyboard save falls through to cancel, restoring the original comment. Deleting is a separate, explicit action (trash → undo toast). Unchanged edit: silently exits edit mode; no request, no spurious "(edited)". Δ2: add in-flight re-submit guard
4
Save — pessimistic commit
await updateTaskComment() runs first; local state is updated only on success (content + updated_at), then edit mode closes. Sub-second on typical latency, so no extra "saving…" affordance is needed beyond the disabled Check button.
✓
Success — "(edited)" everywhere, forever
Editor sees updated content with the (edited) suffix beside the timestamp. The DB trigger (Defect 1 fix) bumps updated_at, so the indicator survives reload (AC1–2) and the realtime UPDATE payload carries the new updated_at to other viewers live (AC3) — the client handler already maps it through. Pencil remains visible for future edits, including on bot comments (AC5).
✕
Error recovery — failure is loud, work is preserved
If updateTaskComment throws — including the new 0-row throw (Defect 2 fix) — toast.error("Failed to update comment") fires and edit mode stays open with the draft intact. Nothing is lost, nothing false is rendered (AC8). The user can retry (Check again) or bail (Esc restores the original). This recovery already exists in the component; the fix simply makes the server stop lying to it.
The pre-fix false-success path, for contrast: owner edits a bot comment → server matches 0 rows (.eq("author_id", user.id) excludes bot-authored rows) → no error → client "optimistically" renders the new content and closes the editor → reload silently reverts it. The user's work evaporates with a success face on. This is the worst failure class — silent data loss — and it is fixed server-side, not with UI.

3UX deltas for Implementation

Everything the implementation step must touch in task-comments-section.tsx. The two backend fixes (DB trigger, 0-row throw in updateTaskComment) are specified by the Requirements step and are out of scope here.

1

Preserve the save/catch flow exactly verify — no change

handleSaveEdit already awaits the server action before touching local state, and its catch shows toast.error("Failed to update comment") while leaving editingId/editContent intact. Once the server action throws on 0 rows, this delivers AC8 with zero client changes. Do not reorder to optimistic-first and do not close the editor on error. Add/confirm a test or manual check: failed save → toast + editor open with draft + list still shows original content.

2

Guard against double-submit during an in-flight save required

The edit textarea stays enabled while savingEdit is true, so Ctrl+Enter can invoke handleSaveEdit again mid-flight (the disabled Check button only guards the click path). Add if (savingEdit) return; at the top of handleSaveEdit. Optionally also pass disabled={savingEdit} to the edit Textarea.

3

aria-labels on all icon-only buttons cheap a11y

Radix TooltipContent does not give a button an accessible name. Add: aria-label="Edit comment" (pencil), aria-label="Delete comment" (trash), aria-label="Save comment" (Check), aria-label="Cancel edit" (X), and aria-label="Send comment" on the composer's Send button while in the file. Five attributes, no visual change. (WCAG 2.1 AA 4.1.2.)

4

Enlarge pencil/trash hit areas cheap a11y

The bare <button>s currently have a ~12px tap target (the h-3 w-3 icon itself). Keep the 12px glyph and visual density; add padding with negative margin so the layout doesn't shift, e.g. className="p-1.5 -m-1 rounded text-muted-foreground/60 hover:text-foreground" → ≥24px effective target. Do the same on the trash button (hover keeps hover:text-destructive).

5

Realtime UPDATE handler: keep as-is verify — no change

The UPDATE subscription already maps content and updated_at into state (lines ~131–139), which is exactly what AC3 needs once the trigger produces a changed updated_at. No client change; just verify with two browser sessions after the migration lands. Note the editor's own optimistic updated_at (client clock) is later overwritten by the server value via this same handler — harmless, by design.

6

Empty-edit behaviour: keep cancel-on-empty verify — no change

Trimmed-empty or unchanged content exits edit mode without a request (AC7). Do not add an error toast for this — an empty edit is a cancelled edit, not a failure; deletion stays an explicit separate action with its own undo toast. The disabled Check button is acceptable because the enabled X/Esc exit sits directly adjacent.

4Intentionally NOT changing

  • The inline edit pattern itself — pencil → textarea → Check/X with Ctrl+Enter/Esc. It matches platform conventions and the ACs; no modal, no redesign, no new components.
  • The "(edited)" presentation — quiet suffix beside the relative timestamp, derived from updated_at !== created_at. No tooltip with edit time, no edit history — out of scope.
  • Delete flow — optimistic removal with undo toast (undoableAction) is untouched.
  • Composer, mentions, markdown rendering — the new-comment form, @-mention autocomplete, and Markdown body rendering are unrelated to these defects.
  • No "saving…" spinner or skeleton states — the pessimistic save is fast; the disabled Check button during savingEdit is sufficient feedback for a sub-second operation.
  • No permission-UI changes — canModifyComment (own + owned-bot comments) is already correct; Defect 2 was the server disagreeing with it.
  • Layout at 375px — header row wraps and the textarea is full-width; verified structurally sound, no responsive changes needed.
