Clickable Attachment References in Step Outputs

UX Design — step 2 of Feature Development · builds on the Requirements step

Surfaces: step Output panel + step comments in step-detail-dialog.tsx · rendered via optional attachments prop on the shared Markdown component

01 · Design rationale — match what already reads as “clickable” here

In this product, inline clickables are colored text runs, not chips: @mentions render as font-medium text-blue-400 hover:text-blue-300 hover:underline, and markdown links as text-primary underline. A chip/badge treatment would collide with the Deliverables badges two sections above the Output panel (which are not clickable) and would swell multi-reference paragraphs. So the attachment reference is an inline link-styled button in the mention blue family, differentiated from a person-mention by a leading Paperclip icon — the exact icon that already heads the Files tab, giving instant information scent: “this opens a file”, not a profile.

It is a <button>, not an <a>, because there is no href at render time — the signed URL is minted on click. Native button semantics give Enter/Space activation and correct screen-reader announcement for free.

5

Wrap, don’t truncate: break-all on the label span. Middle-truncation would hide version suffixes (…v3.2-APPROVED.pdf vs …v3.1-DRAFT.pdf) — exactly the part users need to tell similar files apart. The Output panel is narrow (sm:max-w-md dialog); wrapping matches how inline code already behaves here (break-all).

6

Deleted state = nothing: no tooltip, no dashed underline, no title. Matching runs against the task’s live attachment list, so a deleted file simply never matches — the renderer cannot distinguish “deleted attachment” from “ordinary text that looks like a filename”, and any ghost affordance would produce false positives.

04 · Exact element spec

// Emitted by the attachments pass inside Markdown (mirrors renderMentions)
<button
  type="button"
  title={`Open attachment ${attachment.file_name}`}
  aria-busy={isMinting}
  disabled={isMinting}
  onClick={() => openAttachment(attachment)}   // signed URL → inline for image/pdf/html/text, download otherwise (Files-tab pattern)
  className="inline-flex max-w-full items-baseline gap-1 rounded-sm text-left align-baseline
             font-medium text-blue-400 underline decoration-blue-400/40 underline-offset-2
             hover:text-blue-300 hover:decoration-blue-300
             focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-1
             disabled:cursor-wait disabled:opacity-60 cursor-pointer"
>
  <Paperclip className="h-3 w-3 shrink-0 translate-y-px self-center" aria-hidden="true" />
  <span className="break-all">{matchedText}</span>
  {isMinting && <Loader2 className="h-3 w-3 shrink-0 animate-spin self-center" aria-hidden="true" />}
</button>
  • Icons: lucide Paperclip (leading, always) + Loader2 (trailing, minting only). Both h-3 w-3, matching the dialog’s inline icon scale.
  • Font weight: font-medium at rest (same as mentions); inherits font-semibold when nested inside <strong> — bold precedence: the pass runs on text nodes inside strong/em (as processMentionChildren already does), so the link nests within the bold, never swallows it. A filename split across a formatting boundary (file**name**.html) does not match — matching is per-text-node only, same rule that keeps code spans clean.
  • Display text: the author’s original casing/run from the source text; match is case-insensitive against real board_task_attachments.file_name values, longest-filename-first, with the same word-boundary check the mention matcher uses. Adjacent punctuation (shortlist.html., (shortlist.html)) stays outside the link.
  • Touch target: inline-text exception under WCAG 2.5.8 applies (link sits in a sentence); the full filename — typically 150px+ wide — is the target, and long names wrap to multi-line targets. No minimum-height padding that would break line rhythm.
  • Light mode: text-blue-400/blue-300 hover hold ≥4.5:1 on the light Output tint (bg-blue-500/5 over white) is not guaranteed — use text-blue-600 hover:text-blue-700 dark:text-blue-400 dark:hover:text-blue-300 if the app’s light theme is enabled for these surfaces. (Mentions currently hardcode blue-400; app default is dark. Follow whichever the mention pipeline does — consistency beats local perfection.)
  • Screen reader: announced as “Open attachment {name}, button” via title+content; aria-busy during mint.

05 · Explicitly not changing

  • KEEP@mention rendering, colors, and matching pipeline — attachments pass runs alongside it, filenames matched before mention fallback only where they overlap (no filename starts with @ in practice; if one does, mention match wins after @).
  • KEEPMarkdown code spans and code blocks — never linkified (AC).
  • KEEPStandard markdown <a> links — untouched, still text-primary underline, still open in new tab.
  • KEEPThe Files tab (task-attachments-section.tsx) — its preview/download buttons, icons, and signed-URL logic are unchanged; the open-on-click behaviour here reuses that pattern, it doesn’t fork it.
  • KEEPOutput panel container (border-blue-500/20 bg-blue-500/5) and comment-type left-border styling.
  • KEEPEvery other Markdown consumer (idea descriptions, task comments, discussions) — no attachments prop passed ⇒ zero behaviour change.
  • KEEPStored step output / comment text — linkification is render-time only; the database text is never rewritten.
