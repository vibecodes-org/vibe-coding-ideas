AI Generate Auto-Labelling

UX Design Document — Building on Requirements step findings

UX Impact Assessment

No UI Changes Required

This feature is entirely backend — it enhances the AI prompt and fixes a missing auto-rule trigger. The existing AI Generate dialog, preview table, and insertion flow remain unchanged. Users will simply see better, more relevant labels on generated tasks, and workflow templates will auto-attach.

User Flow: Before vs After

Current Flow

  1. User clicks "AI Generate"
  2. AI creates tasks (may invent random labels)
  3. User reviews & applies tasks
  4. ❌ Labels don't match auto-rules
  5. ❌ No workflows attached
  6. ❌ User must manually label each task
  7. ❌ User must manually trigger workflows

After This Change

  1. User clicks "AI Generate"
  2. AI creates tasks with correct labels
  3. User reviews & applies tasks
  4. ✅ Labels match auto-rule labels
  5. ✅ Workflows auto-attach on insert
  6. ✅ Zero manual labelling needed
  7. ✅ Tasks ready to work immediately

Data Flow (Updated)

1
User opens AI Generate dialog
Selects optional persona, writes prompt. No change to this step.
2
POST /api/ai/generate-tasks CHANGED
Route now fetches board labels, auto-rules (with template names), and template descriptions. Adds them to the AI prompt as classification context.
3
Claude generates tasks with labels IMPROVED
AI is now instructed to classify tasks using existing auto-rule labels. Output schema unchanged: { tasks: [{ title, description, labels, ... }] }
4
User previews in ImportPreviewTable
Labels shown as badges on each task row. Labels are already visible in the preview — no UI change needed. Users may now see more familiar label names (e.g. "Feature" instead of random AI-invented labels).
5
insertTasksSequentially CHANGED
After inserting labels, now calls addLabelsToTask server action which triggers checkAndApplyAutoRules. Previously used direct Supabase insert (bypassed auto-rules).
6
Workflows auto-attach NEW
Auto-rules fire → workflow templates applied → tasks show workflow steps immediately on the board.

Prompt Context Addition

The following context block will be appended to the AI prompt when auto-rules exist:

## Board Labels & Workflow Auto-Rules The following labels exist on this board. Use these EXACT names when labelling tasks: - Feature, Bug, Infrastructure, Research, Quick Win, MVP Launch, UX, Testing The following auto-rules are configured. When you assign one of these labels, a workflow template will automatically be applied to the task: | Label → | Workflow Template | Template Purpose | |---------|------------------|-----------------| | Feature → | Feature Development | Full dev lifecycle: requirements → design → implement → QA | | Bug → | Bug Fix | Reproduce → fix → verify → regression check | | Infrastructure → | Infrastructure Change | Impact assessment → implement → deploy → verify | Classify each task and assign the most appropriate label(s). You may also assign additional labels beyond these (e.g. "MVP Launch", "Quick Win").

Edge Cases & States

ScenarioBehaviour
No auto-rules on board Prompt omits the auto-rules section entirely. Existing label names still included so AI reuses them. Behaviour identical to current.
Auto-rules exist but no labels match task AI assigns no auto-rule labels to that task. No workflow attached. Task created normally.
AI assigns an auto-rule label + extra labels Both applied. Auto-rule fires for the matching label. Extra labels are just labels.
AI invents a new label name Existing behaviour: import.ts creates the label with an auto-assigned colour. No auto-rule fires (no rule for new labels).
Multiple tasks get same auto-rule label Each task gets its own workflow run independently. No conflicts.
Task already has a workflow (replace mode) Duplicate run prevention index blocks second workflow application. Label assigned, no duplicate workflow.

Accessibility & Responsiveness

No accessibility impact. No new UI elements, interactions, or visual changes are introduced. The preview table already renders labels as accessible badges. Keyboard navigation, screen reader compatibility, and colour contrast are unaffected.

Recommendation

This is a clean backend enhancement with zero UI risk. The existing preview flow already handles labels well — users will simply see better labels appearing. No design review of visual changes is needed; the Design Review step should focus on validating the prompt wording and edge case handling documented above.
