Step 2 — UX Design

Task Card Shortcut Menu

A per-card overflow (⋯) menu giving fast access to Move to top, Move to bottom, Archive, and Delete — built on existing board ops. Design grounded in the live board-task-card.tsx and the shadcn DropdownMenu primitive.

Component: src/components/board/board-task-card.tsx Primitive: ui/dropdown-menu.tsx Confirm: ui/alert-dialog.tsx ~80% UI over existing ops

1 Trigger placement

Resolves Open Question 1 → use a ⋯ kebab button, not right-click (rationale in §7).

The trigger mirrors the existing drag handle's reveal logic exactly (board-task-card.tsx lines 411–421): desktop = hover-revealed, mobile = always visible. This keeps the card calm on a dense desktop board while honoring our constraint that primary actions must never be hover-only on touch (mobile has no hover).

Reveal pattern (copy the grip)

Reuse the drag-handle Tailwind recipe so the two affordances feel like siblings:

opacity-100              // mobile: always shown
sm:opacity-0            // desktop: hidden at rest
sm:group-hover:opacity-100
focus-visible:opacity-100 // a11y: keyboard reveal

Add focus-visible:opacity-100 (the grip lacks it) so keyboard users can Tab to the trigger and see it. Also keep it visible while its own menu is open via data-[state=open]:opacity-100.

Position & geometry

  • Top-right corner of the card body, balancing the grip on the left — predictable per Jakob's Law (Linear / Notion / Trello all top-right).
  • Hit area ≥ 44×44px on mobile. The glyph is ~16px; pad the button to meet Fitts/WCAG. On desktop a 26px visual box is fine since it's pointer-driven.
  • Renders inside the card's relative wrapper, absolutely positioned so it overlays the metadata without reflowing layout.

Destructive styling — colour is never the only signal

Per our constraint, Delete is distinguished by icon + label + position + separator, not just red text. shadcn's DropdownMenuItem variant="destructive" already wires the red foreground and a red-tinted focus/hover background; the Trash2 icon and post-separator placement carry the meaning for anyone who can't perceive the red.

Reasonable to skip a separate "Archive vs Delete" separator if it feels heavy — but always keep one directly above Delete. The visual break is a guardrail that reduces mis-clicks on the irreversible action (Hick's Law: it makes Delete feel like a distinct decision, not item #5 in a list).

Delete confirmation — AlertDialog

Resolves Open Question 3 → confirm dialog (rationale §7). Uses ui/alert-dialog.tsx.

Delete this task?

“Design the empty state” will be permanently removed, along with its comments, checklist, and attachments. This can't be undone.

  • Title names the consequence, body names the task by title (information scent — the user confirms which task).
  • Default focus lands on Cancel, not Delete — destructive action is harder to reach, and Esc / click-outside both cancel.
  • The destructive button mirrors the menu item: red fill + the word "Delete" (icon optional). AC: Delete → confirm dialog

Error toast on failure

⚠
Couldn't move taskSomething went wrong. The card has been restored — try again.

Every action's catch block calls toast.error() (per CLAUDE.md: never fail silently) and the optimistic change rolls back. Messages are action-specific: "Couldn't move task" / "Couldn't archive task" / "Couldn't delete task".

4 Interaction & flow per action

All four are ~80% reuse. Optimistic-first: mutate local state, fire the server action, roll back on error. This matches BoardOpsContext, which already returns rollback functions (e.g. deleteTask(taskId, columnId) returns a restore fn).

Move to top / Move to bottom

1
User opens ⋯ → clicks Move to top. (Disabled if already at the edge — see §2.)
2
Optimistic: new helper moveTaskWithinColumn(taskId, columnId, "top"|"bottom") reorders the local column array. Card animates to the new position using the card's existing 120ms dnd-kit transition feel.
3
Persist: moveBoardTask with position = firstPos − POSITION_GAP (top) or lastPos + POSITION_GAP (bottom). POSITION_GAP is the existing constant from @/lib/constants — do not invent a new gap.
4
On error: roll back to prior order, toast.error("Couldn't move task").

Scope: within the task's current column only (resolves Open Question 2). No cross-column movement — that's what drag-and-drop is for.

Archive

1
Item hidden if task.archived already true.
2
Optimistic: new archiveTask(taskId, columnId) sets archived: true locally → card immediately drops to opacity-50 and shows the "Archived" badge (line 457–462 already renders this). Dashboard/active counts exclude it.
3
Persist: updateBoardTask(taskId, ideaId, { archived: true }). No confirm — archive is reversible, so an extra step would violate "don't gate reversible actions." AC: archive optimistic
4
On error: restore archived: false, toast.error.

Delete

1
Click Delete → close dropdown, open the AlertDialog (§3). Nothing has changed yet.
2
Confirm → optimistic deleteTask(taskId, columnId) (existing context fn) removes the card and returns a rollback fn.
3
Persist: deleteBoardTask(taskId, ideaId). On error: call the returned rollback fn to re-insert the card, toast.error("Couldn't delete task").

Keyboard & accessibility (Radix gives most of this free)

  • Trigger is a real <button> with aria-label="Task actions" — reachable by Tab, opens on Enter/Space.
  • Inside the menu: ↑/↓ roam items (skipping disabled ones), Enter activates, Esc closes and returns focus to the trigger. Radix DropdownMenu handles this; don't re-implement.
  • Disabled items use disabled prop → aria-disabled, removed from arrow-key traversal. AC: top/bottom disabled at edges
  • AlertDialog traps focus, defaults to Cancel, closes on Esc; focus returns to the kebab after close.
  • Touch targets ≥44px on mobile for the trigger and each menu row (menu items get generous vertical padding on small screens). AC: ≥44px mobile + keyboard a11y

5 Realtime: don't let the card revert

Board realtime can re-broadcast a task and overwrite optimistic local state. The existing trusted-local-move mechanism (trusted-state.ts, see commit 6e85044 "stop drag-drop bounce-back via trusted local move state") already solves this for drag-drop.

Reuse it for menu moves. Move-to-top/bottom and archive must register the same trusted-local-move marker so an inbound realtime echo of the old position/state is ignored until the server write confirms. This is the same guard drag-drop uses — no new mechanism needed. AC: realtime doesn't revert

7 Open questions — resolved / for review

Q1 — ⋯ button vs right-click → resolved: ⋯ button

Recommend the kebab button (matches the requirements' recommendation). Right-click context menus are undiscoverable, have no mobile equivalent (long-press conflicts with scroll/drag and our cards are already drag sources), and we have no ContextMenu primitive — only DropdownMenu. A visible ⋯ is discoverable, touch-native, and consistent with Linear/Notion (Jakob's Law). Optional later: right-click could be an additive accelerator that opens the same menu — but not for v1.

Q2 — top/bottom scope → resolved: current column

Confirm the assumption: Move to top/bottom operates within the task's current column. Cross-column reordering isn't a meaningful "top/bottom" concept and is fully served by drag-and-drop. Keeps the action predictable (recognition over recall).

Q3 — confirm dialog vs undo toast → resolved: confirm dialog

Confirm dialog for v1. Delete is irreversible and cascades (comments, checklist, attachments). An undo toast is friendlier for reversible/low-stakes actions but risks silent data loss if the user misses the 5s window or it's dismissed by a navigation. We reserve the undo pattern for Archive instead — Archive is reversible, so it deliberately gets no confirm. For review: if the team prefers undo-on-delete later, the optimistic deleteTask rollback fn already makes it cheap to add.
