Super Admin Role — UX Design

Security UX Design — Building on Requirements from Product Owner

1. Design Principles

  • Invisible when irrelevant — regular admins should not see disabled buttons or locked-out UI. Destructive actions simply don't appear.
  • No new badge — no "Super Admin" badge in the UI. The distinction is internal/security only. Both admin types see the same "Admin" badge.
  • Graceful degradation — if a regular admin somehow reaches a super-admin action (direct API call), the server action returns a clear error message.
  • Minimal surface change — only 3 UI touchpoints change. Everything else stays identical.

4. Data Flow

Query changes

Three places currently query .select("is_admin") for the current user. These need to also fetch is_super_admin:

  • src/app/(main)/members/page.tsx — passes isSuperAdmin to MemberDirectory → MemberCard
  • src/app/(main)/profile/[id]/page.tsx — uses isSuperAdmin for delete button guard
  • src/app/(main)/admin/page.tsx — passes isSuperAdmin to AdminTabs → UserCreditsTable

Prop threading

MemberCardisSuperAdmin: booleanGates delete button

5. Error States

Server-side rejection

If a regular admin somehow calls deleteUser or grantStarterCredits directly (e.g. via curl or manipulated form):

  • Server action returns throw new Error("Super admin access required")
  • RPC returns RAISE EXCEPTION 'Super admin privileges required'
  • UI shows toast error via existing catch block patterns

No special error UI needed — existing error handling covers this.
