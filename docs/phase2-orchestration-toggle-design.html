Phase II · Slice 1 · Design

A per-session switch for subagent orchestration

This slice adds one thing: a switch that lets a single Claude session run workflow steps the new way (a fresh subagent per step) while every other session keeps running them the current way. It's deliberately small so we can turn it on in production, test it on a real workflow, and carry essentially no risk to anything else.

1 · Why

Right now a workflow is run by one session that plays every role in turn. Before each step it calls set_agent_identity, which returns a persona's system prompt as tool-result text, and the session is expected to follow it. Two known weaknesses: the persona is just text the model may stop honouring, and over a long run the personas drift and bleed into each other.

The new approach removes the role-play entirely: each step runs in its own fresh subagent whose actual system prompt is that step's persona. Clean context per step, no drift. This slice doesn't build all of that — it builds the switch that lets us start using it, safely and selectively.

2 · What this slice delivers

  • A per-session setting, orchestration_mode, with two values: legacy (today's behaviour) and subagent (the new behaviour). Default is legacy.
  • A new tool, set_orchestration_mode, to change that setting for the current session only.
  • A branch in claim_next_step: when the session is in subagent mode it returns the new instructions; otherwise it returns exactly what it returns today.

Not in this slice: per-role tool permissions, any persona-adherence scoring, and any change to how steps are completed or enforced (that's already handled by the claim token and Fix 1).

3 · The setting is per session, and that's the whole point

"Session" here means one Claude connection — you typically have several open at once. Each connection already has its own identity record on the server; we add the mode to that same per-session record. So turning the switch on in one session changes that session and nothing else.

legacy default

Identical to today. Any session that never calls the new tool stays here. No action needed to keep it, and it's the value we fall back to whenever anything is unset or unexpected.

subagent

Opt-in, set explicitly in the one session you're testing. It persists for that session — including across token refreshes — until you set it back.

Turning it on

1
You, in the session under test
Call set_orchestration_mode("subagent").
2
Server
Records subagent on this session's record only. Other sessions have their own records and are untouched.
3
Server → you
Returns a clear confirmation: the new mode, and a note that it applies to this session only.

Switching back is the same call with "legacy".

4 · What changes when it's on

Only one thing changes: the instruction text that claim_next_step returns. The step itself, the claim token, the list of available agents, and the prior-step context are all identical in both modes. Keeping the change confined to the instructions is what makes this low-risk.

legacy — unchanged

"Before executing this step you must call set_agent_identity to switch to the {name} ({role}) persona…"

Exactly what it returns today.

subagent

"This session is in subagent mode. Do not call set_agent_identity. Keep the claim token. Fetch this step's persona with get_agent_prompt, spawn a fresh subagent whose system prompt is that persona plus this step's description plus the prior context, let it produce the deliverable, then you call complete_step with its result."

The subagent flow

Orchestrator (the session running the workflow) Subagent (spawned fresh for one step) Server
1
Orchestrator
claim_next_step → the step plus a claim token (it keeps the token).
2
Orchestrator
get_agent_prompt(bot_id) → the persona's system prompt for this step.
3
Subagent
Spawned with that system prompt + the step description + the prior context. Runs the step in its own clean context and returns the deliverable.
4
Orchestrator
complete_step(token, output) with what the subagent returned.
5
Server
Verifies the token, attributes the work to the step's agent, and passes the output to the next step as context.
The subagent never calls the MCP server itself — it just produces the deliverable and hands it back. The orchestrator keeps the token and does all the workflow transitions. That's why subagents need no credentials of their own, which is the "no API keys" point from earlier.

5 · Every case, and where it lands

Anything unset or unexpected resolves to legacy, so the safe path is always the default.

6 · The new tool

set_orchestration_mode

  • Input: mode — either legacy or subagent. Anything else is rejected before it does anything.
  • Effect: writes the mode onto the current session's record. It doesn't touch the active persona, the login, or any other session.
  • Returns: the new mode and a reminder that it's scoped to this session.

It mirrors the existing set_agent_identity tool — same per-session record, same "changes only this session" guarantee — so its behaviour is predictable.

7 · Three choices to confirm

  1. Tool name: a dedicated set_orchestration_mode, separate from set_agent_identity. They're different jobs, so I'd keep them separate.
  2. Parity: the subagent instruction should also tell the orchestrator to append a short summary to the task description after each step, the same as legacy does. I'd keep them consistent.
  3. No web UI this slice: tool-only, which fits testing quietly in production. Agreed?
Summary: a per-session, default-off switch that lets one session run the subagent-per-step flow in production, with everything unset or unexpected falling back to today's behaviour.
