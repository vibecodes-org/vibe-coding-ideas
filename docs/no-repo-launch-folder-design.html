No-repo launch folder — UX Design

Make a deep-link-launched local Claude Code session start in the correct project folder for no-repo projects, using an agent-self-reported absolute path.

Feature Development workflow · Step 2 of 6 (UX Design) · Dark Zinc / shadcn New York · WCAG 2.1 AA

Building on Step 1 (Requirements)

The requirements / prior launch work lives in docs/launch-claude-code-design.html and src/lib/launch-claude-code.ts. Two facts from that step are load-bearing here:

  • A web page can never read a folder's absolute path. Neither a file picker nor any browser API yields /Users/you/projects/my-idea. The browser is structurally the wrong place to learn or create the path.
  • Today, no-repo ideas launch in "new" mode: the deep link carries no cwd; the bootstrap prompt tells the agent to mkdir -p ~/projects/<slug> && cd. This works on first run but does not persist the resolved path, so VibeCodes can never pre-set cwd on later launches, and the old per-idea localStorage path is browser-local and unreliable.

This step closes that gap: the agent reports its own absolute path back to VibeCodes via MCP, so subsequent launches can inject it as cwd.

Design thesis — most of this feature is invisible

The substance is backend: a new MCP tool record_project_path(idea_id, hostname, absolute_path), a per-user / per-hostname path table, deep-link cwd injection, and bootstrap-prompt wording. The only pixels are a light status line on the existing launch control in src/components/board/launch-claude-code-button.tsx. We deliberately add no dialog, no blocking step, no new screen — the agent does the work; the UI just sets expectations and reflects what was recorded.

2 · Launch button states

The existing control is a ghost emerald split-button (label + chevron) on the board, plus a terminal-icon variant on task cards. We layer three states over it for no-repo ideas. Repo-backed ideas are visually unchanged (§5). The button label and primary action never move — Jakob's Law: people already know this button.

State A — First launch on this machine (no recorded path)

Board toolbar · hover state
Launch Claude Code First time here
First launch on this machine Claude Code opens in your home folder, sets up ~/projects/vibecodes, and remembers it. Next time it'll open straight in that folder.
  • Information scent. The First time here chip is a non-colour-only signal: it pairs an amber tint with a literal text label and a dot glyph, so it reads in greyscale and to screen readers. It only renders on desktop where launch is available.
  • Why no dialog. Hick's Law / progressive disclosure — there is nothing for the user to decide. The agent picks the folder. A modal here would be friction with no payload.
  • Tooltip, not a wall of text. The reassurance ("opens in home first, then remembers") lives in the tooltip on hover/focus, plus a short helper line in the dropdown (State A dropdown below). The button itself stays one short label.

State C — Recorded path, but folder is gone / unverifiable (edge)

VibeCodes cannot test the filesystem from the browser, so this state is not detected up front — the deep link still fires with the stored cwd. Two things make it safe:

  • Self-heal in the prompt (§7). Every launch re-runs pwd and re-records. If the folder moved or the cwd was stale, the agent lands wherever it can, re-creates / re-locates the project dir, and overwrites the record. The user usually never notices.
  • If the OS rejects the cwd (deep link opens but the directory no longer exists), the agent falls back to the same first-launch recovery: cd ~/projects/<slug> (create if missing) → pwd → re-record. The dropdown's "Forget this folder" is the manual escape hatch.
Optional toast — only if the agent reports it re-located the folder
Project folder was re-located
The old path wasn't found, so Claude Code set up ~/projects/vibecodes again and updated the record for Nicks-MacBook.

This toast is informational, not an error — the recovery already happened. It only fires if the agent's record_project_path call indicates the path changed from the stored value. No action button needed; recovery is automatic.

3.4 Multi-machine (per-hostname)

Path is keyed by hostname. The same idea can have a different recorded path per machine — laptop vs desktop, macOS vs Linux. The launch button only ever shows / injects the path for the machine you're currently on. There is no cross-machine path leakage and no "which machine?" picker — the agent reports its own hostname; VibeCodes matches on it.

3.5 Repo-backed projects — unchanged

An idea with a github_url already launches in "existing" mode: the deep link's repo param resolves the local clone (or clones it). No path recording, no first-time chip, no dropdown "This machine" row. The button looks and behaves exactly as today. record_project_path simply isn't invoked, and the prompt's repo directory block is untouched.

4 · Optional "recorded locations" affordance — recommended (light)

Recommendation: ship the read-only line + Forget, defer anything bigger

The single "This machine: ~/projects/<slug>" row in the State B dropdown (shown in §2) is worth shipping: it gives recognition-over-recall ("where will this open?") and a clean recovery path ("Forget this folder") for when someone reorganises their drive. It is one read-only menu row plus one destructive item — near-zero surface area.

  • Forget = clear the record for this idea + hostname. Next launch reverts to State A (first-launch flow re-records). Confirmation is a sonner toast with an Undo action (no modal), since the action is cheap and reversible by re-launching.
  • Deferred (explicitly out of scope for this step): a full multi-machine "manage all recorded locations" table (every hostname for an idea, edit/delete each). It adds a screen for a rare need; revisit only if users actually juggle 3+ machines per idea. Multi-machine functionality works regardless — this is only about a management UI.

Every state is explicit — no silent failures, no colour-only signals, no disabled control without an adjacent reason.

7 · Bootstrap-prompt UX contract

For this feature the prompt is the primary interface — it's what the agent reads and obeys. It lives in buildBoardBootstrapPrompt / buildTaskBootstrapPrompt via the directoryBlock / newProjectSteps helpers (src/lib/launch-claude-code.ts). The no-repo directory block must instruct the agent, in this exact order:

  1. Get into the project dir FIRST. If ~/projects/<slug> exists, cd into it and reuse as-is (don't overwrite). If not, mkdir -p ~/projects/<slug> && cd. Do this before writing anything.
  2. Confirm where you are. Run pwd and capture the absolute path it prints. This is the authoritative path — not a guess from the prompt.
  3. Record it. Call record_project_path with idea_id, the machine hostname, and the pwd result. Do this on every launch (self-heal), not just the first.
  4. Now, and only now, write project files. CLAUDE.md, .vibecodes/, scaffolding — all of it must land in the project folder. Never write project files while the working dir is home.

⭐ The ordering constraint is the whole point

If the agent writes CLAUDE.md before the cd, the file lands in ~ — exactly the bug this feature exists to prevent. The prompt must make "cd / create → pwd → record → THEN write" feel like one indivisible opening sequence. It's already structured this way (directory block is placed first and protected from length-truncation); we are adding the pwd + record_project_path steps between "cd" and "write files".

What the agent does, first launch (illustrative terminal)
~ $ mkdir -p ~/projects/vibecodes && cd ~/projects/vibecodes vibecodes $ pwd /Users/nick/projects/vibecodes # → record_project_path( # idea_id="62e5…e6dd", # hostname="Nicks-MacBook", # absolute_path="/Users/nick/projects/vibecodes" ) ✓ recorded vibecodes $ # only now: write CLAUDE.md, git init, scaffold…

Plain-language summary of the contract

"Before you do anything else: get into this idea's folder (create it under ~/projects/<slug> if it's not there), run pwd so we know exactly where that is on this machine, tell VibeCodes the path with record_project_path, and only then start creating files — so everything you write lands in the project, not in your home directory."

8 · Accessibility & reuse checklist

  • ✓ No colour-only signals. "First time here" chip = amber tint + literal text + dot glyph; toasts pair colour with an icon and a text title; dropdown markers use a glyph + word.
  • ✓ Contrast. Body text on --bg uses #d4d4d8/--muted-fg (≥4.6:1); emerald/amber/sky accents are used on text ≥14px or as ≥3:1 UI borders, not as the sole carrier of meaning.
  • ✓ Targets ≥ usable size. Split-button halves are 32px tall (matches existing toolbar density); dropdown items have generous hit areas; mobile menu items already use py-2.5.
  • ✓ Visible labels, not placeholder-only. Tooltip + helper rows are real text; the path is shown, not implied.
  • ✓ No content trapped behind hover only. Tooltip content is reachable on keyboard focus (Radix Tooltip); the same facts are duplicated in the dropdown, which is fully keyboard navigable.
  • ✓ No disabled button without a reason. Mobile shows a disabled menu item with the explanation "Open on desktop…". The primary launch is never disabled while loading — it just lacks the path chip until the lookup resolves.
  • ✓ No modal for this. Everything is inline (tooltip, dropdown, toast). The only existing dialog (advanced "Set exact folder") is untouched and stays opt-in.
  • ✓ Reuses design-system components: Button, DropdownMenu*, Radix Tooltip, sonner toasts, lucide icons (Terminal, ChevronDown, Trash). No new primitives.
