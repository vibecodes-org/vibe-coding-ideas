Step-Level Comments over MCP — DX design spec

UX Design step for "MCP gap: agents can't read step-level comments." No visual UI changes — the "users" here are AI agents consuming get_task and claim_next_step. This spec defines exact payload shapes, field placement, truncation semantics, and tool-description copy so the Implementation step can build it verbatim. Grounded in the current code: mcp-server/src/tools/board-read.ts (getTask, ~L204–308), mcp-server/src/tools/workflows.ts (claimNextStep ~L660–948, approveStep ~L1456–1471, addStepComment ~L1485–1514), mcp-server/src/register-tools.ts (descriptions L288, L1163, L1252), migration 00071_board_level_workflows.sql (table + comment_count trigger).

The scenario this design serves A human approves the "Requirements" gate with "Approved — but rename the field to retry_count, not attempt_no." Today that note lands in workflow_step_comments (type approval) and no agent ever sees it. After this change it surfaces in two places agents already look: the step objects in get_task, and a new approval_notes array (plus an APPROVER GUIDANCE instruction block) in every subsequent claim_next_step of the same run.
Human
approve_step with comment → row type: "approval"
Next claim
claim_next_step returns approval_notes[] + instruction block
Any time
get_task shows per-step comments[] (≤10, newest first)

Critical implementation fact The comment_count trigger (00071 L304–326) counts ALL rows including type="output" mirrors, and counts them pre-RLS. Since the API excludes output rows, comment_count > comments.length is normal and is NOT a truncation signal. That is why this design adds an explicit computed comments_truncated boolean — see §03.

03 Nailed-down design decisions

D1 — Author attribution: flat author_name via one join

Raw UUIDs alone fail the "who said this" question — an agent would need a second lookup it has no tool for. Join users(full_name) exactly as getTask already does for task-level comments (board-read.ts L227), but flatten to author_name: string | null instead of a nested author object: the shape is then identical in both new surfaces, and a flat scalar promises nothing more than it delivers (a nested object invites agents to probe for email, avatar…). Keep author_id for correlation with available_agents / identity checks. author_name is null only if the join misses (FK is ON DELETE CASCADE, so rows normally vanish with their author) — consumers must render "unknown", never drop the comment.

D2 — Ordering: newest-first for step comments, chronological for approval_notes

comments: newest-first, confirmed. Two reasons override "chronological reads better for threads": (a) the 10-cap must never evict the latest feedback — newest-first + slice guarantees recency survives truncation; (b) it matches the task-level comments array in the same get_task response (desc, limit 20) — one ordering rule per response. approval_notes: chronological (created_at asc) — it is an uncapped-small narrative of gates passed in run order, read top-to-bottom as accumulated constraints.

D3 — approval_notes scope & placement

Scope: ALL prior steps of the same run, not "since last claim." An approval caveat ("rename the field to retry_count") is a standing constraint on every downstream step — a since-last-claim window would silently drop it for step 4 because step 3 already saw it, and re-claims after context compaction would lose it entirely. Dedup is unnecessary: notes are rare, short (≤5000 chars), and capped at 20. Placement: top-level key immediately after context — same mental slot ("what prior steps hand me"), distinct from rework_instructions ("what I did wrong"), which stays byte-identical. Steps with no run_id, first steps, and the { done: true } early return are untouched; otherwise approval_notes is always present, [] when empty.

D4 — Truncation signal: explicit comments_truncated boolean

comment_count is retained (it rides along in select("*")) but is not the truncation signal: the trigger counts all types including excluded output mirrors, pre-RLS, so comment_count > comments.length is the normal state. The contract agents use: comments_truncated: true ⇔ more than 10 visible non-output comments exist for that step — computed server-side from the post-RLS, post-filter row set (groupedRows.length > 10). Recognition over recall: agents read one self-evident boolean instead of memorizing counting rules. Tool-description sentence (included verbatim in §05): "If a step's comments_truncated is true, older comments exist beyond the 10 returned; comment_count is a raw total that also counts hidden output-mirror rows, so never compare it to comments.length."

06 Tool-description copy — verbatim

Append the highlighted sentences to the existing descriptions in register-tools.ts (information scent: name the field, say where it is, state the one rule agents must not guess).

get_task (L288 — append)
"Get single task detail including workflow steps, comments, and recent activity. […] use claim_next_step to execute steps sequentially rather than implementing directly. Each workflow step carries its own `comments` array — the step's latest discussion, failure reports, and approval notes (newest first, max 10; type 'output' mirrors are excluded because the step's `output` field is canonical). If a step's `comments_truncated` is true, older comments exist beyond the 10 returned; `comment_count` is a raw total that also counts hidden output-mirror rows, so never compare it to comments.length — the full thread is visible in the web UI."
claim_next_step (L1163 — append)
"Claim the next workflow step on a task […] Returns the step with bot_id, available_agents, a `context` array of prior completed steps' outputs, and a one-time `claim_token` […] Also returns `approval_notes` — every note a human left when approving earlier steps in this run (empty array if none). Treat approval notes as binding constraints: 'approved, but change X' makes X a requirement for this and all later steps. Returns { done: true } when all steps are complete."
add_step_comment (L1252 — replace)
"Add a comment to a workflow step. Comments are surfaced to other agents: get_task returns each step's latest 10 (types comment/failure/approval/changes_requested), 'failure' and 'changes_requested' feed claim_next_step's rework_instructions, and 'approval' notes feed its approval_notes. Type 'output' is reserved for the UI mirror of complete_step's output — never use it to communicate with agents; put deliverables in complete_step's `output` parameter instead."

07 Explicitly NOT changing

  • rework_instructions — shape, queries (L661–720), and bytes untouched; approval_notes is a sibling, not a merge.
  • context[] — still built exclusively from step.output columns, never from comments (comment at workflows.ts L741–742 stays true).
  • instruction string when approval_notes is empty — byte-identical (same guarantee pattern as the model-tier and skills blocks).
  • comment_count column, trigger, and value — still counts all types; no migration needed (comments_truncated is computed, not stored).
  • add_step_comment write path and zod schema — description copy only.
  • Task-level comments in get_task (board_task_comments, desc, limit 20) — a different table, untouched.
  • approve_step insert path (L1456–1471) and its bot-caller block.
  • claim_next_step's { done: true } early return.
  • No new MCP tool, no pagination, no RLS policy changes, no web-UI changes.
