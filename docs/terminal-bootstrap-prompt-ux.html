In-App Terminal — Bootstrap Prompt Flow

The in-browser dock launch must start Claude Code with the same compact bootstrap prompt the terminal-window deep link already uses (buildCompactBootstrapPrompt in src/lib/launch-claude-code.ts). Per the ProdOwner Requirements handoff, the happy path is intentionally invisible — same button, same dock, claude simply wakes up already knowing the idea and task. This document therefore designs the system flow and the edge states only.

Designer: Compass (UX) Built on: ProdOwner Requirements (R1, R2, R4 / AC4, AC6, AC8) Surface: board bottom terminal dock — no new surface Palette matched to terminal-dock.tsx Companion: docs/install-first-terminal-ux.html

§1Happy path statement — nothing new to learn

Per the Requirements handoff: no new buttons, no new settings, no new screens. The user presses the same Connect (or a paired browser auto-connects), the same install-first flow from docs/install-first-terminal-ux.html runs unchanged — and the only visible difference is the terminal's first screen: claude opens already primed with the task instead of a blank shell.

✓ The whole visible feature

Today the in-browser terminal opens cold and the user must paste or type context. After this change it opens with the compact bootstrap prompt already delivered — claude connects the board tools, records the project path, and picks up the task on its own. The user's mental model doesn't change; the terminal is just smarter on arrival (Jakob's Law: keep the pattern they already learned, improve what it yields).

The only visible difference from today: the first message in the scrollback is the compact bootstrap prompt (shown in sky above as claude renders the user turn), and claude is already acting on it. Dock bar, pills, controls, and input bar are pixel-identical to the shipped terminal-dock.tsx.

Why the happy path is invisible on purpose
  • No new affordance to find — the deep-link (terminal-window) flow already primes claude this way; the in-browser flow is catching up to parity, not adding a capability the user chooses (recognition over recall: they recognise the same launch, it simply does more).
  • The prompt itself is the receipt. It lands in the scrollback as the first turn, so a curious user can read exactly what claude was told — full transparency without any explanatory chrome.
  • Human-in-the-loop preserved — the user watches claude execute the bootstrap live and can interrupt by typing at any point (the shipped read-write input bar), same as any terminal session.

§2System flow — where the prompt travels

The prompt is built in the browser, rides the vibecodes://launch deep link as an inert string, and is held by the helper until the relay accepts the owner-bound token (the R1 gate, amber below). Only after acceptance does the helper pass it to claude. Rejected token ⇒ the prompt is discarded unexecuted.

§3 — no new UI

The prompt crosses three trust boundaries (web → OS link → helper → relay-authenticated bridge) but is only ever data until the amber gate passes. This ordering is the whole of R1: a crafted link on a victim's machine can make the helper open, but never make anything run.

§3Edge A — pre-auth rejection (R1 / AC4): no new UI

A forged, expired, or foreign-account link fails at the relay gate: the helper attaches nothing and exits quietly. On the machine, nothing happens — which is the security win. In the dock, the launch simply never streams, so the shipped install-first flow already covers it: Connecting… → (~8s) → the calm timeout fallback.

This is the already-shipped timeout fallback from the install-first design — reused verbatim. Zero new strings, zero new states.

Why no new “security” state — deliberate decision
  • The legitimate user can't reach this state through the product. Every dock launch mints a fresh valid token milliseconds before firing. Expired/forged/foreign tokens arrive only via stale or crafted links opened outside the dock — and then there is no dock waiting, so there is nothing to show. When a dock is waiting and the helper doesn't stream (whatever the cause), “Didn't connect yet → Retry” is both accurate and the correct recovery: Retry mints a fresh token, which fixes the token-expiry case by construction.
  • An alarm state would misfire. The dock cannot distinguish “token rejected” from “helper not running” (the helper never streams in either case). A red “security” banner on what is usually a sleeping helper violates the no-alarm-for-non-actionable-info rule and trains users to ignore red.
  • Don't brief the attacker. Distinct rejection UI would give a probing attacker a feedback channel. Silence on the machine + a generic “not yet” in the dock leaks nothing.
  • Browser-leg failures are already covered — the dock's own socket has explicit error states (“Couldn't verify this session”, “This bridge belongs to another account”) in terminal-dock.tsx; those are untouched and out of scope here.

§4Edge B — truncated prompt (R2 / AC6): no in-dock signal

On very long task titles/context the URL cap forces deterministic truncation — but enforcePromptLength always preserves the MCP-setup head, so claude still connects the board tools and re-fetches the full task from the board itself (get_task). The trimmed tail was a convenience copy, not the source of truth. Recommendation: no dock UI at all.

The …(truncated) marker (already appended by enforcePromptLength) is visible in the scrollback for the curious. That is the entire signal.

Why silence is correct here
  • Nothing is lost. The head (project dir + MCP connect + record path + task id) survives by contract; claude reads the authoritative task body from the board over MCP anyway. Outcome for the user: identical to the untruncated case.
  • Non-actionable ⇒ no alarm. There is no user action that improves anything (“shorten your idea title” is absurd advice). A banner would add a decision point with zero payoff — pure Hick's-Law cost, and it would fire on exactly the boards with the richest context, punishing power users.
  • Never a dropped launch (R2). Truncation is the overflow behaviour; the launch always proceeds. A warning would imply something failed when nothing did.
  • Escape hatch exists for the 1%: the marker in the scrollback is honest and findable; anyone who cares can also use the copy-command flow, which carries the full verbose prompt with no URL limit.

§5Edge C — version skew (R4 / AC8): one low-key hint, feasibility-gated

New web + old installed helper: the old helper ignores the unknown prompt param and claude opens cold — exactly today's behaviour, so nothing breaks (R4). But unlike Edge B, here the user silently loses real value they can actually recover with one action: update the helper. This is the one place a small piece of new UI earns its keep — a single dismissible hint line, built exactly like the dock's shipped amber “Lost the connection” strip, on the calm sky accent.

The hint sits between the terminal and the input bar — same slot and construction as the shipped amber reconnect strip, so it reads as “dock status”, not an error. The session above it is fully usable.

Behaviour & why
  • Why hint at all (vs Edge B's silence): here the loss is real (claude opens cold; the user must hand-type context) and the fix is a single user action. Actionable + recurring on every launch until fixed ⇒ worth one quiet line. Non-actionable Edge B stays silent.
  • Never blocks, never modal. The session is connected and fully usable; the hint is informational (role="status", not alert). “Get the update” opens the existing /download/terminal-helper route — same link, same styling family as every helper-download affordance in the dock.
  • Dismiss is remembered (localStorage, per browser) so it shows at most once per user decision, not once per session — no nagging. It naturally disappears for good once the updated helper delivers the prompt.
  • Feasibility gate: the hint requires the dock to know the helper is old. Proposed detection: the new helper announces its version in a TEXT control frame on attach (the dock already ignores TEXT frames today, so old helpers are safely mute) — no version frame before first PTY bytes ⇒ old helper ⇒ show the hint. If engineering judges this signal unreliable, ship Edge C silent (graceful cold launch is already the R4 requirement); the hint is an enhancement, not a dependency.
  • WCAG: sky text on near-black ≥4.5:1; “Get the update” and “Dismiss” are real links with ≥44px effective touch targets (padded); icon + bold lead-in + text — never colour alone.

§8Design-system, responsive & scope notes

Reuse & consistency
  • All frames use the shipped dock chrome from terminal-dock.tsx — #141417 header, #0c0c0e body, zinc-700 borders, emerald/amber/sky/rose status accents, the same pills and input bar.
  • The Edge C hint is structurally the existing bottom strip (amber “Lost the connection” bar) re-skinned to the sky accent — same slot, same padding, same inline-link pattern. No new component class.
  • Edge A reuses the install-first timeout fallback screen and copy with zero changes.
375px behaviour
  • The Edge C hint wraps to two lines at phone width; “Get the update” and “Dismiss” keep ≥44px padded targets and never truncate. Everything else in this design is existing, already-verified dock chrome.
Scope guard (no gold-plating)
  • No “task context attached ✓” badge, toast, or preview panel on the happy path — the prompt in the scrollback is the receipt.
  • No truncation warning (§4 justification) and no distinct security state (§3 justification).
  • The Edge C hint is the only new UI, and it is explicitly droppable if version detection proves unreliable — the feature ships whole without it.
