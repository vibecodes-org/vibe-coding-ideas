UX Design · step 2 of 6 (Feature Development) · Compass

Component: src/components/ideas/github-link-dialog.tsx · Building on the Requirements step (FR-1–FR-4, V1–V6, NFR-1/2) — those are fixed inputs; this document designs the visual + interaction layer only.

TL;DR — decisions for the reviewer

  • Three peer tabs replace today's footer text-link: My repos · New repo · Paste URL. All three labels are visible at initial render with equal weight — this is what makes the "any repo" path first-class (FR-1/FR-2) rather than the buried footer link it is today.
  • Smart default tab mirrors the Personal Template Library pattern: connected → My repos (FR-3 unchanged); not connected → Paste URL (the only path that works without OAuth); editing an existing link → Paste URL prefilled.
  • One clean gating rule: URL format gates the Save button (V5 is the only block); repo reachability only informs. Verification never disables Save.
  • Feedback lives directly under the input (Gestalt proximity), not in a top-of-screen toast. Every V-state = icon + coloured panel + text — never colour alone (WCAG 1.4.1).
  • FR-4 caveat is always-on helper copy under the input, before any verification runs: repo can be owned by anyone; cloning uses the user's local Git credentials.
  • Non-blocking checking state: subtle inline spinner row, 500ms debounce + on-paste + on-blur, 5s timeout → V6. Typing never blocks (NFR-1/2).

01 · Current-state analysis — what's wrong today

Reading github-link-dialog.tsx as it stands, five problems block the requirements:

  • The "any repo" path is literally in the footer. Line 422–428 renders paste a URL instead as a 12px muted text-link in DialogFooter, styled text-muted-foreground. This is the definition of a buried power-user affordance — it reads as a fallback, directly contradicting FR-1/FR-2.
  • No-connection users hit a dead end. Lines 238–249: when there's no GitHub connection and mode isn't manual, the dialog shows only a "Connect GitHub" gate. A user who just wants to paste someone else's URL must first notice the footer link. The working path is hidden behind the one thing they're trying to avoid.
  • Manual mode has no verification at all. handleManualSave (line 197) runs a client-side parseRepoUrl regex and throws a binary toast.error. There is no V1–V6 behaviour: no resolved owner/repo confirm, no private note, no 404 caution, no rate-limit path. This is the entire feature gap.
  • Feedback is a toast, far from the input. Errors surface at the top-right of the viewport via sonner — maximal distance from the field the user is editing. Violates proximity; poor error recovery.
  • Copy under-serves FR-4. Line 415 says only "Paste any github.com repo URL — public or private." It never states the repo can be owned by anyone, nor that cloning depends on the user's local Git credentials.
Minor layout bug worth noting to Engineering: when mode === "manual" and a connection exists, the connection ? <Tabs> block (line 250) and the manual block (line 403) both render, so the Browse tabs and the URL input stack together. The tab redesign removes this by making the URL path a real tab panel.

02 · Design rationale — why three tabs, not a two-zone split

The requirement is explicit: the URL path must "read as first-class… not a buried power-user tab." The trap is reading "not a tab" too literally. What makes something buried is visual demotion (a footer link, an overflow menu, a low-contrast secondary), not the tab pattern itself.

  • Jakob's Law: segmented tab groups for "pick your input method" are the convention in GitHub, Linear and Notion import dialogs. Users already know how to read three equal segments.
  • Hick's Law: three choices is a trivial decision surface; a full always-on second zone plus tabs would add clutter and a second scanning axis on a 375px screen.
  • First-class by equal weight: all three tab labels render immediately, same size, same contrast, same row. None is hidden behind hover or an overflow. The "Paste URL" tab carries a link icon for scent and its panel headline is the requirement's own phrasing — "Working in someone else's repo?"
  • Smart default finishes the job: when the user can't browse (no OAuth), we open on Paste URL. So the disconnected user — the exact person FR-1 protects — lands on the working path with zero clicks.

08 · Accessibility notes (WCAG 2.1 AA)

  • Never colour alone (1.4.1): every V-state pairs a distinct icon + a text title with the colour. V5 additionally reddens the input border.
  • Live regions: panel is role="status" aria-live="polite"; V5 uses role="alert" aria-live="assertive". Input gets aria-invalid="true" + aria-describedby pointing at the message id on V5.
  • Contrast: state text uses the -400 shades (blue/emerald/amber/red) on zinc-950 for titles and --foreground/--muted-fg for body — all ≥4.5:1 for the 12.5–14px text; icon/border UI elements ≥3:1 (1.4.3 / 1.4.11).
  • Visible labels: "Repository URL" is a real <label>, not a placeholder (2.5.3 / 3.3.2). Placeholder is example-only.
  • Disabled button has adjacent reason: Save is disabled only at V5, where the error text directly above it is the explanation. Uses aria-disabled so it stays focusable and its state is announced.
  • Keyboard: Radix Tabs give arrow-key tab navigation + roving tabindex out of the box. Full tab order: tabs → input → Cancel → Save. Enter in the field triggers verify/Save when valid.
  • Touch (2.5.5): tab triggers, list rows and buttons are ≥44px tall on mobile.
  • Reduced motion: the checking spinner respects prefers-reduced-motion — falls back to a static "Checking repo…" text row.

09 · Open questions for Design Review

  • V3 privacy disclosure. Confirming "Private repository — visible to your account" tells the user a private repo exists at that path. Since resolution runs with their own token they could see it on GitHub anyway, so this seems safe — but flagging it explicitly for sign-off.
  • Verify triggers. Proposed: on-paste + on-blur + 500ms debounce. The dominant real action is paste, so on-paste matters most — OK to confirm we don't also fire on every keystroke?
  • V6 reset timer. Requirement says keep V6 neutral; confirm we intentionally omit any "try again in Xm" from the rate-limit reset header (available on GithubApiError.rateLimitResetAt).
  • Editing an existing link → default tab. Proposed: open on Paste URL prefilled. Alternative: open on My repos so users can re-pick. Which is the expected mental model?
  • Auto-switch on paste. If a user pastes a full GitHub URL into the Browse filter, should we offer to jump to Paste URL? Nice-to-have; recommend deferring to a follow-up.
  • Button relabel on V4. "Save anyway" vs. keeping plain "Save". The relabel signals the caution; confirm it's not over-warning for a supported, common case (linking a teammate's repo).
