Overview & Flow Architecture

Building on the Requirements step: replace the current 5-step wizard (Welcome → Profile → Pick Agent Team → Create Idea → Success) with a 6-step Golden Path (Welcome → Profile → Your Project → Your Board → Connect MCP → You're Live). The key UX shift is from "choose agents, then figure things out" to "describe what you're building, and we set everything up."

Key UX Principles

  • No dead ends: Every step advances toward a populated board. Skipping still completes onboarding.
  • Show don't tell: Board preview with real data beats abstract descriptions.
  • Progressive disclosure: Complexity revealed only when relevant (kit details after selection, MCP after board ready).
  • Consistent skip affordance: Every step has "I'll do this later" in identical position — bottom-center, ghost text style.
  • Non-dismissable: No close button, no outside click dismiss, no escape key. Only skip or complete.

Board preview design decisions

3-column preview shows Backlog, To Do, and In Progress only — enough to prove the board has real content without overwhelming the dialog. On mobile (<480px), columns stack vertically showing only the first 2 tasks each.

"+N more" indicator when a column has more than 2 tasks — prevents the preview from growing too tall.

Confirmation badges show three key results: agents allocated, workflow applied, auto-rules active. These map directly to what the kit provides.

No back button — this step is post-creation, going back doesn't make sense. The "I'll do this later" skip calls completeOnboarding() and navigates to the board.

Failure fallback: If board generation failed, hide the board preview and show: "Your idea is created. Go to your board to generate tasks with AI." with a "Go to your board →" CTA.

Interaction Flows & State Management

Step Navigation Rules

  • Steps 0–2: Back button enabled, can navigate backwards freely.
  • Steps 3–5: No back button — these are post-creation, forward-only.
  • Skip from any step: Calls completeOnboarding() and closes the dialog. Router refresh follows.
  • Skip from Step 2: Creates no idea. Calls completeOnboarding(), navigates to dashboard.
  • Skip from Steps 3–4: Calls completeOnboarding(), navigates to the board (idea already exists).

Step 2 → Step 3 Transition (the critical path)

This is the most complex transition. On "Create & Generate Board →" click:

  1. Show loading state (replace form content, keep step indicator on step 2)
  2. Call createIdeaFromOnboarding({ title, description, tags: [], kitId, visibility })
  3. On success: store ideaId and kitResult in state
  4. Call generateBoardFromOnboarding(ideaId) (new server action, uses platform key)
  5. On success: store taskCount, advance to Step 3
  6. On generation failure: still advance to Step 3 with generationFailed = true
  7. On idea creation failure: show toast error, return to form (do NOT advance)

Accessibility

Keyboard Navigation

  • Kit chips: role="radiogroup" with arrow key navigation (reuse ProjectTypeSelector pattern). Tab moves to the group, arrows move between options.
  • Visibility toggle: role="radiogroup" with two role="radio" buttons.
  • All buttons: Focusable, keyboard-activatable. Tab order follows visual order.
  • Skip links: <button> elements (not <a>) — keyboard-accessible without href.
  • Step indicator: aria-label="Step {n} of 6" on the container, individual dots are decorative (aria-hidden).

Screen Reader

  • Dialog title: sr-only text "Welcome to VibeCodes" (existing pattern).
  • Kit summary: aria-live="polite" region — announces when kit selection changes.
  • Loading progress: aria-live="polite" on the progress checklist — announces each completed step.
  • Board preview (Step 3): aria-label="Board preview with {n} tasks across 3 columns".
  • Terminal block (Step 4): role="region" aria-label="MCP connection command".

Design Decisions & Rationale

Why chips instead of grid cards for kit selection?

The existing ProjectTypeSelector uses a grid of cards. In the constrained dialog (580px), a grid of 6 cards would require 2 rows of 3, consuming ~250px of vertical space. Compact chips in a flex-wrap layout fit in ~80px, leaving room for the form fields, kit preview, and enhance CTA without scrolling.

Implementation note: Reuse ProjectTypeSelector component but pass a new variant="chips" prop (or use the existing compact mode with CSS override). The existing radiogroup keyboard navigation transfers directly.

Why remove tags from onboarding?

Tags add friction without clear value during onboarding. The kit already categories the project type. Tags can be added later from the idea detail page. This saves ~60px of vertical space.

Why forward-only after Step 2?

Once the idea and board are created, going "back" would mean... what? Recreating the idea? That creates confusing state. Forward-only post-creation is simpler — if users want to change something, they can do it from the board page.

Why show a board preview instead of redirecting to the board?

Redirecting breaks the wizard flow and loses the context of what just happened. The preview gives users a "proof of work" moment — they can see the result before the MCP step. The board page is one click away from Step 5.

Why not auto-detect MCP connection?

MCP connection detection relies on mcp_connected_at which is only set on the first successful tool call. Polling for this during onboarding would require the user to complete the MCP auth flow in another tab and then make a tool call — too complex. The "I've connected" button is a trust-based approach. The first-run dashboard will show the real connection status after onboarding.
