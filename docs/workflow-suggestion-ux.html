Mismatched-Workflow Suggestions — UX Design

When an auto-rule matches a task but the workflow template looks wrong for it, we no longer apply it silently. Instead the task carries a suggestion the owner resolves with Keep / Replace / Remove. This document defines how that surfaces on the board and in task detail, all interaction states, the mobile layout, and accessibility.

Step 2 · UX Design Persona: Compass (UX Designer) Revised for Requirements v2 · AI-hybrid + v3 agent pick-up protocol shadcn · Zinc · dark
Building on Requirements v2 (ProdOwner) — AI-hybrid detection.

Per Requirements v2, detection is now a hybrid pipeline: a cheap keyword pre-filter runs synchronously, and for suspect cases an AI adjudication runs async (Realtime-delivered) returning a recommended template + confidence + one-line rationale. High confidence (≥0.85) auto-applies silently; uncertain → a suggestion; AI unavailable → falls back to the keyword heuristic. This design still implements US-1/US-5 (resolvable suggestion + visible reason), US-2 (confident matches auto-apply), US-4 (DB-backed, survives reload), US-6 (label removed → suggestion disappears), and follows FR-6b for the heuristic reason string. New for v2: a suggestion's reason can be AI or heuristic — the UI must show which (provenance), the AI's recommended fit is pre-selected in Replace, and the async arrival is surfaced honestly. Card indicator distinct from an active run (FR-14); resolution inline, not a modal (FR-15).

1b · The async micro-state — “checking fit…” (NEW, v2)

Because AI adjudication runs async, the suggestion arrives a moment after the label is added (delivered over Realtime), not instantly. We surface that honestly with a brief, low-key transient state — not a blocking spinner. It resolves into one of three outcomes: a silent auto-apply (≥0.85), a suggestion appearing, or — if AI is unavailable — the heuristic result with no AI wait.

2 · Task-detail suggestion panel (default state)

Inline at the top of the task’s workflow area — the place the running-workflow UI normally lives, so the user looks in one consistent spot. Not a modal (keeps the task context visible; works on mobile). New in v2: a small provenance badge states whether the reason came from ✨ AI or the ⛭ Rules heuristic — icon + text, never colour-only.

2b · Heuristic-sourced suggestion (source = heuristic — AI unavailable / fallback)

When AI is unavailable or the case stayed in the keyword path, the reason is the original category/keyword explanation (FR-6b). The badge reads ⛭ Rules so the user knows it’s a rule-of-thumb, and there is no AI-recommended template to pre-select.

  • Provenance (v2): the badge is ✨ AI or ⛭ Rules — icon + word, so the source is legible without colour. It sits next to the heading, before the reason it qualifies.
  • Confidence stays human: we phrase it (“likely mismatch”) rather than exposing the raw 0.85 threshold or a percentage — confidence is internal.
  • Information scent: “Keep & attach” says exactly what happens (a workflow gets attached). Not “OK”.
  • Fitts’s Law: Keep is leftmost/primary (emerald, filled); Dismiss is text-weight and last, so the destructive-ish action is hardest to hit by accident.
  • The reason (FR-6b/US-5) is the AI rationale when source=ai, else the heuristic’s two categories + matched keywords — either way explainable and debuggable in the wild.

3 · Replace — template picker (AI-recommended pre-selected)

Progressive disclosure: “Replace…” expands the picker in place (still no modal). Scoped to the idea’s templates (Requirements OQ1). v2 change: when source=ai and a recommended_template_id exists, the picker opens with that template already selected and pinned to the top, labelled ✨ AI recommends. So Replace becomes a one-click accept of the better fit. The originally-labelled template (from the auto-rule) still appears, marked so the mismatch relationship is explicit.

How the two templates relate

  • Pre-selected (top): the AI-recommended better fit — violet ring + ● glyph + ✨ AI recommends pill (icon + text, not colour-only). Enter/click = attach it.
  • “Originally labelled” (dimmed): the template the auto-rule would have attached — selecting it is identical to Keep, so Replace can fall back without leaving the picker.
  • Heuristic-sourced suggestions have no AI pick, so the picker opens with nothing pre-selected (the “originally labelled” row is still tagged for reference).

Empty / edge states

  • No other templates: empty state — “This idea has only the labelled template. Create a workflow template or Dismiss instead.” Never a dead end.
  • AI pick already in the list: never duplicated — it’s the same row, just pinned + labelled.
  • > 8 templates: list scrolls; a filter input appears (Hick’s Law). The AI pick stays pinned above the filter results.

Auto-dismiss (label removed before resolving — US-6 / AC-8)

If the triggering label is removed while the suggestion is open, the panel and card chip simply disappear (the suggestion was only relevant to that label). A non-blocking, low-emphasis trace appears in the task activity log: Suggestion dismissed — “Research” label removed. No toast, no interruption — the user’s own action already explains it.

6 · Interaction flow (hybrid pipeline, v2)

  1. Trigger + sync pre-filter. A label is added (board UI, MCP create_task, or retroactive). The cheap keyword pre-filter runs synchronously. Clear match → normal auto-apply (existing path, nothing new). Clear non-issue → nothing. Suspect → continue.
  2. Async adjudication. For suspect cases, AI adjudication is queued and runs async; meanwhile the card/panel shows the low-key “checking fit…” micro-state (§1b). No run, no suggestion committed yet. If AI is unavailable, skip to the heuristic verdict (no wait).
  3. Verdict (Realtime). AI returns template + confidence + rationale. ≥0.85 → silent auto-apply (running UI appears, no decision asked). Uncertain → write a workflow_suggestions row with source=ai, recommended_template_id, rationale. AI down → heuristic suggestion, source=heuristic. Delivered to any open board via Realtime — the micro-state resolves in place.
  4. Inspect. User opens the task (or clicks the 💡 chip) → inline panel with the provenance badge (✨ AI / ⛭ Rules), template name + reason.
  5. Decide. Keep & attach → applies the labelled template (one-click) · Replace… → in-place picker, AI pick pre-selected when present → attach chosen · Dismiss → resolves with no run, label stays.
  6. Confirm. Optimistic update → toast on success → panel collapses; on Keep/Replace the running-workflow UI replaces it (chip 💡→▶); on Dismiss the panel just closes.
  7. Recover. On error, roll back to the open suggestion + Retry. On label-removed before resolving, auto-dismiss + low-emphasis activity trace.

Agent-run tasks — pause & surface (decided)

For an autonomous agent run, an uncertain match does not auto-Keep. The agent pauses and surfaces the suggestion exactly as for a human (DB-backed, US-4), and waits for a human resolution before any workflow is attached — resolution stays human-only for v1. A high-confidence (≥0.85) match still auto-applies for agents, same as humans. So in the flow above, step 3’s “uncertain” branch on an agent-run task lands in the same suggestion state and simply blocks the agent until resolved (it can pick up other steps meanwhile). See “Agent pick-up — what the agent sees” below for the exact MCP responses.

Agent pick-up — what the agent sees (Requirements v3)

A suggestion is not an applied workflow — there are no workflow steps yet. So when an AI agent (Claude Code) picks up a ticket whose only workflow state is an open suggestion, claim_next_step must not silently return “done / nothing to claim” (which would read as “nothing to do” and risk the agent starting work blind). This is a protocol/behaviour note — no new pixels — but it’s what the human-facing pause & surface above looks like from the agent side.

  • get_task surfaces it. The response now includes open workflow_suggestions, and when there’s a suggestion and no steps it returns a suggestion-aware instruction: “do NOT begin work; surface this to a human to Keep / Replace / Remove; agents cannot resolve it.”
  • Precedence: if there is a claimable step (an active run), normal claim always wins — the suggestion only blocks when there is no claimable step.
  • Human-only resolution: agents may read a suggestion but cannot Keep / Replace / Remove it — there is no MCP tool to resolve a suggestion in v1.

The agent can get back one of three shapes from claim_next_step:

💡 Blocked on suggestion
{
  "blocked_on_suggestion": true,
  "suggestion": { … },
  "instruction": "Do not begin work. Surface to a human to Keep / Replace / Remove; agents cannot resolve."
}

Returned instead of done:true when the only workflow state is an open suggestion. The agent pauses and surfaces it.

✨ Adjudication pending
{
  "adjudication_pending": true,
  "retry_after_seconds": 5,
  "instruction": "Verdict not in yet — re-claim shortly."
}

Transient: the async AI verdict (§1b) hasn’t landed. The agent re-claims after the hint. Bounded by a timeout that then falls back to the heuristic verdict.

▶ Normal step
{
  "step": { … },
  "bot_id": "…",
  "context": { … }
}

An active run exists with a claimable step — unchanged existing behaviour. Wins over any suggestion (precedence above).

In short: blocked-on-suggestion (pause & surface, human resolves), adjudication-pending (re-claim shortly, timeout → heuristic), or a normal step (proceed). The agent never silently no-ops on a suggestion-only ticket, and never resolves the suggestion itself.

7 · What the AI hybrid changes (vs the v1 design)

Per Requirements v2, the detection backend changed from a single keyword heuristic to a hybrid pipeline. The resolution UI (Keep / Replace / Remove, inline, DB-backed) is unchanged — these are the only UX deltas:

  • Provenance badge (§2): every suggestion now states its source — ✨ AI (shows the AI’s one-line rationale) or ⛭ Rules (shows the category/keyword explanation). Icon + text, never colour-only.
  • AI pick pre-selected in Replace (§3): when source=ai and a recommended_template_id exists, the picker opens with that template pinned + selected (“✨ AI recommends”), making Replace a one-click accept of the better fit. The originally-labelled template still appears, tagged.
  • Async “checking fit…” micro-state (§1b): because AI runs async, the suggestion arrives after the label via Realtime — surfaced as a calm, non-blocking transient state that resolves into auto-apply, a suggestion, or the heuristic fallback.
  • Subtle confidence: phrased in words (“likely mismatch”), never the raw 0.85 threshold or a percentage.
  • Unchanged: three-card distinction, inline panel (no modal), Keep/Replace/Remove, all resolving/success/error states, auto-dismiss on label removal, mobile 375px, full keyboard + AA contrast.

8 · Accessibility notes

  • Region: panel is role="region" aria-label="Workflow suggestion"; the async micro-state is role="status" aria-live="polite" so its arrival/resolution is announced without stealing focus; resolving sets aria-busy; success/error via aria-live="polite".
  • Provenance & AI recommendation are conveyed by text (“AI”, “Rules”, “AI recommends”) plus an icon — not colour. Screen readers read the source; the violet tint is reinforcement only.
  • Keyboard: card chip is a button (Enter/Space). In the panel, focus order is Keep → Replace → Dismiss; the picker is an arrow-key listbox (↑↓ + Enter), Esc = Back. When the AI pick is pre-selected it’s the initially-focused option, so Enter immediately attaches it.
  • Contrast: amber and the violet-tinted AI badge text/borders chosen ≥4.5:1 (text) / ≥3:1 (UI) on the zinc surfaces; every chip pairs icon + word so colour is never the sole signal.
  • Motion: the “checking” dots are subtle; honour prefers-reduced-motion by holding them static (the text label alone conveys the state).
  • Focus visible: 2px blue ring, 2px offset on every interactive element.
  • Labels: all buttons have text labels (icons are decorative aria-hidden).

9 · Decisions & residual open questions

Decided (folded into this design)

  • Keep = one-click (reversible — the run can be removed). No confirm step.
  • Card indicator = outlined amber footer chip (calm, footer placement).
  • Bulk “N to review” = deferred — out of scope for v1, not designed here.
  • Agent-run uncertain match = pause & surface (see §6) — never auto-Keep; waits for human resolution.
  • AI confidence threshold = 0.85, internal — surfaced only as human wording, never a raw number.
  • Auto-dismiss on label removal keeps a low-emphasis activity-log trace for auditability.

Residual for Design Review

  • Micro-state on dense boards. Is the “checking fit…” chip too much visual noise if many cards are adjudicating at once after a retroactive apply? Option: cap concurrent visible chips or debounce. Flag if retroactive bulk is common.
  • AI rationale length. The reason box shows the AI’s one line verbatim — do we need a hard truncation/“show more” if the model returns something long? Proposed: clamp to 2 lines + expand.
  • Provenance wording. “⛭ Rules” vs “Keyword match” vs “Heuristic” — which reads clearest to non-technical owners?
