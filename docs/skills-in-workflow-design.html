Skills in the Workflow Loop — Interaction Contract & Flow

UX Design step (Compass) · revised after board-owner rework feedback. This is a backend MCP change: the "user" is the workflow orchestrator (Claude Code) + the spawned subagent persona — not a GUI. So this document designs the interaction contract: the shape of the MCP tool responses and the orchestration flow from trigger → completion → error recovery.
Building on the Requirements step (not repeated here). Requirements defined: the optional agent_id parameter on get_agent_skill_content, the available_skills advertisement on claim_next_step, instruction-only loading (no server-side injection), the back-compat / no-regression constraints, and 13 acceptance criteria. Below is the concrete contract + flow that satisfies them. ACs are cited inline as AC#n.
Rework applied — model inversion

The previous revision had the orchestrator fetch (get_agent_skill_content) and fold each relevant skill body into the subagent prompt before spawning. Per rework feedback, that is now the Fallback only. The Primary model is: the orchestrator only advertises — it forwards available_skills + the agent_id + a subagent-addressed directive — and the subagent self-loads its own skills, on demand, while doing the work, using its own get_agent_skill_content MCP tool. The orchestrator never reads or folds skill bodies in the primary path.

1Flow — a workflow step executing, skilled vs skill-less

Three actors: ■ Orchestrator (Claude Code, advertises only), ■ Subagent (fresh persona context, self-loads skills), ■ MCP server. The branch point is the claim_next_step response.

Orchestrator → Server claim_next_step({ idea_id }) — picks up the next pending step.
↓
Server → Orchestrator (response) Returns step, one-time claim_token, available_agents, context, expected_deliverables, instruction — and now available_skills + the agent_id (the step's bot_id) + a subagent-addressed SKILLS directive iff step.bot_id owns skills.

Fallback path — orchestrator pre-loads + folds in only when the subagent has no get_agent_skill_content tool

Some clients spawn subagents that do not inherit MCP tools — those subagents cannot call get_agent_skill_content themselves, so the primary self-load model is impossible for them. This mirrors the existing "no subagent capability" EXCEPTION already present in the claim instruction (where the orchestrator executes the step directly instead of spawning).

Orchestrator — RELEVANCE GATE (on behalf of the subagent) For each advertised skill, judge relevance to THIS step.
↓  (only for relevant skills)
Orchestrator → Server get_agent_skill_content({ agent_id: "<bot_id>", skill_name }) — the orchestrator fetches because the subagent can't.
↓
Orchestrator Folds the returned instructions into the subagent's system prompt before spawning.
↓
Subagent Executes the step with persona + pre-folded skill instructions. Makes no skill calls.
↓
Orchestrator → Server complete_step({ step_id, claim_token, output })

Same MCP contract, same tool, same call shape — only who calls it shifts from subagent to orchestrator. The server is unaware of which path ran.

Trigger → completion: trigger = a pending step exists; completion = complete_step succeeds. Error recovery threads through both paths (claim_token re-claim, not-found skill fallback) — see §4.

2get_agent_skill_content — contract (before → after)

Add an optional agent_id (uuid). Resolution becomes .eq("bot_id", agent_id ?? ctx.userId). Supplying agent_id means the caller (a subagent that never ran set_agent_identity) can load skills directly; omitting it preserves today's identity-based behaviour. AC#2

BEFORE (today)
schema = {
  skill_name: string   // required
}

// resolves on the ACTIVE identity
.from("agent_skills")
 .eq("bot_id", ctx.userId)
 .eq("name", args.skill_name)
 .maybeSingle()

Requires set_agent_identity to have run first — a subagent has no such ambient identity.

AFTER (proposed)
schema = {
  skill_name: string,  // required
  agent_id?: uuid      // NEW · optional
}

const targetBot = args.agent_id ?? ctx.userId;
.from("agent_skills")
 .eq("bot_id", targetBot)
 .eq("name", args.skill_name)
 .maybeSingle()

With agent_id: no identity needed (the subagent self-loads). Without: falls back to ctx.userId → back-compatible.

Two call shapes (same tool)

With agent_id — new flow (subagent self-load)
get_agent_skill_content({
  agent_id: "a1b2-...-bot",
  skill_name: "api-design-review"
})
// ↳ resolves bot_id = a1b2-...-bot
// ↳ no set_agent_identity required
Without agent_id — back-compat
get_agent_skill_content({
  skill_name: "api-design-review"
})
// ↳ resolves bot_id = ctx.userId
// ↳ same as today

Not-found error — concrete copy for BOTH cases

Error must name the resolved target so the caller can self-correct. Provide a concrete, actionable message — never an abstract "not found".

Case A — agent_id supplied, no row:
Skill "api-design-review" not found for agent <agent_id>. Check the skill name matches one from this step's available_skills list, and that agent_id is the step's bot_id.
Case B — agent_id omitted (back-compat), no row:
Skill "api-design-review" not found for the active agent. Make sure you've called set_agent_identity first and the skill name matches one from available_skills. (today's message, unchanged)

Both keep the existing .maybeSingle() → throw new Error(...) shape; only the branch on whether agent_id was supplied changes the wording. Information scent: the message tells the caller exactly which knob (skill name vs. identity vs. agent_id) is wrong.

RLS rulings (per-mode authorization)

  • Remote (per-user RLS). agent_skills stays scoped to the caller's owned bots. A passed agent_id the caller does not own resolves to no row → returns the Case-A not-found error — never another tenant's skill. No leak.
  • Stdio (service-role). The service-role client bypasses RLS, so it can read any agent's skills by agent_id. This is acceptable: stdio mode is the trusted local/server context and already operates with full service-role reach.

3claim_next_step — response addition

When step.bot_id owns skills, add available_skills: [{name, description}] (names + one-liners only, no bodies) AC#5 and append a subagent-addressed SKILLS directive to the instruction string AC#6. When the bot has no skills, omit the available_skills key entirely and add no directive AC#8.

Skilled — response shape
{
  done: false,
  step: {...},
  claim_token: "otk_...",
  available_agents: [...],
  context: [...],
  expected_deliverables: [...],
  agent_id: "a1b2-...-bot",   // the step's bot_id
  available_skills: [
    { name: "api-design-review",
      description: "Checklist for REST/RPC contracts" },
    { name: "a11y-audit",
      description: "WCAG 2.2 AA pass for new UI" }
  ],
  instruction: "...persona/subagent rules... \n\n SKILLS: ..."
}
Skill-less — response shape
{
  done: false,
  step: {...},
  claim_token: "otk_...",
  available_agents: [...],
  context: [...],
  expected_deliverables: [...],
  // available_skills key OMITTED entirely
  instruction: "...persona/subagent rules..."
  // ← NO SKILLS block appended
}

Exact NEW instruction text to append (only when skills exist)

Appended to instruction as one more contextParts block (joined with \n\n, same as the existing CONTEXT / DELIVERABLES blocks). It is addressed to the subagent — the orchestrator forwards it verbatim into the spawned subagent's prompt and does nothing else with it. <persona> and <bot_id> are interpolated from the matched agent; <skill list> is "name — description" per line. The trailing EXCEPTION mirrors the existing "no subagent capability" exception in the claim instruction.

SKILLS: You have these skills available:
- api-design-review — Checklist for REST/RPC contracts
- a11y-audit — WCAG 2.2 AA pass for new UI

As you work, if one of these is relevant to your task, call
get_agent_skill_content with agent_id "<bot_id>" and the skill name,
then follow its instructions. Load ONLY the skills that are relevant
to this step — do not load all of them. You do NOT need
set_agent_identity; pass agent_id directly.

EXCEPTION — if your spawned subagent does NOT have the
get_agent_skill_content tool in its own context: you (the orchestrator)
call get_agent_skill_content yourself for each relevant skill and fold
the returned instructions into the subagent's system prompt before
spawning it.
Information scent (why this wording): the directive is written in the second person to the subagent ("You have these skills… call… follow its instructions") so the actor that holds the MCP tool is the one told to act. It (a) lists name + one-liner so relevance can be judged without a round-trip, (b) gives the exact next call verbatim with the agent_id already filled in, (c) states the load-only-relevant constraint and the no set_agent_identity shortcut up front, and (d) carries the EXCEPTION for tool-less subagents so the orchestrator knows precisely when — and only when — to do the loading itself. Concrete, not abstract.

4All states — concrete behaviour for each

Stateclaim_next_step responseSkill load?Concrete outcome
Skilled + skill relevant available_skills populated + agent_id + SKILLS directive SUBAGENT LOADS Primary: the subagent itself calls get_agent_skill_content({agent_id, skill_name}) while working, follows the returned instructions, finishes the step. Fallback (tool-less subagent): orchestrator pre-loads + folds in. AC#5
Skilled + NOT relevant available_skills populated + agent_id + SKILLS directive ADVERTISED, ignored Skill is advertised but the subagent's relevance gate ignores it → no content fetch. Lean prompt. Advertise ≠ load. AC#6
Skill-less agent available_skills key omitted · no directive NOTHING Byte-identical to today. Zero new calls, zero regression. AC#8
step.bot_id = null available_skills key omitted · no directive NOTHING No assigned agent ⇒ nothing to advertise. Orchestrator still picks an agent from available_agents as today. (Skills resolve later only if it chooses to load them by that agent's bot_id.)
Not-found skill (skill advertised but lookup misses — typo / wrong agent_id / unowned agent_id under RLS) ERROR Clear thrown error naming the resolved target (§2). Recovery: caller re-reads available_skills, fixes the name/agent_id, or proceeds without the skill — the step is NOT blocked.

Error recovery, end to end

  • Wrong skill_name / agent_id → not-found error → the subagent (or orchestrator, in the fallback) re-reads available_skills, retries with the corrected value; a missing skill never fails the step (the persona prompt alone is still a valid baseline).
  • Lost claim_token after compaction → re-call claim_next_step for a fresh token + a fresh available_skills list + agent_id (unchanged existing recovery).
  • Subagent drops mid-execution → orchestrator retries/resumes (it has the agent_id and the directive); skills reload identically since loading is idempotent and driven by the same advertised list.

5Why subagent-self-load (instruction-only), not server-inject or orchestrator-fold

Keeps prompts lean and avoids paying for skill bodies that the step doesn't need — the subagent loads full content on demand, by relevance, in the same context that does the work, instead of the server force-feeding every skill body into every claim, and instead of the orchestrator round-tripping bodies it never reads. The subagent is the actor with the persona context and the MCP tool, so it is best placed to judge relevance and apply the skill. Orchestrator-folding remains available as an explicit fallback purely for clients whose subagents cannot reach the MCP tool. AC (instruction-only loading)

+Contract notes confirmed with Requirements

  • Empty vs. absent available_skills. Resolved: omit the key entirely when the bot has no skills (consistent with the SKILLS directive simply not being appended) so clients branch on a single signal.
  • Relevance ownership. Resolved per rework: the subagent decides + self-loads (primary), since it owns the persona context and the MCP tool. The orchestrator only advertises. Orchestrator-load is the explicit fallback for tool-less subagents.
  • Authorization of cross-agent agent_id. Resolved (§2 RLS rulings): remote/RLS scopes agent_skills to owned bots → unowned agent_id returns not-found (no leak); stdio/service-role can read any, which is acceptable.
