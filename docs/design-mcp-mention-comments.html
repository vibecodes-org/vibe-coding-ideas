MCP @mention in Task + Step Comments

Step 2 — UX Design (API surface). Backend feature: both comment tools gain human @mention notifications. No new web UI. The "user" here is the calling agent (reads schemas + response) and the notified human (reads the bell). Designs to fixed FR-1..15 / AC-1..12.

§0 TL;DR — Decisions

  1. Both add_task_comment and add_step_comment gain one optional param: mentioned_user_ids: uuid[]. Plus automatic @Full Name parsing of content. The two sources are unioned.
  2. Name matching is exact, normalised, greedy longest-match against the idea's human roster (author + collaborators). No fuzzy / partial / prefix matches.
  3. Mentions are never fatal. The comment always posts. Unresolved / ambiguous / non-member / opted-out / self tokens are reported in mentions.unresolved[] with machine-stable reason enums.
  4. Notification row mirrors web exactly: {type:"task_mention", idea_id, task_id, actor_id, user_id}. Fire-and-forget; a failed insert never fails the tool call.
  5. Step comments resolve their parent task via one lookup (task_workflow_steps.task_id) so the notification points at a real task. If that lookup fails, the comment still posts and mentions are skipped with a warning.
  6. Self-skip excludes both ctx.userId (active identity) and ctx.ownerUserId (human owner).
  7. Both MCP modes insert notifications with the same request-scoped ctx.supabase — valid today under the permissive with check (true) policy.

§1 Current State — what the tools return today

add_task_comment (mcp-server/src/tools/comments.ts)

Params: task_id, idea_id, content. Inserts into board_task_comments, logs comment_added activity, returns:

{ "success": true, "comment": { "id", "content", "created_at" } }

add_step_comment (mcp-server/src/tools/workflows.ts)

Params: step_id, idea_id, content, type (comment|output|failure|approval|changes_requested). Inserts into workflow_step_comments and returns the raw row (no success wrapper):

{ "id", "step_id", "idea_id", "author_id", "type", "content", "created_at" }

A workflow step belongs to exactly one board task via task_workflow_steps.task_id. The step comment carries no task_id of its own — this is the lookup §5 adds. The type field is unrelated to mentions and is preserved untouched; a mention fires regardless of comment type.

Web precedent to mirror The board web client (task-comments-section.tsx ~L247) inserts notifications rows { user_id, actor_id, type:"task_mention", idea_id, task_id }, skipping self, skipping non-team members, skipping notification_preferences.task_mentions === false, fire-and-forget with an error log. The MCP path reproduces this shape and skip logic verbatim. The web name matcher is autocomplete-only (regex /(?:^|[\s])@(\S*)$/ against full_name); MCP has no autocomplete, so it needs the server-side parser specified in §4.

§2 Roster — who is mentionable

The roster is the idea's human team: author + collaborators, from getIdeaTeam().teamMembers (src/lib/idea-team.ts). This matches the web task-comment mention source (which uses teamMembers, not allMentionable).

  • Agents / bots are NOT in this roster. Agent-directed pings use the separate discussion @agent path (agent-mentions.ts). A bot full-name in @Name resolves to not_team_member, not a notification.
  • Each roster entry contributes its full_name (nullable). Entries with full_name = null are not matchable by name (they can still be targeted by explicit mentioned_user_ids).
  • Explicit mentioned_user_ids that are not in the roster resolve to not_team_member.

§4 Parser spec — @Full Name detection

4.1 Normalisation (applied to both roster names and candidate text)

  1. Unicode NFKC normalise.
  2. Casefold (lowercase).
  3. Collapse any internal whitespace run to a single space; trim ends.

Diacritics are preserved (é ≠ e) — full names are identity data; we normalise form + case + spacing only. A pre-built map normalised(full_name) → user_id is the match index; a name that normalises to a value already in the map marks that value ambiguous (§4.4).

4.2 @ boundary rules

A mention token starts at an @ that is at start-of-string or immediately preceded by a non-alphanumeric character (whitespace or punctuation: space, newline, (, [, ", ', >, etc.). An @ preceded by a letter or digit is ignored — this is the rule that makes foo@bar.com, a@b, and email addresses never trigger.

4.3 Greedy longest-match after @

From the character after a valid @, read the following run of name-characters (letters, digits, spaces, and intra-name punctuation like - . '), then find the longest roster name (by word count, then char length) whose normalised form equals a normalised prefix of that run ending on a word boundary. Matching is whole-name-only: a token equals a roster name in full or it does not match at all — there is no partial credit.

  • Trailing punctuation adjacent to the name (, . ! ? : ; ) ] ") is not part of the name — @Nick Ball, matches Nick Ball.
  • Prefix / longest wins: if the roster has both Nick and Nick Ball, @Nick Ball ⇒ Nick Ball; @Nick, ⇒ Nick. If the roster has only Nick Ball, a bare @Nick matches nothing (no partial) ⇒ unknown_name.
  • After a match consumes N words, scanning resumes after the matched span, so one comment can carry several distinct mentions.

4.4 Decisions on tricky surface

Markdown / code spans — parse everywhere (do not strip) We deliberately do not exclude fenced blocks or inline code from parsing. Rationale: matching is exact-full-name against a small human roster, so a false positive requires an @ plus a real teammate's exact full name literally inside code — negligible, and arguably still intent. Stripping markdown would add a parser dependency and a fresh class of boundary bugs (nested fences, unterminated backticks) for near-zero benefit. Keep it simple; revisit only if reviewers object (see open questions).

  • Email-like strings (ballathesenior@gmail.com): the @ is preceded by r (alnum) ⇒ ignored by 4.2. Never triggers.
  • Ambiguous roster: two members whose names normalise identically ⇒ that name resolves to ambiguous; neither is notified. (Explicit mentioned_user_ids is the disambiguation escape hatch.)
  • Bare @ or @ followed by no roster match ⇒ token recorded as unknown_name only if it consumed a plausible name run; a lone @ with no following word is silently ignored (no noise).

§5 Response schema — appended to existing payload

A mentions object is added to each tool's existing return value. For add_task_comment it sits beside success/comment; for add_step_comment it is added to the returned row object. Absence of any mention → mentions is still present with empty arrays (predictable shape, no branching for the agent).

{
  /* ...existing fields (success/comment  |  raw step row)... */
  "mentions": {
    "notified": [
      { "user_id": "uuid", "full_name": "Nick Ball" }
    ],
    "unresolved": [
      { "token": "Nicky", "reason": "unknown_name" },
      { "user_id": "uuid", "reason": "not_team_member" }
    ],
    "warning": "1 name could not be matched to a team member: \"Nicky\"."
  }
}

5.1 Field semantics (information scent)

  • notified[] — exactly the members a notification row was written for. { user_id, full_name } so the agent can echo who it pinged.
  • unresolved[] — every token/id that did not produce a notification, each with a machine-stable reason. Name-derived entries carry token (the raw text after @); id-derived entries carry user_id.
  • warning? — present only when unresolved is non-empty; one human-readable sentence summarising the misses. Gives the agent a ready-to-surface line without parsing the array.

5.2 reason enum — stable, exhaustive

reasonMeaningCarries
unknown_name@token matched no roster name.token
ambiguous@token matched ≥2 members with identical normalised names.token
not_team_memberExplicit user_id not in the idea roster.user_id
opted_outResolved member has task_mentions === false.user_id
selfResolved to caller (ctx.userId or ctx.ownerUserId).user_id
step_task_unresolvedStep comments only — parent task lookup failed, so no mention could be routed (§5.3, §7).token/user_id
Why surface self / opted_out at all? They aren't errors, but hiding them makes an agent think its ping landed when it didn't. Reporting them (non-fatally, distinct enums) preserves information scent: the agent can see "you can't @-mention yourself" or "that member has mentions off" and stop retrying. warning only counts genuine misses (unknown/ambiguous/not_team_member/step_task_unresolved), so a self-only mention produces empty-ish noise-free output.

5.3 Resolution pipeline order

  1. Parse @names from content → set of (token → user_id | unknown | ambiguous).
  2. Union with explicit mentioned_user_ids.
  3. Dedupe by user_id.
  4. Drop self (ctx.userId, ctx.ownerUserId) → self.
  5. Drop non-roster ids → not_team_member.
  6. Drop task_mentions === false → opted_out.
  7. Survivors → insert notification rows → notified[].

§6 Step → parent-task resolution

notifications.task_id is required for the bell to route to a task view, but a step comment only has step_id. Add one lookup:

ctx.supabase
  .from("task_workflow_steps")
  .select("task_id")
  .eq("id", params.step_id)
  .maybeSingle()
  • Resolved → use task_id for the notification row; proceed as task comments do.
  • Null / error (step deleted, RLS miss, race) → degrade, don't fail: the step comment still posts and returns normally; the mention pipeline is skipped; every would-be mention is reported as step_task_unresolved and warning explains it. This is a fire-and-forget concern, not a data-integrity one.

idea_id is already a param on both tools, so no extra lookup is needed for it.

§7 Per-mode client decision (notification insert)

Both modes write the notification with the same request-scoped ctx.supabase already in hand — no second client is constructed.

Modectx.supabaseInsert into notifications
Local (stdio)service-role, bypasses RLSAlways permitted.
Remote (HTTP)per-user, RLS enforcedPermitted today: the notifications INSERT policy is with check (true), so a user may insert a row targeting another user_id.
Fallback if the RLS policy tightens If notifications INSERT is ever restricted (e.g. with check (auth.uid() = actor_id) would still pass, but a stricter user_id = auth.uid() check would block cross-user inserts), the remote path must escalate to a service-role client for the notification insert only — the comment insert stays per-user. Flagged so the reviewer confirms the current policy is intended to stay permissive. Until then: one client, both modes.

§8 Notified human's experience

  • Bell: a task_mention notification appears in the human's bell, identical to a web @mention — existing rendering and routing are reused, nothing new to build.
  • Click target: existing task_mention routing lands the human on the task view (via idea_id + task_id). Task-comment mentions land precisely on that task's comments. Step-comment mentions also land on the parent task — the notification cannot deep-link to the specific workflow step. This slight imprecision is accepted: the human arrives at the right task and can find the step from there. (No new deep-link route in scope.)
  • Actor display: actor_id = ctx.userId — the posting identity. When an agent posts under a bot identity, the bell shows the bot's name (bots have a users row with full_name), which correctly attributes the comment to its author. We deliberately do not substitute ownerUserId here, so "who pinged me" matches "who wrote the comment."

§11 Open questions for the review gate

  1. Code-span parsing (§4.4): confirm parse-everywhere is acceptable, vs. stripping fenced/inline code. Recommendation: keep simple.
  2. Surface self / opted_out in unresolved (§5.2)? Design says yes for information scent; alternative is to hide them for a quieter payload.
  3. Notifications RLS (§7): confirm with check (true) is intended to stay, so the remote path can insert cross-user notifications with the per-user client.
  4. Step deep-link (§8): accept landing on the parent task (no step anchor) for step-comment mentions, or is a step-anchored route wanted later?
