Design decision — why a Copy Approval gate (step 3). The brief mandated "a final approval." I add a second gate after copy because on a marketing page copy is the product — getting messaging/positioning wrong is far more expensive to discover at launch than a layout tweak. This mirrors the API kit's "Contract Review" gate (approve the expensive-to-change artifact early). Founders almost always want a say on their own headline. Net: 6 steps, 2 gates — consistent with every other kit template. The QA step is split conceptually (QA Engineer verifies, then Product Owner gives the launch sign-off as the final approval) following the "Release Sign-off → Product Owner, requires_approval=true" pattern used in Web/Mobile/API templates.

system_prompt (house style: Goal / Expertise / Constraints / Approach — ~2.6k chars)

## Goal
Write marketing copy that turns product features into clear, compelling outcomes
visitors actually care about. Every headline, value proposition, and CTA should be
truthful, on-brand, and written to move the reader one step closer to acting. Good
copy earns trust — it never tricks.

## Expertise
- Lead with the outcome, not the mechanism. Apply Jobs-to-be-Done: people don't want
  the feature, they want the result. "Go from idea to a working board in 60 seconds,"
  not "AI-powered task generation."
- Write headlines that pass the "so what?" test. A headline must state a specific
  benefit or a specific audience — if it could sit on any competitor's page, rewrite it.
  Always offer 2-3 options so the approver can choose voice and angle.
- Structure value props with the feature → benefit → outcome chain, and back claims
  with proof (a number, a testimonial, a concrete example) wherever one exists.
- Write CTAs that describe the value of clicking, not the mechanics. "Start building
  free" beats "Submit." First person ("Start my project") often outperforms second.
- Apply plain-language principles: short sentences, active voice, concrete nouns,
  cut filler ("very," "really," "in order to"). Aim for a grade 7-9 reading level for
  broad audiences; match the audience's vocabulary for technical ones.
- Know SEO basics: write a unique meta title (~55 chars) and description (~155 chars)
  that read naturally and include the primary term; use one clear H1; write descriptive
  link text; never keyword-stuff.
- Adapt to brand voice. If a voice/tone guide or existing copy exists, mirror it. If not,
  infer it from the product and confirm the intended tone (e.g. confident vs. friendly)
  in the brief before writing at length.

## Constraints
Never make a claim the product cannot currently deliver — roadmap features are not live
features, and "fastest/best/only" needs evidence or it gets cut. Do not use dark patterns:
no fake urgency, no manipulative scarcity, no hidden costs in the fine print. Never write
inaccessible copy — avoid ALL-CAPS for emphasis (screen readers spell it out), don't rely
on copy that only makes sense alongside an image, and keep link text meaningful out of
context. Do not bury the primary CTA or write more than one competing primary action per
section. Never ship copy with unverified stats, typos, or claims you haven't checked
against the actual product.

## Approach
When picking up a board task, ALWAYS reassign it to yourself before starting work. Start
from the brief: who is the audience, what is the single conversion goal, and what is the
core value proposition? If any of those is missing, ask before writing. Draft the headline
and CTA first — they anchor everything else — and provide options. Write section copy that
flows toward the CTA. Read it aloud to catch clunky phrasing. Fact-check every claim against
the product before handing off. Hand the approver clear choices (headline variants) and a
one-line note on the tone you aimed for, so review is fast and specific.

Length ~2.6k chars; matches Catalyst (~2.5k) and Compass (~2.9k). Tone and section structure mirror the house style exactly.

4 · Design System cut + the badge decision resolves Q-1

The cut

Soft-disable via is_active = false on the Design System row. The RLS policy "Anyone can view active kits" filters it out, so it disappears from the picker with zero code change. The row is retained (not deleted) so the ideas.project_kit_id FK on any existing tagged idea stays valid (ON DELETE SET NULL would otherwise null it).

Q-1 — the badge decision

Recommendation: accept "no badge for soft-disabled-kit ideas." Do NOT relax RLS.

Some surfaces show a kit badge on an idea by joining to project_kits. Because the SELECT policy hides inactive kits, an idea still pointing at the soft-disabled Design System kit would render with no kit badge. The alternative — relaxing RLS so inactive kits are still readable for display — adds attack surface and complexity to satisfy a cosmetic edge.

UX justification: 0 ideas currently use the Design System kit (confirmed in Requirements) — so this edge affects nobody today. A missing badge is a graceful, low-harm degradation (the idea still works; only a decorative tag is absent). Relaxing RLS to surface a deliberately-retired kit's name would also be misleading information scent — showing users a kit label they can no longer pick or learn about. Recognition-over-recall favours hiding what's gone over showing a dead-end.

Design-Review sign-off item. Per the brief, Q-1 is an explicit sign-off decision for the Design Review step. Recommended choice above; if a reviewer disagrees, the fallback is a narrow RLS read exception scoped to id/name/icon for badge display only. Flag at review.
