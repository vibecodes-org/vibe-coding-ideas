Role-Agent Mapping — UX Options

3 options for showing which agent matches each workflow role in the Role Coverage panel.

Option A: Tooltip on Hover

Chips stay compact. Hover to see the matched agent name and match tier.

Role Coverage — tooltip on hover
Role Coverage
Matched: Atlas (substring)
✓ developer
✓ product owner ✓ qa engineer ✓ ux designer
⚠ devops engineer

Pros

  • No layout change — chips stay compact
  • Info available on demand
  • Shows match tier for debugging
  • Smallest implementation effort

Cons

  • Not discoverable — user must hover each chip
  • Can't see all mappings at a glance
  • Doesn't work well on mobile (no hover)

Option B: Inline Agent Name

Agent name shown directly in each chip: "developer → Atlas". Visible at a glance.

Role Coverage — inline names
Role Coverage
✓ developer → Atlas ✓ product owner → Nick's PO ✓ qa → Sentinel ✓ ux → Compass ✓ frontend engineer → Atlas ✓ devops engineer → Atlas ✓ security analyst → Shield ⚠ data engineer → no agent

Pros

  • All mappings visible at a glance
  • Immediately shows incorrect matches
  • Works on mobile (no hover needed)
  • Most informative

Cons

  • Chips are wider — may wrap more on small screens
  • Doesn't show match tier (could add tooltip for that)
  • Slightly more visual noise

Option C: Expandable Table

Replace chip row with a collapsible table showing role, matched agent, and match tier.

Pros

  • Most detailed — shows all info (role, agent, tier)
  • Easy to scan for incorrect matches
  • Match tier visible without hover
  • Familiar table pattern

Cons

  • Takes more vertical space
  • Heavier than chips — could feel admin-like
  • Needs collapse/expand to avoid clutter
Recommendation: Option B (inline names) gives the best balance — all mappings visible at a glance without hover, immediately shows incorrect matches, works on mobile. Could combine with Option A's tooltip to also show the match tier on hover for power users who want to debug the fuzzy matching.
