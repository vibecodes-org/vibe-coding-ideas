Apply a Kit to an Existing Idea

UX design for surfacing kit application on boards that have tasks but no agents / workflows / kit yet. Author: Compass (UX). Builds directly on the Requirements step (ProdOwner). Kit application already works via ApplyKitDialog; this step is about discoverability, dead-link fixes, idempotency, and the post-kit Enhance chip.

AC-1 tileAC-2 direct dialog AC-4 no tasks noteAC-6 idempotency AC-8/9 dead-link fixAC-12 enhance chip
Ground truth carried from Requirements. Kit application is already wired (ApplyKitDialog seeds agents + labels + workflows + triggers, zero tasks; persists ideas.project_kit_id; no migration). The only confirmed defect is dead <Link href="?tab=workflows"> navigation in three places — Next <Link>/pushState does not fire popstate, which board-page-tabs.tsx listens for. The working pattern is pushState + dispatchEvent(new PopStateEvent("popstate")) (see workflows-tab.tsx:1231-1232). This design avoids that fragility entirely by opening the dialog directly.

1. User flows

Flow A — Apply a kit from the empty-state "Get started with AI" dialog

Open board (tasks=0, agents=0, workflows=0, kit=null) → "Get started with AI" dialog auto-shows → Click Apply a Kit tile (AC-1) → Empty-state dialog closes; ApplyKitDialog opens directly (AC-2) → Select project type → preview → Apply → Success toast + board refreshes; tile no longer eligible (AC-6)

Back-nav: ApplyKitDialog Cancel / Esc / overlay-click returns the user to the board (NOT back to the empty-state dialog — re-opening it would be a loop trap). The empty-state dialog is already dismissed at that point; if the board is still empty of agents/workflows, the setup banner (Flow B) becomes the persistent re-entry point. This honours recognition-over-recall: the kit affordance is never lost.

Flow B — The fixed banner "Apply a Kit" button

Board has tasks but ≥2 gaps and no kit (showKitShortcut true) → Blue setup banner: "Apply a project kit…" → Click Apply a Kit button → ApplyKitDialog opens directly (no tab switch) (AC-8/9)

Today this button is a dead <Link href="?tab=workflows"> (setup-completeness-banner.tsx:208) — it changes the URL but the board tab does not switch, leaving the user stranded. Same dead pattern in the empty-state tiles (board-empty-state.tsx:73,84) and workflows-tab.tsx:980. See §6 for the recommendation.

Flow C — Enhance-with-AI chip appears after a kit is applied (AC-12)

Idea has project_kit_id set → Open Enhance with AI on the idea → Configure phase shows violet "Tailoring for X" chip

The shared EnhanceDialogShell already renders the chip when kitContextLabel is passed (enhance-dialog-shell.tsx:454-458). The create-flow passes it; the existing-idea wrapper enhance-idea-dialog.tsx does not. The fix is purely passing the resolved kit name down — no new component. The chip is informational only (no extra clicks, no new state machine).

Error recovery (all flows)

  • Kits fail to load → existing behaviour: toast.error("Failed to load project kits") and the body shows the empty selector. We add a visible inline retry (see §3 error state) so the user is not stuck on a silent empty list.
  • Apply fails → existing toast.error; dialog stays open, selection preserved, Apply button re-enabled so the user can immediately retry. No navigation occurs, so no lost context.
  • Enhance fails → unchanged shell behaviour: toast + revert to configure phase. The kit chip stays visible (it is config metadata, not result state).

6. Dead-link fix — UX recommendation (AC-8 / AC-9)

Recommendation: open ApplyKitDialog directly via an onClick handler. Do NOT navigate to ?tab=workflows.
CriterionDirect dialog (recommended)Switch to Workflows tab
Information scentStrong — the label "Apply a Kit" maps 1:1 to the kit dialog. The user gets exactly what the label promised.Weak — user lands on a tab and must hunt for the kit affordance among templates/agents. Scent is lost at the tab boundary.
Fitts's Law1 click → goal. Target is large and immediate.1 click → tab → locate "+" / kit button → another click. More distance, more targets.
ReliabilityNo URL state involved. Cannot regress.Relies on pushState + dispatchEvent(PopStateEvent); the current bug exists precisely because a plain <Link> omits the event. Fragile.
ContextUser stays on the board they were reading; success refreshes in place.Yanks the user away from their tasks; back-nav ambiguity.

Concrete changes (engineering will implement; listed for design completeness):

  • setup-completeness-banner.tsx — replace the dead <Link href="?tab=workflows"> wrapping "Apply a Kit" (line 208) with a <Button onClick={onApplyKit}>; the banner accepts a new onApplyKit callback that the board page wires to setKitDialogOpen(true) (the dialog already lives in the board tree).
  • board-empty-state.tsx — the new kit tile (and ideally the agents/workflows tiles) become <button onClick>. The kit tile calls onApplyKit; agents/workflows tiles, if kept as tab links, must use the working pushState + dispatchEvent(new PopStateEvent("popstate")) pattern instead of <Link>.
  • workflows-tab.tsx:980 — same dead-link pattern; convert to the working pushState+PopStateEvent pattern already used at lines 1231-1232.

For tab-destination tiles that we keep as navigation (agents / workflows), the nudge link in the expanded banner checklist (CHECKLIST_ACTION_TEXT, "Workflows tab →") must also use the working pattern, not <Link href={gap.action.href}> — that is the same dead pattern and is part of AC-8/9.

7. Idempotency & visibility rules (AC-6)

The kit affordance must vanish once setup exists. Signals, in priority order:

SignalSourceEffect on kit tile / banner
ideas.project_kit_id != nullidea row → hasKit in idea-health.tsHide. A kit was already applied. (showKitShortcut = !hasKit && …, line 169.)
≥1 idea agentideaAgents.length > 0 (kanban) / allocatedAgentCount (health)Hide kit tile.
≥1 workflow templatehasWorkflowTemplates (kanban) / workflowTemplateCountHide kit tile.
Tile visibility (AC-1, AC-6): show the "Apply a Kit" tile only when !hasAgents && !hasWorkflows && !hasKit. The empty-state dialog already receives hasAgents / hasWorkflows (kanban-board.tsx:1128-1129); it needs one new prop hasKit (derived from ideas.project_kit_id). When any becomes true the tile is removed and the grid collapses cleanly (2 tiles → row of 2; or the dialog may not auto-open at all if the board already has setup).
Edge case — agents/workflows exist but no kit. Per AC-6 the kit tile hides if agents OR workflows exist. The banner's showKitShortcut uses missing.length >= 2, so a board with agents-but-no-workflows (1 gap) won't show the kit banner anyway — consistent. No conflict, but worth noting the empty-state tile rule (any one signal hides) is intentionally stricter than the banner's gap-count rule. Both correctly suppress once a kit is applied.

9. Accessibility (WCAG 2.1 AA)

  • Tile as control. Convert the kit tile from a coloured <div>/link to a real <button> with an accessible name "Apply a Kit — set up agents, workflows, labels and triggers". Keyboard-focusable, Enter/Space activate.
  • No colour-only meaning. Every tile and pill pairs colour with an icon + text label. The AC-4 note and the load-error both use an icon (ℹ️ / ⚠️) plus text, never colour alone.
  • Contrast. Blue tile title #60a5fa on rgba(96,165,250,.05) over #09090b ≈ 6.3:1 (passes AA normal text). The 11px muted description uses var(--muted-fg) #a1a1aa ≈ 6.9:1. Violet chip text #a78bfa on its tinted bg ≈ 5.8:1. All meet 4.5:1; UI borders meet 3:1.
  • Focus management. On "Apply a Kit" click: close empty-state dialog, then move focus into ApplyKitDialog (Radix Dialog auto-focuses the first focusable — the project-type selector). On ApplyKitDialog close, return focus to the board / the control that opened it (Radix restores trigger focus when a trigger is used; for the programmatic open from the empty-state tile, set focus back to the board region or the setup banner button).
  • Live regions. Loading body gets aria-busy="true" + visually-hidden "Loading project kits". Sonner toasts already announce via role="status"/alert.
  • Visible labels. Project-type selector and the prompt use real labels (not placeholder-only). The kit chip is descriptive text, not a tooltip.
  • No hover-only actions. Tile arrow/hover lift is decorative; activation works on tap/keyboard. Mobile (no hover) reaches every action.
  • Dialog semantics. Reuse Radix Dialog/Drawer (focus trap, Esc to close, labelled title/description) already in place.
