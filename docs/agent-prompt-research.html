RQ3: Optimal Prompt Length & Structure

  • Length: 300-600 words is the sweet spot. Under 200 words lacks depth; over 800 words sees diminishing returns as later sections get less attention. Current prompts average ~150 words — too short for expertise.
  • Structure: Consistent sections work best. Proposed new structure:
    • ## Goal — 1-2 sentences on the role's purpose (keep this)
    • ## Expertise — NEW section: 5-8 domain-specific principles, frameworks, or heuristics the agent knows
    • ## Constraints — Specific, named anti-patterns (upgrade from generic)
    • ## Approach — How the agent works on tasks (keep, enhance with domain-specific workflow)
  • Style: Dense, principle-per-line format works better than prose paragraphs. Each line should teach something.

RQ4: Hub Templates vs Default Templates

Decision: Hub templates (the 15 VibeCodes-owned agents in the database) and default templates (BOT_ROLE_TEMPLATES in constants.ts) should use identical prompts for matching roles. Both are tech-agnostic starting points. The hub templates are what users clone; the code templates are what the "create agent" UI offers. They should say the same thing.
  • Hub templates (15 agents): These are the VibeCodes-owned agents in bot_profiles that appear in the community hub. Users clone them. They should be the gold standard.
  • Default templates (8 roles in constants.ts): These appear in the "create agent" dialog. They should match the hub templates for overlapping roles.
  • Gap: The code has 8 templates but the hub has 15. The 7 extra hub roles (Backend Engineer, Data Engineer, Code Reviewer, Product Manager, Technical Writer, CEO/Founder, Marketing Strategist, Sales Lead, Finance & Ops) should be added to BOT_ROLE_TEMPLATES too.
  • Project-specific context: NOT included in templates. Users' CLAUDE.md files and project context handle that. Templates should be universally applicable.

RQ5: Competitive Landscape

Analysis of how other platforms approach agent/role prompts:

  • Cursor Rules / .cursorrules: Project-level, not role-level. Focuses on conventions and tech stack, not personas. No domain expertise embedding.
  • Windsurf Rules: Similar to Cursor — project conventions, not role personas.
  • Devin: Single agent, no role system. Uses a monolithic system prompt with broad engineering knowledge.
  • GitHub Copilot Workspace: No configurable personas. Uses task context, not role expertise.
  • Claude Code CLAUDE.md: Project-level instructions. No role system.
Insight: No major competitor embeds domain expertise into role prompts — they all focus on project context. VibeCodes has a differentiation opportunity here: agents that actually know their craft, not just follow a process checklist. This is a genuine competitive advantage if done well.
