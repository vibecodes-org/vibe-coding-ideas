Workflow System

A full step-based pipeline replacing simple checklists — supporting multi-agent orchestration, approval gates, cascade rejection, fuzzy role matching, and automated template application.

System Overview

The workflow system is the backbone of task execution in VibeCodes. When a workflow template is applied to a board task, it creates a workflow run containing ordered steps that agents claim, execute, and complete — with optional human approval gates at any point in the pipeline.

Database Layer

Six tables form the workflow system:

TablePurpose
workflow_templates Idea-scoped reusable step sequences. Steps stored as JSONB array with title, role, description, requires_approval, deliverables.
workflow_auto_rules Maps a board label to a template. When a label is added to a task, the app auto-applies the matching template. UNIQUE(idea_id, label_id).
workflow_runs Tracks each application of a template to a specific task. One active run per task enforced by partial unique index. template_id nullable (ON DELETE SET NULL) so runs survive template deletion.
task_workflow_steps The actual executable steps. Trigger maintains denormalized workflow_step_total and workflow_step_completed on board_tasks. Realtime-enabled with REPLICA IDENTITY FULL.
workflow_step_comments Inter-agent communication. Five types: comment, output, failure, approval, changes_requested. Trigger maintains denormalized comment_count.
workflow_library_templates Admin-managed global template library. Seeded with 7 built-in templates. Case-insensitive unique name. RLS: all authenticated read, admin-only write.

Fuzzy Agent Matching

When a template is applied to a task, agents from the idea's pool are auto-matched to step roles using a three-tier system defined in src/lib/role-matching.ts:

1
Exact Match
Case-insensitive, trimmed string equality
"Developer" = "developer"
2
Substring Match
Bidirectional containment, minimum 2 characters
"Dev" ⊂ "Full Stack Developer"
3
Word Overlap
Prefix match on tokenized words, minimum 3 characters
"frontend" ≈ "Front End Engineer"
Why app code, not a trigger

Auto-rule application was originally a Postgres trigger (auto_apply_workflow_on_label), but was moved to application code in migration 00083 because the DB couldn't perform fuzzy agent matching. The auto_run toggle was subsequently removed in migration 00085 — rules always fire.

The buildRoleMatcher function is called once per template application and returns a closure, so agent candidate preprocessing happens only once.

Concurrency Guards

The workflow system is designed for concurrent multi-agent access. Every status mutation is guarded at two levels:

Application Level

Every step mutation uses the pattern: 

const { data } = await supabase
  .from("task_workflow_steps")
  .update({ status: "in_progress" })
  .eq("id", stepId)
  .eq("status", "pending")     // ← status guard
  .maybeSingle()              // ← returns null if guard fails

If another process already changed the status, the update returns null instead of corrupting state. This prevents concurrent claims, double-completions, and race conditions.

Database Level

A partial unique index on workflow_runs prevents duplicate active runs per task: 

CREATE UNIQUE INDEX idx_unique_active_workflow_run
  ON workflow_runs(task_id)
  WHERE status NOT IN ('completed', 'failed');

bot_id vs claimed_by

ColumnSet WhenPurpose
bot_id Template is applied to a task The agent pre-matched from the idea's pool based on fuzzy role matching. Preserved for reference — not overwritten at claim time.
claimed_by claim_next_step is called The agent (or user) who actually claimed and executed the step at runtime. May differ from bot_id.

The MCP claim_next_step tool sets claimed_by (not bot_id), and returns both bot_id and available_agents so the client can switch identity via set_agent_identity.

Cascade Rejection

When a reviewer rejects work at an approval gate, they can send work back to any earlier step in the pipeline — not just the immediately preceding one.

How it works

  1. Reviewer calls fail_step with optional reset_to_step_id
  2. System snapshots outputs of all affected steps as output-type comments (preserving work)
  3. All steps from the target onward are reset to pending — output, timestamps, and claimed_by are cleared
  4. A changes_requested comment is posted on the cascade target step with the reviewer's feedback
  5. When the target step is re-claimed, claim_next_step returns rework_instructions containing the prior failure output and changes_requested comments
Two tiers of rework context

claim_next_step first checks the step itself for failure/changes_requested comments. If none are found, it falls back to checking sibling steps in the same run — catching cascade-reset intermediate steps that weren't directly failed.

Auto-Rules

Auto-rules create a label → template mapping within an idea. When a matching label is added to a task, checkAndApplyAutoRules() in src/lib/workflow-helpers.ts fires and applies the template automatically.

  • Rules always fire — the auto_run toggle was removed in migration 00085
  • Guard against existing active runs via the partial unique index
  • Can be applied retroactively to all existing tasks with the matching label via applyAutoRuleRetroactively
  • UNIQUE(idea_id, label_id) constraint prevents duplicate rules
  • Deleting a rule optionally removes related workflows (removeRelatedWorkflows flag)

When a label is removed, checkAutoRuleWorkflow() / removeAutoRuleWorkflow() can detect and optionally clean up the orphaned workflow.

Template Propagation

When a template's steps are edited, propagateTemplateEdits() pushes changes to pending steps in active workflow runs:

  • Updates: title, description, role, approval gate, deliverables
  • Skips runs where step count doesn't match (structural changes)
  • Does NOT re-match bot_id — use rematchWorkflowAgents for that
  • Called from both the updateWorkflowTemplate server action and the MCP tool

MCP Orchestration

The claim_next_step Tool

This is the central orchestration interface. It does far more than claiming — it provides the agent with everything needed to execute:

  1. Finds the first pending step by step_order, then position
  2. Atomically claims it (status guard prevents concurrent claims)
  3. Updates the workflow run to running
  4. Fetches rework instructions — prior failure output + changes_requested comments
  5. Fetches available agents for identity switching
  6. Fetches context from prior completed steps (output column)
  7. Builds a detailed instruction string including:
    • Identity switch directive (which agent to assume)
    • Context chaining rules (how to reference prior step outputs)
    • Expected deliverables with format detection (HTML, JSON, etc.)
    • Human approval warnings
    • Cascade rejection guidance
    • File-based deliverable handling (write to docs/ directory)
Approval gate enforcement

The approve_step MCP tool blocks bot callers — only humans can approve human-gated steps. It checks users.is_bot and instructs the agent to switch to human identity first. The complete_step tool routes to awaiting_approval instead of completed when the gate is active, with a stop message preventing self-approval.

src/lib/workflow-helpers.ts

Four shared functions used by both server actions and MCP tools:

FunctionPurpose
checkAndApplyAutoRules Called when a label is added. Looks up matching auto-rules, checks for existing runs, applies template. Non-throwing by design.
checkAutoRuleWorkflow / removeAutoRuleWorkflow Called when a label is removed. Detects/removes orphaned workflows from auto-rules.
checkAndCompleteRun Checks if all steps are terminal (completed or skipped). If so, marks run as completed.
propagateTemplateEdits Pushes template step changes to pending steps in active runs. Skips mismatched step counts.
