Claim-Token Protocol — Design Document Rev 3

Compass (UX Designer) · Step 2 of 6 · Task d0824927 · Builds on the Requirements deliverable — references inline.
Rev 3 changelog (Design Review round 2, reviewer: Nick). The complete-time persona enforcement check is KEPT — Rev 2 deleted it as "theater", but the reviewer correctly identified it as a true-positive detector for work done by the wrong agent, now reliable since per-connection identity shipped (e8925b6). Enforcement becomes two-layer: claim token (capability) + persona consistency (quality), and the persona-mismatch rejection is non-destructive — the destructive reset-to-pending is the only thing removed. Amends the Rev 2 FR2 note: ambient identity IS consulted at complete, as a consistency gate beside the token.
Rev 2 changelog (Design Review feedback, reviewer: Nick). Rev 1 flipped the loop to set-identity-before-claim to enable persona validation at claim — a chicken-and-egg: the orchestrator only learns which step (and bot) is next from the claim response. Rev 2 keeps the canonical order and changes the claim semantics instead: the claim is made on behalf of the step's assigned bot, by construction. This also amends Requirements FR1/AC3: "validate persona at claim" is superseded — there is nothing to validate when the claim is definitionally for the assigned bot; AC3 now applies only to bot_id = null steps with an explicit agent_id override.

1 · The loop — order unchanged, one value added

Today (ambient identity)

agentclaim_next_step(task) → bot_id + "set identity to X"
agentset_agent_identity(X) ← state another session can overwrite
agent…work…
agentcomplete_step(step) ← re-reads ambient identity: race → reset

Rev 3 (claim token) — same order

agentclaim_next_step(task) → bot_id + "set identity to X" + claim_token ← claimed FOR X by construction
agentset_agent_identity(X) persona + attribution + passes the complete-time check
agent…work…
agentcomplete_step(step, claim_token) ← two-layer: token (capability) + persona check (kept, non-destructive)
Jakob's Law, applied to an API: orchestrators keep the loop they already know. The deltas an existing loop sees are exactly two: one extra response field, one extra request param.

2 · API design

claim_next_step (Rev 2 semantics)

// request — unchanged; agent_id optional, used ONLY when step.bot_id is null
{ "task_id": "…", "agent_id": "…(optional)" }

// behaviour
//  step.bot_id set  → claim is recorded FOR that bot: claimed_by = step.bot_id.
//                     Nothing to validate — the step belongs to that persona.
//  step.bot_id null → claimed_by = agent_id param ?? ambient identity ?? owner
//                     (today's behaviour, now also captured truthfully).
//  Either way: mint token, store sha256(token) on the step, return plaintext ONCE.

// response (additions marked ★)
{
  "step": { …, "claimed_by": "" },   ★ semantics fixed
  "claim_token": "ct_8f3a…",                                           ★
  "instruction": "set_agent_identity(bot_id) for persona/attribution,  ★ wording
                  do the work, then complete_step with step_id AND claim_token",
  "context": [ … ], "available_agents": [ … ]
}
Note on the persona check (revised in Rev 3): it catches "orchestrator forgot to call set_agent_identity (or set the wrong one)" — which matters: the persona's system prompt shapes the work and ambient identity drives mid-work attribution. Rev 2 dismissed this as theater; the review correctly overruled that — the check stays as a hard gate. What goes is only its destructive consequence: a mismatch no longer resets the step (see E3, §4). Bonus: claimed_by stops echoing whatever identity happened to be ambient at claim time (often the previous step's bot — a long-standing audit-trail oddity) and becomes the bot the step was claimed for.

complete_step / fail_step (FR2, two-layer per Rev 3)

// request (additions marked ★)
{ "step_id": "…", "claim_token": "ct_8f3a…", "output": "…" }     ★

// server checks, in order:
// 1. step exists & in_progress                       (unchanged)
// 2. claim_token_hash matches sha256(claim_token)    ← capability (hard)
// 3. step.bot_id set && caller ≠ step.bot_id → E3    ← persona check, KEPT (hard,
//    non-destructive: no DB write, claim stays valid)
// 4. on success: status→completed/awaiting_approval, claim_token_hash→NULL
Why both layers (Rev 3): the token proves the completer is the claimer; the persona check proves the right agent is at the wheel — it catches "claimed, never adopted the persona, worked persona-less," which no token can detect. With per-connection identity now shipped, the check's false positives are gone; only its destructive consequence (reset-to-pending) is removed.

3 · Flows & states (every state designed)

3a · Concurrent sessions — the June 2026 scenario, now safe (AC1)

session Aclaims step (for Atlas) → receives token TA → sets identity Atlas → works
session Bset_agent_identity(ProdOwner) on a different board ← attribution only; cannot touch A's claim
session Acomplete_step(step, TA) → succeeds. No cross-talk possible by construction.

3b · Lost token (context compaction / restart) — FR4 recovery

agentcomplete_step(step) — no token → E2 (see §4): names the exact next action
agentclaim_next_step(task) → re-claims (last-claim-wins; old hash overwritten = old token dead, FR3; displacement logged) → fresh token
agentcomplete_step(step, fresh token) → success. One-step recovery, no human needed.

3c · Cascade rejection / reset (FR3)

reviewerfail_step(step 3, reset_to_step_id: step 1) → steps 1–3 pending, claim_token_hash → NULL on each
any holdercomplete with a pre-reset token → E2 — stale tokens die with the reset

3d · Human paths unchanged (FR5)

UI server actions and approve_step authenticate the human session — no token field appears anywhere in the web UI. Boundary rule: tokens are for unattended agents; humans are authenticated people. The approval flow's screens are untouched.

Contrast with today: the current mismatch error resets the step (destructive, surprising — the agent loses its claim through no fault of its own). E2 and E3 deliberately leave the step in_progress: an error message should never be a state mutation. The reset never achieved a redo anyway — orchestrators re-claim and resubmit the same output — so nothing of value is lost by removing it.
