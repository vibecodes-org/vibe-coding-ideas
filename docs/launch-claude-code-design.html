Feature Development workflow · Step 2 of 6 (UX Design) · Target stack: Next.js 16, Tailwind v4, shadcn/ui "New York" / Zinc, dark theme default.

Design Review rework — what changed in this revision

This document is a revision of a previously rejected design. It re-submits the same approved structure (toolbar split-button, per-task surfaces, inline path dialog, fallback matrix, WCAG notes) and addresses two Design Review change requests:

  1. Add a “Browse…” button to the path field as a best-effort folder picker (File System Access API), designed around the hard browser limit that a web page cannot read a folder’s absolute path. See §2.3a and the updated dialog mockup in §2.3.
  2. Create a brand-new project (new directory) from the dialog — stakeholder request from Nick. Adds an “Existing folder / Create new project” mode toggle and delegates directory creation + clone to the launched local Claude Code. See §2.3b, the new flow in §4b, and the conditional bootstrap prompt in §3.

Approved decisions are preserved unchanged: path stored per-user-per-idea via localStorage (machine-specific; no DB column/table), inline path dialog (not idea settings), env-derived MCP URL, q ≤ 5000 guard, copy-command fallback, full fallback matrix, WCAG AA.

1. Intro & context

Citing the Requirements step (step 1)

The Requirements step defined a “Launch Claude Code” action on the board (toolbar-level and per-task) that opens the user’s local, subscription-authed Claude Code via a claude-cli://open?q=…&cwd=…&repo=… deep link. The link pre-fills a prompt that (a) auto-installs the vibecodes-remote MCP connector if it is missing and (b) tells the agent to pick up work via MCP. The deep link only pre-fills the prompt — the human presses Enter to start (human-in-the-loop). Runs on the user’s own subscription; no credentials in the link; desktop-only.

Key design flag — per-user-per-idea path (stored in localStorage)

The local project path (which becomes cwd) is a machine-specific absolute path and is stored per-user-per-idea in localStorage — not as a shared idea field and not in a DB table. Two collaborators on the same idea each have their own checkout in a different folder, often on different machines, so the value is inherently local-only. The UX treats the path as “your path on this machine,” private to you, and never displays one user’s path to another. This contrasts with the shared github_url field that lives on the idea itself (src/components/ideas/idea-edit-form.tsx).

Storage key: vibecodes:launch-path:<ideaId>. No migration, no RLS surface, no leakage between collaborators by construction.

Firm requirements honoured

  1. Per-idea launch path config (localStorage) → becomes the deep-link cwd (see §2.3).
  2. Launch from the board (toolbar + per-task) with auto-MCP-setup baked into the prompt (see §2.1, §2.2, §3).
  3. Best-effort Browse picker on the path field (see §2.3a). rework
  4. Create a brand-new project / directory from the dialog (see §2.3b, §4b). rework

2.3a — “Browse…” button (best-effort picker) rework

Hard browser constraint we design around

A web page cannot read a folder’s absolute path from any picker. Both window.showDirectoryPicker() (File System Access API) and <input type="file" webkitdirectory> expose only the folder name (plus an opaque handle / relative entries) — never /Users/you/projects/my-idea. So Browse cannot fill the field by itself. It is a convenience that pre-fills the folder name and positions the caret; the text field stays the editable source of truth.

Behaviour

  • Click Browse… → call window.showDirectoryPicker(). The user selects a folder in the native OS picker.
  • On pick → take handle.name (e.g. my-project) and pre-fill the field as /Users/you/projects/my-project using the user’s last-known parent prefix when available, otherwise just the bare name with the caret placed at the start so the user types the absolute prefix. The folder name is selected/emphasised so it’s obvious what was filled in.
  • The user completes the absolute path and the existing light-touch validation runs (warn if not absolute).
  • Unsupported browsers (Firefox, Safari, and any context without the File System Access API): the Browse button is hidden (preferred — avoids a dead control) with an adjacent one-liner; where layout needs it kept, it is rendered disabled with an always-visible explanation beside it (never a bare disabled button). Feature-detect via typeof window.showDirectoryPicker === "function".

2.3b — “Create new project” mode rework

Stakeholder request (Nick): start a fresh project — a new directory — straight from this UI, not only point at an existing folder.

Design decision & justification — who creates the directory

The same browser constraint applies: the page cannot create a directory at an arbitrary absolute path, and even the File System Access getDirectoryHandle(name, {create:true}) can only create inside a handle the user already granted, and still can’t yield an absolute path. So the browser is the wrong place to create the project.

Decision: delegate directory creation to the launched local Claude Code. It runs on the user’s machine with filesystem + git access, so the bootstrap prompt (§3) instructs it to mkdir -p <path> if missing, then — if the idea has a GitHub repo — git clone into that path, else git init, and only then proceed with MCP setup and work. This is more robust and portable than the File System Access route (which is also unsupported in Firefox/Safari). We may additionally use getDirectoryHandle({create:true}) as an optional nicety where supported, but it is not required and is not the source of truth.

In Create new project mode the single path field becomes two fields: the absolute parent path (where the new folder will live — Browse can pick this) and the new folder name (typed). The full intended path is previewed read-only beneath them so the user sees exactly what Claude Code will create.

2.4 — Fallback states

(a) No path set

Covered above: dropdown shows the hint state (§2.1) and per-task launch opens the path dialog (§2.3). The button is never disabled without explanation.

(b) Claude Code not installed (deep link doesn’t resolve)

The browser can’t confirm whether a custom scheme handler exists. We launch optimistically, then run a short timeout race: if the page never loses focus / visibility within ~1.2s, we assume nothing handled the link and surface a recovery toast.

Recovery toast — nothing opened

If Claude Code isn’t installed yet, install it and try again — or copy the command to run it manually.
⧉ Copy launch command   Install guide ↗

sonner toast (already in stack). Two recovery routes: copy-command fallback and a link to the install docs. Never a silent failure (per CLAUDE.md “all client catch blocks toast.error”).

(c) Browser blocks the claude-cli:// scheme

Some browsers/policies block custom-scheme navigation or throw on it. We catch the navigation error and immediately show the copy-command fallback inline rather than relying on the timeout.

Copy-command fallback popover
Your browser blocked the automatic launch. Paste this in your terminal to start Claude Code with this work loaded:
cd /Users/you/projects/my-idea && claude "<pre-filled prompt>"
⧉ Copy command

The copy-command fallback is the same content the deep link would have delivered, reframed as a shell command. For create-new mode the command is prefixed with mkdir -p <path> && cd <path> && (git clone <repo> . || git init) && claude "…" so the manual path matches the delegated bootstrap. Universal escape hatch reachable from every surface.

(d) Mobile viewport — desktop-only feature

Claude Code runs on a desktop/laptop; launching from a phone is meaningless. Below md we hide the launch button entirely from the toolbar action cluster (matching how the existing buttons collapse), and in the per-task surfaces we replace the action with a short explanatory state so a tapping user isn’t left confused.

3. The pre-filled prompt & deep link

The deep link is claude-cli://open?q=<prompt>&cwd=<path>&repo=<github_url> (all params URL-encoded). The MCP URL embedded in the prompt is env-derived from NEXT_PUBLIC_APP_URL (so staging vs production resolve correctly), not hard-coded. The q prompt is what the human reviews and submits, and is guarded to ≤ 5000 chars (truncate the work-context tail, never the MCP-setup head). It instructs the agent to self-install the MCP connector if missing, then pick up work.

(a) Existing-folder prompt

First, ensure the VibeCodes MCP connector is available. If "vibecodes-remote" is not already configured, run: claude mcp add -s user --transport http vibecodes-remote {NEXT_PUBLIC_APP_URL}/api/mcp Then connect (you'll be prompted to authorise via your browser the first time). Once connected, pick up my work on the VibeCodes board for this idea: • Idea: "{ideaTitle}" (idea_id: {ideaId}) • {taskScope} ← e.g. "Start on task: Add OAuth refresh-token rotation (task_id: …)" or, from the toolbar: "Call get_my_tasks and start the top item." Use the MCP tools (get_task / claim_next_step / set_agent_identity / move_task / …) to do the work. Move the task to In Progress and comment as you go.

(b) Create-new-project prompt rework

When the dialog is in Create new project mode, the prompt gains a bootstrap preamble (added by the prompt builder) before the MCP block. cwd is set to the parent so the relative mkdir/cd is unambiguous; the new absolute path is the composed parent/name.

Set up a new local project for this idea, then pick up work. 1. Create the project directory if it doesn't exist: mkdir -p {newProjectPath} && cd {newProjectPath} 2. {repoStep} • If the idea has a repo: git clone {github_url} . (only into an empty dir; else pull/skip) • If no repo: git init 3. Ensure the VibeCodes MCP connector is available. If "vibecodes-remote" is not configured, run: claude mcp add -s user --transport http vibecodes-remote {NEXT_PUBLIC_APP_URL}/api/mcp then connect (browser auth the first time). Now pick up my work on the VibeCodes board for this idea: • Idea: "{ideaTitle}" (idea_id: {ideaId}) • {taskScope} Use the MCP tools to do the work. Move the task to In Progress and comment as you go.

{repoStep} is resolved by the builder at compose time: if github_url is present it emits the git clone line, otherwise the git init line — so the prompt is unambiguous and the agent doesn’t guess. The mkdir -p is idempotent (handles the “already exists” case safely); cloning is gated to an empty directory to avoid clobbering existing work. Toolbar launch → {taskScope} = “run get_my_tasks and begin”. Per-task launch → names the specific task id/title.

5. Interaction & accessibility notes

WCAG 2.1 AA

  • Contrast. Emerald button text on its tint and all helper/muted text meet ≥4.5:1 for text and ≥3:1 for UI borders (Zinc dark tokens). The 🔒 “Private to you” and ⎇ “Repo detected” badges convey meaning by icon + text, not colour alone. The amber inline-warning rows pair the ⚠ icon and text with their tint.
  • Mode toggle (rework). Implemented as a Radix-backed role="radiogroup" (segmented control) with two role="radio" options carrying aria-checked; arrow keys move between modes, Space/Enter selects. The selected mode is shown by background + position, not colour alone. Switching mode re-labels fields and primary CTA and announces via the dialog’s live region.
  • Browse… (rework). A real <button type="button"> ≥36px tall adjacent to the field (Fitts). Has an accessible name “Browse for folder”. When unsupported it is removed from the tab order (hidden) or, if shown, aria-disabled with the explanation programmatically associated via aria-describedby — no disabled control without an adjacent reason. The post-pick name insertion moves focus back to the text input with the prefix region selected so a keyboard user lands ready to type.
  • Create-new fields (rework). Parent path and folder name each have a visible <label>; the “Will create” preview is exposed as aria-live="polite" so it’s announced as the user types. Validation messages use aria-describedby + aria-invalid, never colour-only.
  • Targets (Fitts). Toolbar button h-8 with full label; mobile menu rows py-2.5 giving ≥44px. The split-button caret keeps its own ≥24px hit area. Browse and the new-project fields are ≥36px tall.
  • Keyboard. Button and caret focusable; DropdownMenu/Popover are shadcn (Radix) primitives — arrow-key nav, Esc to close, focus trap + restore. Dialog traps focus, Esc cancels, the first field is auto-focused on open.
  • Labels. Icon-only surfaces carry aria-label + visible Tooltip (“Launch in Claude Code”). All path fields use real visible <label>s, not placeholders-as-labels.
  • No hover-only primary actions. The launch button is always visible. Menu items are keyboard- and tap-reachable like the existing share action.
  • No unexplained disabled states. “No path”, unsupported Browse, and empty/invalid create-new fields all show an inline explanation + fix, not a greyed dead control.
  • Feedback. Successful copy → success toast “Command copied”. Every failure path → a toast.error with a recovery action.

Acceptance criteria for QA (step 5)

  • Existing-folder launch: path set → one-click Open works; dropdown shows Open / Copy / Change / Start new.
  • No path: toolbar dropdown hint with both “Set folder” and “Start new project” links + working Copy; per-task opens path dialog.
  • Browse (supported): clicking Browse opens the native picker; on pick the folder name is inserted and the caret/selection lets the user type the absolute prefix; the helper line “…your browser can’t read the full path…” is present.
  • Browse (unsupported — Firefox/Safari): Browse is hidden, or disabled with an adjacent explanation; the text field still accepts a typed/pasted path; no dead control in the tab order.
  • Create-new happy path: choose Create new → enter parent + valid name → “Will create” preview matches → Create & launch composes the deep link with the create-new prompt (mkdir + clone/init); value persists to localStorage.
  • Create-new — dir exists: the “Use existing instead” switch flips to existing mode pre-filled with the composed path; relaunching is safe (idempotent mkdir -p; clone only into empty dir).
  • Create-new — no repo: repo row reads “will git init”; the generated prompt contains git init and no git clone.
  • Create-new — empty/invalid fields: inline ⚠ messages appear and focus the field on submit; CTA is never a bare disabled button.
  • Mode toggle a11y: radiogroup is keyboard operable (arrows + Space/Enter); fields and CTA re-label on switch; selection not conveyed by colour alone.
  • Env-derived MCP URL: the prompt’s claude mcp add URL matches NEXT_PUBLIC_APP_URL (staging vs prod), not a hard-coded host.
  • q ≤ 5000 guard: with a very long idea title / task scope the prompt is truncated from the work-context tail while the MCP-setup head is preserved.
