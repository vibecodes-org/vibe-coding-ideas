Proposal 1 Project Kits — Idea Archetypes

Addresses audit findings F6, F12, F22 (critical) + F1, F18, F19, F24 (high). Inspired by Notion's use-case segmentation, Shortcut's workflow presets, and Linear's opinionated defaults.

The Core Idea

When a user creates an idea (during onboarding or later), they select a project type. This single choice auto-configures: a workflow template, a recommended agent team composition, suggested board labels, and optionally pre-generated starter tasks. Think of it as "Notion templates but for the entire VibeCodes project setup."

User Feedback

"Workflow templates and agents tend to go together. A website build implies certain workflow templates and certain agents. Other project types need different ones." — This validates bundling configuration by project type rather than configuring agents, workflows, and labels independently.

Project Type Selector

Replaces the current bare "Title + Description + Tags" idea creation form. Appears in both onboarding (Step 3) and the standalone "Create Idea" page.

Competitive Precedent

Notion: Use-case segmentation pre-loads relevant templates. Shortcut: Workflow presets (Standard / Simple / Custom) let users self-select complexity. Jira: Project templates bundle work types + workflows + board type simultaneously. VibeCodes' Project Kits take this further by also bundling agent teams.

Implementation Notes

  • Data model: New project_kits table (or extend workflow_library_templates) with: name, icon, description, workflow_template_id, agent_roles (jsonb), label_presets (jsonb), board_column_presets (jsonb)
  • Admin-managed: Kits configured via the existing admin panel (extends the template library concept)
  • "Custom" escape hatch: Always available for users who want manual control
  • Retroactive: Existing ideas can adopt a kit via a new "Apply Kit" option on the board's Workflows tab
  • Replaces: Featured Teams (absorbed into kits) and the separate template library import (kits include templates)

Proposal 3 MCP as First-Class Activation Step

Addresses user feedback: "MCP is fundamental — it's the orchestrator." Elevates MCP connection from an optional footnote to the core activation moment.

User Feedback

"Connecting Claude Code via MCP is absolutely essential — it's the way agents are controlled, it's the orchestrator of the work — it's fundamental." The current UX treats MCP as an optional afterthought (checklist item marked "OPTIONAL", no detection, hardcoded as incomplete). This proposal makes it central.

Changes Required

1. Onboarding: Dedicated MCP step (shown in P2 above)

MCP connection becomes Step 5 of the Golden Path, with explanation of why it matters and what happens when you connect.

2. Dashboard: Persistent MCP banner for unconnected users

Dashboard — MCP Connection Banner
📡
Your agents are waiting — connect Claude Code
You have 5 agents and 8 tasks ready to go, but agents need Claude Code (MCP) to work. Run this command in your terminal:
$ claude mcp add vibecodes https://vibecodes.co.uk/api/mcp

This banner appears until MCP connection is detected. Shows on dashboard and board pages.

3. MCP Connection Detection

Instead of hardcoding done: false, detect MCP connection by checking the mcp_tool_log table for any tool invocations by this user. If any exist, the user has connected successfully.

  • Query: SELECT EXISTS(SELECT 1 FROM mcp_tool_log WHERE user_id = $1 LIMIT 1)
  • Cache: Store result on users.mcp_connected_at (new column, set on first MCP tool call)
  • UI impact: Dashboard banner disappears, checklist step auto-completes, board unlocks agent-powered features

4. Board: Conditional agent-work indicator

On the board page, when MCP is connected, show agent activity indicators (which agent is working on what step). When NOT connected, show a subtle bar: "Your agents need Claude Code to work. Connect now"

Proposal 4 Cross-Feature Connective Tissue

Addresses F7, F10, F13, F18, F23, F26, F27 — features that exist in isolation without bridges between them.

Principle: Every screen tells you what to do next

Instead of dead-end empty states, every screen should show the logical next action based on the user's current state. This follows Notion's "learn by doing" and Linear's "constraint as teacher" patterns.

Design Rationale

This replaces the current "wall of empty states" with a focused, state-aware dashboard. Key principles: Linear's "constraint as teacher" (only show what's relevant), Notion's interactive checklist (progress visualization), and the research finding that hiding 4 of 5 empty state actions increased activation by 34%. The first-run dashboard shows only what the user needs NOW, not everything the app can do.
