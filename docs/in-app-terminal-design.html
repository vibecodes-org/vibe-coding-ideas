UX Design · Step 2 · In-App Local Claude Code Terminal · Revision 2 · P1

“In the browser” — a new way to run the same Launch Claude Code button

Compass · UX Designer This is not a new button. The existing Launch Claude Code control on your idea board now asks where Claude should run — you pick one: “In a terminal window” (on your own computer — exactly how it works today, nothing to install) or “In the browser” (new). Pick In the browser and — the first time only — you install a small VibeCodes desktop helper; the OS asks “Open VibeCodes?”, the helper starts claude on your machine, and the live session renders in a terminal panel docked at the bottom of the board (VS Code convention). You run Claude in one place at a time — a terminal window or the browser, never both. Opt-in, feature-flagged, owner-bound, read-write with a read-only kill-switch, mac + Windows. No command typing in the everyday path.

⟳ What changed in this revision

Sent back at the Design-Review gate. The fixes:

  • One entry point, not two. Launch is now a mode of the existing “Launch Claude Code” button — terminal window (today) vs in the browser (new) — pick one; the helper install is only needed for the in-browser mode. No second, parallel “Launch terminal” button anywhere (§1–§2).
  • Launch is a button, not a pasted command. Choosing In the browser = install the helper app once → OS “Open VibeCodes?” → terminal appears. The old npx paste survives only behind an Advanced disclosure (§2–§3).
  • Pairing is automatic on the same machine. The vibecodes:// deep link carries the signed session identity, so the common case has no code to type. The manual device-code is now just the labelled cross-machine fallback (§4).
  • New hero: the board mockup. The terminal is shown docked on a real idea board — collapsed (slim bar) and expanded (live session) — at realistic proportions (§1).
  • End-to-end “how it works” strip added: install once → click Launch → OS opens helper → terminal appears (§2).
  • Gap fixes folded in: a distinct “bridge belongs to another account” owner-mismatch error; non-alarming idle-timeout / max-duration ended copy; a subtle slow-connection indicator on the Connected pill; and the two independent clocks (connect timeout vs session limits) called out (§6–§7).
  • The helper ships signed, notarized & verified. The first-run OS step now shows the verified-developer / verified-publisher experience, not the scary “unidentified developer” warning — plus an “Is this safe?” reassurance panel for non-coders (§3). Code-signing + notarization is now a hard P1 requirement (§3, gate).
Building on Step 1 (Requirements). The ProdOwner slice fixed: single-attach (one helper per session), opt-in + feature-flagged, owner-bound (the session belongs to the authenticated human, not a bot identity), read-write with a read-only kill-switch, and mac + Windows. The relay is opaque (no plaintext at the Cloudflare DO). Two open questions went to UX: Q2 pairing UX; Q3 where the panel lives. Both are answered below and reflect the reviewer’s direction: pairing should be click-to-launch, auto-paired for non-coders, with manual codes kept only as a fallback.

3 · First run — install the helper (signed, notarized & verified)

This step is reached only when the user picks “In the browser.” The “In a terminal window” choice never lands here — it needs no helper. On the very first in-browser launch there’s no helper to open yet; we catch that and walk the user through a normal download → install → trust flow. Because the helper is code-signed and notarized, the OS shows a calm verified-developer confirmation — there’s still one click the first time, but it reads as safe and routine, not a scary “unidentified developer” warning. We’re honest that the click exists; we just make sure it’s the trustworthy version of it.

6b · Two independent clocks

Naming the two clocks prevents a whole class of confusion. A timeout while connecting means “we never reached your machine” (network/helper problem → retry). A timeout while connected means “we ended a long-idle session on purpose” (expected, safe → relaunch). They look different, read differently, and never share copy — so the user always knows whether something broke or simply expired.

The kill-switch is signalled three ways — never colour alone. (1) a violet 🔒 Read-only badge in the header, (2) the toggle showing aria-pressed="true" + “on”, and (3) the input bar replaced by an explanatory note (not merely disabled — a disabled box with no reason violates the persona rule). Flipping back is one click. Critically, the stream keeps flowing in read-only — freezing input must never mean going blind. Violet keeps it distinct from the green “Connected” state, so “connected” and “read-only” read as two independent facts.

8 · All connection states

Every state is distinct, labelled, and carries icon + text + actionable next step. A terminal that only handles “connected” is unshippable.

◴ Waiting to pair

Cross-machine fallback only — bridge started, code not yet entered. (Same-machine launch skips straight to Connecting.)

→ Enter the code from the remote machine, or just use Launch on your own computer.

◐ Connecting…

Launch clicked / code accepted; relay handshaking. Animated indicator, no input bar yet. Connect-timeout clock runs (≈30s) → Error.

→ Click Open in the OS prompt; otherwise just wait. Cancel reverts to idle.

● Connected

Live stream + read-write input (or read-only if killed). May carry a 🐢 slow-connection note (§6a). The hero state.

→ Watch, or type to steer.

◌ Disconnected — reconnecting

Bridge dropped (laptop slept, Wi-Fi blip). Auto-reconnect with backoff; last screen dimmed + held, not cleared.

→ Auto-retrying (2s, 4s, 8s…). “Reconnect now” + “End session” available.

◾ Session ended

Clean stop — End pressed, bridge exited (Ctrl-C), or a session-limit clock fired (idle / max-duration). Reason named; scrollback frozen + readable.

→ “Launch again” reopens the flow.

✕ Error

Connect timed out, helper not found, version mismatch, relay rejected, or owner mismatch. Names the cause + a fix.

→ Specific recovery (retry / install helper / update / sign in as owner).

Owner mismatch is its own first-class error, not a generic failure. When a redeemed code (or a relay attach) belongs to a different human, we say so plainly — “This bridge belongs to another account” — name who started it, and explain the rule (a bridge attaches only to its launcher). The recovery isn’t “retry” (that can’t help); it’s switch account or start your own bridge. This makes the owner-binding requirement legible at exactly the moment it bites. Ended ≠ Error: a clean/idle stop is calm grey with one forward action; an error is rose, names the specific cause, and hands over the exact fix.

9 · Responsive — desktop vs 375px

9a · Desktop (≥768px)

Full dock as shown in §1b — a bottom panel ~40% of the board height, drag-to-resize via the grip, fullscreen toggle, persistent input bar. Header controls sit in one row; identity truncates middle-out (Nick’s MacBook Pro · ~/pro…/recipe-saver · a3f9) so the state pill + End never get pushed off.

9b · 375px mobile — the honest call: read-only, view-first

…/recipe-saver/board
● Connected 🔒 Read-only
● Edit recipe-list.tsx ⎿ +28 -4 ● Bash npm run test ⎿ ✓ 142 passed watching live…
📱 Read-only on mobile. Driving a terminal needs a keyboard + width. Open on desktop to type — or use the kill-switch & End here.
The honest call: a phone can watch but shouldn’t drive. A full xterm with a soft-keyboard, escape sequences, and arrow-key navigation is genuinely bad UX — pretending otherwise fails the “does it work at 375px?” test dishonestly. So below 768px the panel is read-only by default: the live stream stays, and the two safety controls that matter on the go — the read-only kill-switch (already engaged) and End (a ≥44px target) — remain; keystroke input hides behind an honest note pointing to desktop. Safety controls are never desktop-only — you must be able to stop a runaway agent from your phone.

Option 3 · Always-on background helper FUTURE · own task

The endgame is zero clicks: a tiny background helper that’s already running and signed in, so choosing Launch Claude Code → In the browser attaches instantly with no OS prompt at all — the deep link goes straight to a listening local agent. That’s the consumer-grade ideal.

  • P1 ships Option 2 (this doc): install once, click Launch, one OS confirm. It’s the honest, shippable middle — real one-click feel without a always-resident background process to design, secure, auto-update, and earn trust for.
  • Option 3 is tracked as its own task — it adds a login-item/agent lifecycle, background-permission story, and auto-update channel that deserve their own design + security review. Building Option 2 first doesn’t block it; Option 3 layers onto the same signed deep-link + relay.
Progression: Option 1 (paste a command, now demoted to Advanced) → Option 2 (P1: install once + Launch button) → Option 3 (always-on, zero clicks). Each step removes friction without changing the underlying signed-link + opaque-relay model.

Design-Review gate — what to sign off

  • Launch model: Approve making in-browser a mode of the existing “Launch Claude Code” button — a pick-one menu (“In a terminal window” = today, unchanged / “In the browser” = new, install-once helper), never a second parallel button — as the P1 primary path (npx demoted to Advanced)?
  • Pairing: Approve automatic same-machine pairing via signed vibecodes:// deep link, with the manual device-code kept only as the labelled cross-machine fallback?
  • Home (Q3): Approve the per-idea board terminal dock (collapsible bottom panel) as the P1 home?
  • OS trust step: Approve forewarning the OS “Open VibeCodes?” prompt honestly in-app rather than hiding it?
  • Signing (hard requirement): Confirm code-signing + notarization is a hard P1 requirement — the helper must show the verified-developer / verified-publisher prompt, with no “unidentified developer” warning ever reaching a user?
  • Kill-switch: Approve read-only that freezes input but keeps the stream live, badged three ways?
  • Mobile: Approve read-only-on-mobile with safety controls retained, driving redirected to desktop?
  • Flag-off: Approve not rendering the entry point when the flag is off?
  • Gap fixes: Approve the distinct owner-mismatch error, idle/max-duration ended copy, slow-connection indicator, and the two-clocks model?
