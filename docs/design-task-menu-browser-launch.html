Run Claude Code in the browser — task card menu item

UX Design step deliverable · Task: “Add a run Claude Code in browser menu item”

Compass (UX Designer) · Building on the completed Requirements step (see that step's document on this board for FR1–FR7 and acceptance criteria 1–9)

1 · Overview

Building on the Requirements step: the board toolbar's split-button dropdown already lets people choose where Claude Code runs — “In a terminal window” or “In the browser” (Beta) — but the per-card ⋮ context menu only offers the terminal-window launch. Users who work task-by-task from the card menu have no path to the in-browser terminal without walking back to the toolbar and losing the task scope.

This design adds a second launch item to the card menu's existing launch section, rendered by the variant="task-menu-item" branch of src/components/board/launch-claude-code-button.tsx inside src/components/board/task-card-menu.tsx. It follows FR1–FR7: terminal first, Globe + Beta parity, byte-identical menu when the flag is off, desktop-only, reuse of handleLaunchInBrowser (task-scoped prompt + cwd parity), PostHog parity, and pick-one semantics.

2 · Design decisions

2.1 Label wording — “Launch in browser terminal”

Decision: the existing item keeps its exact label, “Launch in Claude Code” (Terminal icon), and the new item is “Launch in browser terminal” (Globe icon + Beta pill), placed second.
  • “Launch in Claude Code” stays untouched. FR3 requires the flag-off menu to be byte-identical to today, and keeping the label constant in both flag states means muscle memory never breaks — the first item is always the same action with the same name (Jakob's Law: don't move or rename the default).
  • Why not “Launch in browser”? Information scent: in a task card menu, “Launch in browser” could plausibly read as “open this task in a browser tab.” “Launch in browser terminal” says exactly what appears — a terminal, in the browser — and echoes the toolbar's description copy (“A live terminal docked on this board”) in one word.
  • Why not reuse the toolbar's “In the browser”? The toolbar items sit under a labelled “Launch Claude Code” split button, so the fragment “In the browser” inherits its context. The card menu has no such header — items are self-contained, verb-first commands (“Move to top”, “Convert to conversation”, “Archive”). A bare “In the browser” next to “Move to bottom” has weak scent. Verb-first parallel construction (“Launch in…” / “Launch in…”) keeps the pair visually and grammatically grouped (Gestalt similarity) while the shared prefix + adjacency (proximity) signals “same action, different destination.”
  • Mental-model parity is preserved where it matters: same ordering (terminal first), same icons (Terminal / Globe), same Beta pill recipe. A user who learned the choice on the toolbar recognises it instantly here.

2.2 Single-line items, no descriptions

Decision: both launch items are single-line — no 11px muted description lines in the card menu.
  • Internal consistency beats cross-surface mimicry. Every item in the card menu is a compact single-line command. Introducing one two-line island would break the menu's rhythm (Gestalt similarity within the group the user is actually scanning) and visually over-weight the launch section against Archive/Delete.
  • The toolbar's descriptions solve a different problem. There, the fragments “In a terminal window” / “In the browser” need the explanatory second line; here, the fuller labels already carry that meaning, so the description would be redundant (recognition over recall — the label itself is recognisable).
  • Width and scan cost. The card menu is a narrow, quick-action surface; two-line items nearly double its height and slow the scan for the destructive actions below (Hick's Law — keep per-item decision cost low).
  • The Beta pill alone flags the newer, less-proven path — the same signal the toolbar uses.

2.3 Placement and grouping

  • Both items live in the existing launch section in task-card-menu.tsx — between the separator after “Move to bottom” and the separator before “Convert to conversation.” No new separators between the two launch items: they are one choice, two destinations (pick-one, FR7), so they share a group.
  • Order (FR1): terminal window first, browser second — identical to the toolbar and to launchModeOptions() which always returns ["terminal-window", "browser"].
  • Item padding keeps the existing py-2.5 sm:py-1.5 recipe — ≥44px touch height on small screens, compact on desktop.

4 · Interaction flow — “Launch in browser terminal”

1

Trigger

User opens the card's ⋮ menu (click, or keyboard focus + Enter — the trigger already supports focus-visible reveal). Board is editable; viewer is on desktop; terminal flag is on.

2

Choose destination

The launch group shows both items. Arrow keys move item-to-item; typeahead “L” cycles the two Launch items. User selects Launch in browser terminal (Enter, Space, or click).

3

Menu closes, event fires

Radix closes the menu on select (pick-one — no lingering state). handleLaunchInBrowser captures the PostHog event launch_claude_code_clicked with method: "in_browser" (FR6) and dispatches the vibecodes:terminal-browser-launch CustomEvent via requestBrowserLaunch with the compact prompt head/tail + resolved cwd.

4

Dock opens at the bottom of the board

The board's terminal dock (a page-level sibling) opens — or comes to the fore if already open — mints a session, and auto-launches via its vibecodes:// deep link, budget-truncating the prompt tail to fit the URL ceiling.

5

Session boots task-scoped

Because the component was rendered with variant="task-menu-item", buildCompactParts already embeds taskId — Claude boots against this task, in the same cwd the terminal-window deep link would have used (FR5: prompt + folder parity for free).

Selecting Launch in Claude Code instead runs today's flow unchanged: handleLaunch → claude-cli:// deep link, with the existing double-click guard (launchingRef), fallback toast, and copy-command escape hatch.

5 · Accessibility

  • Keyboard: both items are ordinary Radix DropdownMenuItems — full arrow-key navigation, Home/End, Enter/Space to select, Esc to close and return focus to the ⋮ trigger, plus typeahead. No custom key handling needed. onCloseAutoFocus is already suppressed by the card menu so closing doesn't yank scroll position.
  • No hover-only affordance: the item is always visible in the open menu; the ⋮ trigger itself is always visible on mobile and revealed on desktop by hover or focus-visible or open state — existing behaviour, unchanged.
  • Beta pill contrast: text-sky-400 (#38bdf8) on bg-sky-500/15 composited over the popover (~#18181b) computes to ≈ 6.6 : 1 — passes WCAG 2.1 AA for normal text (4.5:1) despite the 10px size, and stays ≥ 4.5:1 over the bg-accent (zinc-800) focus state. It's the exact recipe already shipped in the toolbar, so no new tokens.
  • Not colour-only: the two destinations differ by icon (Terminal vs Globe), label text, and the literal word “Beta” in the pill — colour is reinforcement, never the sole carrier.
  • Target size: items keep py-2.5 below sm (≥44px tall full-width rows) and sm:py-1.5 on desktop pointer contexts.
  • Screen readers: the pill is plain text inside the item, so the accessible name reads “Launch in browser terminal Beta” — self-describing, no aria-label override required.
