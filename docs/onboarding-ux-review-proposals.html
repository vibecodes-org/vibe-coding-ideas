2 · MCP & Launch reframe highest priority

Today the "how do I make agents actually run?" answer is taught in three places, and the easy answer is the hidden one:

SurfaceWhat it teachesScopeSource
Onboarding step 4 "Connect Claude Code"Manual claude mcp add … vibecodes-remote + decorative "✓ Connected"-s useronboarding-dialog.tsx:908–947 via constants.ts:321
First-run dashboard "Connect Claude Code to Your Board"Same manual command (MCP_COMMAND)-s userfirst-run-dashboard.tsx:385+
Board "Launch Claude Code" flowAuto-prepends mcpSetupHead() to every launch prompt — already sets up MCP for the user-s locallaunch-claude-code.ts:353–358

The Launch flow is the gentlest path (one click → a prompt that includes the MCP setup), yet it is board-only and desktop-only, while the harder hand-typed command is what every new user meets first. The ordering is backwards.

2.1 Target model: Launch is the primary "connect / activate" action

Everywhere MCP is currently taught, lead with a single primary CTA — Launch Claude Code — that opens Claude Code with the MCP setup pre-loaded. The manual command moves under a "No Claude Code? · Manual setup" disclosure.

Hierarchy decision: Primary = Launch Claude Code (deep-link / copy-launch-command). Secondary = Copy manual command inside a collapsed disclosure. Tertiary = "I'll do this from the web UI" / skip. This applies identically to onboarding step, first-run card, and the dashboard banner so the user learns one mental model (Jakob's law — consistency across surfaces).

2.2 Resolve the scope inconsistency (-s user vs -s local)

Recommendation — make -s local canonical. The Launch flow already chose -s local for a stated reason (launch-claude-code.ts:357): "local scope intentionally overrides any existing project vibecodes-remote (e.g. a local stdio server) so there is no conflict." That rationale is correct and the manual command in constants.ts:321 should match it. Adopt -s local everywhere and update MCP_COMMAND.
Flag for engineering / the gate: Confirm this is a documentation inconsistency, not a live connection bug. If users who ran the -s user command currently end up with a working-but-differently-scoped connection that the app then can't see (e.g. a project-scoped stdio server shadows it), that is a real connection bug and should be fixed alongside the copy change, not after. This is a UX-flagged engineering decision, not a UX call.

2.3 Close the verification gap (connection is self-reported)

Step 4 advances on a button labelled "I've connected — let's go!" with no check; the "✓ Connected as You" terminal lines are hardcoded decoration (onboarding-dialog.tsx:924–939). Users who didn't actually connect believe they did → silent failure later.

  • Replace the self-report button with a live status row that polls a connection signal and renders three honest states: Waiting for connection… → Connected ✓ (auto-advances), with a manual "Skip — I'll connect later" always present.
  • Truth source: a mcp_connected_at timestamp on the user (or a lightweight "connections" probe) set when the MCP server first authenticates a real session.
Dependency: Needs a small backend confirm — surface real mcp_connected_at and a client poll/realtime subscription. Until that lands, ship an interim: keep the manual button but relabel it honestly to "I'll connect now — take me to my board" (no false "✓ Connected"), so we don't claim verification we don't have. The auto-advance is the dependent slice; the honest copy is shippable today.

2.4 Mobile off-ramp

Onboarding's connect step on mobile currently instructs "Run this in your terminal where you code" (onboarding-dialog.tsx:967) with no escape — a dead end on a phone. The fix already exists nearby: FirstRunMcpCard renders a cyan "You'll need a computer for this step" card with a copy-setup-link when isMobile (first-run-dashboard.tsx:388, 423–436). Reuse that exact pattern in the onboarding connect step, and let mobile users finish onboarding without connecting.

3 · Dashboards decision

Recommendation: MERGE into one dashboard. Render the standard dashboard for everyone, with a dismissible / auto-collapsing "Getting set up" checklist that disappears on completion. Retire the first-run/standard boolean and the manual "Switch to full dashboard" override.

Why merge (not keep two)

  • The audit found the only genuinely first-run-only element is the Setup-Progress tracker; everything else is a reduced copy of the standard panels. Two near-identical dashboards is duplicated maintenance with no payoff.
  • computeIsActivated (dashboard-activation.ts:11–27) mis-fires: hasManualEngagement = hasUserActivity || hasMcpConnection treats AI-auto-created board content as not-yet-engaged, so a user with a complete generated board is still stuck on first-run. The existence of a manual "Switch to full dashboard" override is the tell that the heuristic is wrong.
  • A checklist that completes itself is honest information scent: the user sees exactly what's left, and the surface fades when there's nothing left — instead of a hidden boolean deciding which of two screens they get.

The activation-logic change this implies

Replace the "which dashboard?" boolean with a per-item checklist completion model. Keep the same underlying signals (hasTasks, hasAgents, hasWorkflows, hasMcpConnection) but use them to drive checklist item states, not a dashboard fork. The checklist auto-collapses when all required items are done (or the user dismisses it). This deletes the need for computeIsActivated's fragile AND-chain and the FIRST_RUN_OVERRIDE_KEY escape hatch (first-run-dashboard.tsx:17).

Minimal-change fallback (if merge is deferred): if the team wants to keep two dashboards short-term, the mis-fire must be fixed regardless — count an AI-generated board as activation (treat hasTasks && hasAgents from a generated board as engagement), and fix the "First task" bot-vs-human tick (F3.4) so a task created by an agent doesn't falsely tick a "you created your first task" step. But this is the inferior option — it preserves the duplication.

5 · Onboarding flow fixes

5.1 Step counter

The counter is wrong and asymmetric: StepIndicator is told totalSteps={6} (onboarding-dialog.tsx:278) but hardcoded labels say "Step 2 of 6" (:381), "Step 3 of 6" (:496) — and the loading and live states have no counter at all. Fix: derive the label from the same source as the indicator, label every step (including loading as "Setting up…"), and drop the count by one once the profile step moves out (see 5.2) so it reads honestly as 4–5 steps.

5.2 Reconsider the profile step's placement

Recommendation: defer the profile step to post-activation. The profile step (bio + GitHub, both optional — :368–479) sits between Welcome and the value moment (creating a project), interrupting the path to first value (Fitts/Hick — fewer choices before the payoff). Move it to a dismissible "Complete your profile" nudge on the dashboard after the board exists. Display name can default from the account; bio/GitHub are not needed to get value.

5.3 Escape affordance contrast

The modal blocks Escape and outside-click (:267–268) and the only exit is a very low-contrast "Skip for now" at text-muted-foreground/60 (:360) — fails WCAG 1.4.3 contrast for the one escape route. Fix: keep the forced modal but raise the skip link to a properly-contrasted ghost button (≥ 4.5:1), and add a visible top-right close that maps to "Skip for now" so the escape is discoverable (recognition over recall).

5.4 Honest creation-progress copy

The progress bar fakes momentum (creeps 55→92% on a timer regardless of real state — :690) and the timer copy says "almost there" before completion. Fix: tie bar segments to the real createProgress milestones only, and use neutral honest copy ("Generating tasks — this can take up to a minute"). Don't claim "almost there" when we don't know.

6 · The launch-button-in-onboarding question (task c4614bf5)

The separate task c4614bf5 proposes introducing the Launch Claude Code button into onboarding. That task is exactly what §2.1 does — but §2 is broader: it makes Launch the primary connect action everywhere MCP is taught (onboarding step + first-run card + dashboard banner), resolves the -s user/-s local scope conflict, and closes the verification gap. A Launch button dropped into onboarding without those is half a fix (it would coexist with the contradictory manual command it's meant to replace).

Recommendation: absorb c4614bf5 into this work as the implementation slice for §2.1. Keep it as the named ticket that delivers "Launch button in onboarding," but scope it to follow the §2 hierarchy (primary Launch, demoted manual disclosure) rather than shipping a bare button. The scope fix (§2.2) and verification (§2.3) are sibling tickets in the same slice.

  • Decide Scope conflict is doc-only vs real bug. Is the current -s user command producing connections the app can't verify (shadowed by a project-scoped stdio server)? Engineering confirm needed before §2.2 copy change ships. (Engineering call, UX-flagged.)

  • Decide Merge vs keep-two dashboards. UX recommends merge (§3). If product prefers two for non-UX reasons, the mis-fire fix is non-negotiable either way.
  • Decide Defer the profile step? UX recommends deferring (§5.2). Product owns whether bio/GitHub capture at signup has a business reason to stay inline.
