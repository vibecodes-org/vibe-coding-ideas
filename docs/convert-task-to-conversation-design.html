Convert Task to Conversation

A shortcut-menu action that turns a board task into an idea discussion (idea_discussion) and archives the task off the board, keeping a two-way link between them.

↳ Builds on the Requirements step Dark · Zinc · New York Non-destructive action Optimistic UI

1. Intro & design goals

This document designs the UI/UX for the Convert to conversation feature against the decisions locked in the Requirements step. Those decisions are treated as fixed constraints here, namely:

  • A new “Convert to conversation” item in the task ⋯ menu (src/components/board/task-card-menu.tsx), in its own group above Archive/Delete, rendered in non-destructive style with a MessagesSquare icon, shown only to team members (!isReadOnly).
  • A confirm dialog “Convert to a conversation?” explaining the task is archived (not deleted) and linked to the new conversation, with a conditional workflow warning line when workflow_step_total > 0; buttons Cancel / Convert (Convert is primary, not destructive).
  • On confirm: optimistic removal from the board, a success toast that links to the new discussion, and error toast + rollback on failure.
  • The archived task keeps a discussion_id back-link; the detail dialog already renders a “From discussion — View source thread” badge (task-detail-dialog.tsx) which we reuse for the reverse direction.

Design goals. (1) Recognition over recall & information scent — the action and its consequence are self-explanatory before the user commits. (2) Reversibility-minded — make it obvious this is archive, not delete, so the action feels safe (Fitts: the safe action lives away from the destructive Delete). (3) Consistency / Jakob's Law — reuse the menu, dialog, and toast patterns already in the board exactly, so nothing new has to be learned. (4) All states handled — default, workflow-warning, in-flight, error+rollback, and long-title edge cases. (5) WCAG 2.1 AA throughout.

2. End-to-end user flow

Trigger → confirm → optimistic removal → success/failure → re-finding the discussion later, including error recovery and back-navigation.

1
Trigger. A team member hovers a task card (or focuses it via keyboard) and opens the ⋯ actions menu. The new Convert to conversation item sits in its own group, visually separated above Archive/Delete. Read-only viewers never see the menu, so the item is gated automatically by the existing !isReadOnly wrapper in board-task-card.tsx.
2
Confirm. Selecting it opens the AlertDialog “Convert to a conversation?”. The body names the task in quotes and explains the outcome: a new conversation is created and this task is archived, not deleted. If the task carries a workflow (workflow_step_total > 0) an amber warning line appears, telling them the attached workflow won't carry over. Focus lands on Cancel (the safe default for an irreversible-feeling step).
3
Commit (optimistic). On Convert, the button enters a busy state, then the dialog closes and the card is removed from the board immediately using the same ops.trustRemoval() + incrementPendingOps() recipe Archive uses — so a lagging Realtime snapshot can't re-show it.
✓
Success. A success toast confirms the conversion and offers a View conversation action that deep-links to /ideas/{ideaId}/discussions/{discussionId}. The card stays gone.
!
Failure & recovery. If the server action throws, the card rolls back into place at its previous position and an error toast explains what happened. No data was lost; the user can retry from the same menu. This mirrors the catch/rollback branch of handleArchive.
4
Re-finding it later. The new discussion lives in the idea's Discussions list like any other. The archived task is still reachable (Show archived filter on the board / dashboard exclusion aside) and its detail dialog shows the existing blue “From discussion — View source thread” backlink, so the two halves always point at each other. Back-navigation from the discussion returns to wherever the user came from via normal browser history.
Success path Workflow-warning variant Error → rollback → retry

5. Accessibility notes (WCAG 2.1 AA)

  • Keyboard flow. The ⋯ trigger is reachable by Tab and reveals on focus-visible (already implemented). ↑↓ move through menu items, Enter/Space selects, Esc closes — inherited from Radix DropdownMenu. Disabled items (e.g. Move to top at the edge) are skipped.
  • Dialog focus management. Radix AlertDialog traps focus, moves initial focus to Cancel (safe default), and restores focus to the trigger on close. Esc and the backdrop both cancel. While submitting, focus stays within the dialog and both buttons are disabled with an explanatory busy label (no bare disabled button).
  • Contrast. Primary Convert button is near-white on the zinc surface (well above 4.5:1). The amber warning text (#fcd34d) on the dark amber-tinted panel exceeds 4.5:1; the blue backlink and constructive label text also pass. Muted body copy (zinc-400) on zinc-900 meets 4.5:1 for the ≥14px sizes used.
  • Non-colour signalling. The workflow warning uses a triangle-alert icon + explicit text + a bordered container, not colour alone. The menu item conveys its constructive nature through label + icon, not tint. Toasts pair a check/X icon with text.
  • Touch targets / mobile (375px). The trigger is 44×44 on mobile (h-11 w-11, shrinking to h-7 w-7 on sm) and always visible on touch. Menu items use py-2.5 on mobile for a ~40px+ row. The dialog is full-width with max-w margins; footer buttons stack comfortably and stay above the fold. The success-toast View conversation action button is ≥44px tall on touch.
  • Announcements. Success toast uses role="status" (polite), error toast uses role="alert" (assertive) so screen-reader users hear the rollback. The dialog is a true alertdialog with aria-labelledby/aria-describedby.
  • Labels not placeholders. No free-text input in this flow, so no placeholder-as-label risk. The trigger keeps its aria-label="Task actions".

7. Handoff to Implementation

Reuse — no new design-system primitives needed

  • Menu: add one DropdownMenuItem + a wrapping DropdownMenuSeparator group inside task-card-menu.tsx, between the Launch group and the Archive group. Icon MessagesSquare from lucide-react. Match sibling classes py-2.5 sm:py-1.5. Do not set variant="destructive".
  • Dialog: a second AlertDialog alongside the existing delete one, driven by a new confirmConvertOpen state. Reuse AlertDialogHeader/Title/Description/Footer/Cancel; the Action button is default variant (not destructive).
  • Optimistic mutation: model handleConvert on the existing handleArchive — ops.archiveTask()-style removal (or a dedicated ops.convertTask() returning a rollback), incrementPendingOps(), trustRemoval(), and the catch branch that calls rollback() + trustRemoval(id,false) + toast.error.
  • Toast action: use the sonner action: { label, onClick } shape already used in kanban-board.tsx / launch-claude-code-button.tsx; onClick routes to /ideas/{ideaId}/discussions/{discussionId}.
  • Reverse backlink: already exists in task-detail-dialog.tsx (gated on task.discussion_id) — no UI change; just ensure the server action sets discussion_id on the archived task.

New work for Implementation (non-UI)

  • A server action (e.g. convertTaskToDiscussion(taskId, ideaId) in src/actions/discussions.ts, sibling to the existing convertDiscussionToTask) that: creates the idea_discussion from the task's title/description, archives the task, and writes discussion_id back onto the task — returning { discussionId } for the toast link. Use the project's concurrency-guard pattern (.eq("archived", false) + .maybeSingle()) so a double-click can't double-convert.
  • An optional ops.convertTask() helper in BoardOpsContext returning a rollback, if a distinct activity-log verb ("converted") is wanted versus reusing archiveTask.
