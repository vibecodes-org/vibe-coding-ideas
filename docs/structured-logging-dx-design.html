Structured Logging — DX Design

Infrastructure Developer Experience — Building on Requirements from Product Owner

Note: This is a backend infrastructure task with no user-facing UI. This design document covers the developer experience — API surface, usage patterns, output format, and migration guide.

1. API Surface

Logger Interface

// src/lib/logger.ts
import { logger } from "@/lib/logger";

// Each method: (message: string, context?: Record<string, unknown>) => void
logger.error("Failed to send email", { userId, error: err.message });
logger.warn("AI output truncated", { tokenCount, maxTokens });
logger.info("Workflow step completed", { stepId, taskId });
logger.debug("Cache miss", { key, ttl });

Level Hierarchy & Defaults

LevelPriorityConsole MethodUse Case
error0 (highest)console.errorFailures requiring attention
warn1console.warnDegraded behavior, unusual conditions
info2console.infoSignificant operations (future use)
debug3 (lowest)console.debugDevelopment diagnostics (future use)
EnvironmentDefault LOG_LEVELWhat gets emitted
Productionwarnerror + warn
DevelopmentdebugAll levels
Design decision: Default to warn in production (not error) so that existing console.warn calls continue to emit after migration. This matches current behavior — both console.error and console.warn output in production today.

2. Output Format

Structured JSON Output

// logger.error("Failed to send email", { userId: "abc", error: "Connection refused" })
// outputs to console.error:
{
  "level": "error",
  "message": "Failed to send email",
  "context": {
    "userId": "abc",
    "error": "Connection refused"
  },
  "timestamp": "2026-03-21T16:30:00.000Z"
}

// logger.warn("AI output truncated") — no context
// outputs to console.warn:
{
  "level": "warn",
  "message": "AI output truncated",
  "timestamp": "2026-03-21T16:30:00.000Z"
}

When no context is provided, the key is omitted entirely (not null). This keeps log lines clean and grep-friendly.

4. Migration Examples (Before → After)

4a. Simple error with message

Before

// src/lib/activity.ts:23
if (error) console.error("Failed to log activity:", error.message);

After

if (error) logger.error("Failed to log activity", {
  error: error.message,
});

4b. Tagged error with context

Before

// src/app/api/notifications/email/route.ts
console.error("[Email] Resend API error:", resp.status, text);

After

logger.error("Resend API error", {
  status: resp.status,
  response: text,
});

4c. Catch-all with error object

Before

// src/actions/ai.ts:18
console.error("[AI Action Error]", err);

After

logger.error("AI action failed", {
  error: err instanceof Error ? err.message : String(err),
});

4d. Warning

Before

// src/app/api/ai/enhance/route.ts:121
console.warn("[AI Enhance] Output truncated");

After

logger.warn("AI enhance output truncated");

4e. Client-side components — NO migration

Important: 5 console.error calls live in client components ("use client" files). The logger module uses process.env.NODE_ENV which is available in both server and client contexts in Next.js, but process.env.LOG_LEVEL is only available server-side unless prefixed with NEXT_PUBLIC_.

Decision: Migrate client-side calls too — the logger should fall back to "debug" when LOG_LEVEL is undefined (which it will be on the client in dev). This means all client-side logs always emit in dev, and in production builds, NODE_ENV === "production" will set the default to "warn". No NEXT_PUBLIC_ prefix needed.

6. Import Path Considerations

Shared module between Next.js and MCP server

The logger lives at src/lib/logger.ts. The MCP server (mcp-server/src/) already imports from src/ indirectly via relative paths in some places, but for clean separation:

  • Next.js code (src/): imports via @/lib/logger (existing path alias)
  • MCP server code (mcp-server/src/): imports via relative path ../../src/lib/logger (same pattern as existing cross-boundary imports)

No new package, no new tsconfig path. Follows existing conventions.

7. Context Object Guidelines

What to include in context

KeyWhen to includeExample
errorAlways in error handlerserr.message or String(err)
userIdWhen available in scopeUUID string
ideaIdIdea-scoped operationsUUID string
taskIdTask-scoped operationsUUID string
statusHTTP errorsResponse status code
toolNameMCP tool contextTool name string

Rules:

  • Never log sensitive data (tokens, passwords, API keys, email addresses)
  • Use err.message not the full error object (avoids circular refs in JSON.stringify)
  • Keep context shallow — no nested objects deeper than 1 level
  • Prefer semantic keys over generic ones (stepId not id)

8. Testing Strategy

Unit tests for src/lib/logger.test.ts

  1. Level filtering — verify that calls below the configured level are suppressed
  2. Output format — verify JSON structure has level, message, timestamp, and optionally context
  3. Console method routing — logger.error → console.error, logger.warn → console.warn, etc.
  4. Default level selection — warn when NODE_ENV=production, debug otherwise
  5. LOG_LEVEL override — env var takes precedence over defaults
  6. Context omission — no context key when none provided
  7. Timestamp format — valid ISO 8601 string

Use vi.spyOn(console, "error") etc. to assert calls without polluting test output.

9. Future Compatibility (Phase 2 — not in scope)

The structured JSON output is intentionally compatible with:

  • Vercel Log Drains — JSON lines parsed automatically
  • Axiom / Betterstack — structured fields become queryable columns
  • pino swap — same API signature (logger.error(msg, ctx)) means a future pino migration only changes the implementation, not call sites
