UX Design v2: Reapply Fuzzy Matching on Agent Add/Remove

Agent: Compass (UX Designer) Task: Reapply fuzzy matching when adding or removing agents Date: 2026-03-22 Version: v2 (reworked) Status: No UI Changes Required

1. Summary

Building on the findings from the Requirements v2 step, this design review confirms that no UI changes are needed for this feature. The v2 requirements expand the scope significantly — when an agent is added, ALL pending steps are re-evaluated (not just unmatched ones), and a new match_tier column enables quality-based upgrade decisions. Despite this expanded backend scope, the existing UI components already handle all result states correctly.

This document validates that decision by walking through every user flow affected by the change, including the new tier-based upgrade scenario (a poorly-matched step being reassigned to a better-fit agent), confirming that all states are handled, and flagging future improvements.

2. Affected User Flows

Flow A: Adding an Agent to the Pool (v2 — tier-aware)

Agents Tab Server Action Workflow Steps UI ───────── ───────────── ───────────────── User clicks Steps show: "Add Agent" button Step 2: Atlas (word-overlap) │ Step 4: ⚠ no match ▼ Selects "Compass" (UX Designer) │ ▼ allocateAgent() ──────────► Insert idea_agents │ ▼ rematchWorkflowAgents() [NEW — fire-and-forget] │ ▼ Re-evaluate ALL pending steps with agent_role │ ┌─────┴─────┐ ▼ ▼ Step 4: Step 2: bot_id=NULL tier=word-overlap → assign new=exact (better!) Compass → upgrade to Compass │ │ ▼ ▼ Toast: "Agent revalidatePath() added to pool" │ │ ▼ ▼ Steps UI refreshes: Agent grid ✓ Step 2: Compass (upgraded) shows new agent ✓ Step 4: Compass (new match) ✓ Warning banner gone

✅ Decision: No UI change needed

The v2 flow now re-evaluates ALL pending steps, not just unmatched ones. When a poor match (e.g., word-overlap tier) is upgraded to a better one (e.g., exact), the bot_id changes on the step. From the UI's perspective, this is identical to the v1 flow — the warning banner checks bot_id === null and the step list re-renders on revalidation. The tier upgrade is invisible to the user (they just see a better agent assigned). The role coverage panel in the Agents tab also re-fetches on revalidation.

Flow B: Removing an Agent from the Pool

Agents Tab Server / DB Workflow Steps UI ───────── ────────── ───────────────── User clicks trash Steps show: icon on agent card Step 3: "Atlas (Dev)" │ Step 5: "Atlas (Dev)" ▼ removeIdeaAgent() ─────────► Delete idea_agents │ ▼ DB trigger clears board_tasks.assignee_id │ ▼ DB trigger clears bot_id + match_tier on PENDING steps [NEW — R3 trigger] │ ▼ rematchWorkflowAgents() [NEW — fire-and-forget] │ ▼ Try matching from remaining pool │ │ ▼ ▼ Toast: "Agent revalidatePath() removed from pool" │ │ ┌──────┴──────┐ ▼ ▼ ▼ Agent grid Match found: No match: removes agent Step updated ⚠ Warning banner with new bot reappears

✅ Decision: No UI change needed

When a removed agent's bot_id is cleared from pending steps and rematch runs, two outcomes are possible — both already handled by existing UI:

  • Replacement found: Step shows new matched agent. No warning.
  • No replacement: Warning banner reappears with the unmatched role(s). This is the correct behaviour — it tells the user to add an agent with the missing role.

Flow C: Collaborator Removed (Cascade)

Idea Settings DB Cascade Workflow Steps ───────────── ────────── ────────────── Author removes collaborator │ ▼ removeCollaborator() ─────► Delete collaborator row │ ▼ Trigger: delete all idea_agents by that user │ ▼ Trigger: clear bot_id + match_tier on pending steps [NEW — R3 trigger] │ ▼ App-level rematch [NOT triggered — cascade only, no server action]

Note: The cascade path (collaborator removal) will clear stale bot_id references via the extended DB trigger, but cannot trigger the app-level rematch. This is acceptable — the warning banner will surface unmatched steps, and the next agent allocation or manual rematchWorkflowAgents call will fill them in.

4. Accessibility & Responsiveness Check

No new UI elements = no new accessibility concerns. The existing warning banner uses semantic HTML, has sufficient colour contrast (amber on dark), and the Agents tab controls are already keyboard-navigable. No changes to the responsive layout are needed since the feature is purely backend.

6. Future Improvements (Not in Scope)

These are pre-existing UX gaps surfaced during this review. They are not blockers for the current task but worth recording for the backlog.

💡 F1: Show matched agent name on workflow steps

Currently, StepDetailDialog and the step list only show the agent_role badge, not which specific bot is assigned. Showing the bot's name and avatar alongside the role would help users understand who is handling each step — especially when multiple agents share a role.

Backlog — separate task, small effort

💡 F2: Toast or inline notification when rematch changes assignments

When an agent is added/removed and the background rematch reassigns steps, the user gets no explicit notification that assignments changed. A subtle toast like "Workflow steps rematched — 2 steps updated" would close the feedback loop. Currently the warning banner appearance/disappearance is the only signal.

Backlog — nice-to-have, low priority

7. Conclusion

Recommendation: Proceed with backend-only implementation as described in the Requirements step. All existing UI components handle the affected states correctly. The revalidatePath calls after rematch ensure the UI reflects the new state without any frontend code changes.

The two future improvements (F1, F2) should be logged as separate backlog items if the team agrees they add value.
