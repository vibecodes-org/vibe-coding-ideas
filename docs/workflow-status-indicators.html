UX Design Proposal

Workflow Status Indicators v2

Board task cards currently show only a step fraction counter. This proposal adds contextual status indicators so users can scan the board and immediately spot tasks that need attention, are actively being worked, have failed, or have gone stale.

Interaction Behaviors

Approval cards demand attention

The violet border glow persists (not just on hover) so users scanning the board immediately spot what needs their action. Tooltip shows the step name awaiting approval.

Spinner matches workflow step UI

The spinning loader icon and blue color scheme are identical to the existing in_progress step badge in the task detail dialog. Users who've seen steps running will immediately recognize this pattern on the card.

Stale after 2 hours

An in_progress step transitions to "stale" when started_at is more than 2 hours ago. Shows elapsed time (e.g. "4h", "1d"). The amber left-border accent and clock icon signal something needs investigation.

Failed state is persistent

The red left-border accent and "failed" badge stay visible until the step is retried or the workflow is reset. Clicking opens the task with the failed step scrolled into view.

Priority cascade

When multiple states apply, show the highest-priority: Failed > Approval > Stale > Active > Idle. One badge per card to avoid clutter.

Step counter stays visible

The overall progress fraction (e.g. "2/4") is always shown alongside the status badge — either inside the badge for complete/idle, or as a separate muted counter for active/approval/failed/stale states.

Implementation Approach
Recommended

Denormalized columns on board_tasks

Add new columns maintained by database triggers:

workflow_step_in_progress (int)
workflow_step_failed (int)
workflow_step_awaiting_approval (int)
workflow_step_started_at (timestamptz) — earliest active step start, for stale detection
workflow_active_step_title (text) — title of the current active/failed/approval step

+ Zero additional queries at board load
+ Consistent with existing workflow_step_completed pattern
+ Works with Realtime subscriptions
+ Stale computed client-side from workflow_step_started_at
- Requires a migration + trigger logic
Alternative

Subquery join at fetch time

Modify the board page query and MCP getBoard to include a grouped count subquery on task_workflow_steps per task.

+ No schema changes needed
+ Always accurate (no trigger lag)
- Increases query complexity and payload size
- Breaks the existing denormalized count pattern
- Doesn't update via Realtime without separate subscription
