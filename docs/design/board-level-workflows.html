Design Specification

Board-Level Workflows

Workflows are a property of how your team works, not a property of any single agent. Move templates to the board. Let Claude Code orchestrate. Kill the agent taxonomy.

01

Executive Summary

The current workflow system is technically complete but practically unusable. Templates are buried inside agent configuration dialogs. Users must learn an orchestrator/worker taxonomy before they can use workflows. This redesign moves workflows to where work already happens — the board.

Before & After

ConceptCurrentProposed
Template ownership On agent profiles On the idea's board
Who orchestrates Designated "orchestrator" agent Claude Code (user's session)
Agent types orchestrator / worker None — all agents equal
How templates trigger Orchestrator creates steps Label auto-rules or manual pick
Where users find templates Inside agent config dialogs "Workflows" tab on board page
Agent-to-step matching Orchestrator decides Role-based auto-match from pool
Key insight: Claude Code IS the orchestrator. It reads workflow steps via MCP, assumes each agent's persona, executes the work, and saves output. The user's Claude Code subscription pays for compute. No server-side AI calls. No special agent type needed.

Label Auto-Rules

Zero-touch automation. A database trigger auto-applies workflows when a task gets a matching label. By the time Claude Code reads the task, the workflow is already there.

🏷
Label Applied
Task gets the feature label
→
⚡
Trigger Fires
Postgres trigger on board_task_labels
→
⚙
Run Created
workflow_run + steps auto-populated
→
🔍
Claude Reads
get_board returns task with pending workflow
→
▶
Executes
Claude works through each step automatically
Three things work together: (1) Auto-rule in the database maps label → template. (2) Postgres trigger on board_task_labels INSERT creates the workflow run. (3) MCP tools return workflow data inline with the task — Claude Code sees "status": "pending" and naturally knows there's a workflow to execute. No special instructions required.

10

Migration Path

How to get from the current implementation to this proposal. Forward-only migrations.

Create new tables. Add workflow_templates, workflow_auto_rules, workflow_runs with RLS policies using is_idea_team_member().
Migrate existing templates. Extract workflow_templates jsonb from bot_profiles → insert into new workflow_templates table, scoped to each idea where the agent is pooled.
Add columns to task_workflow_steps. Add run_id, step_order, agent_role, output, requires_approval. All nullable for backwards compatibility with existing steps.
Create auto-rule trigger. Postgres trigger on board_task_labels INSERT that checks workflow_auto_rules and auto-creates workflow_run + steps.
Remove agent type system. Drop agent_type, workflow_templates (jsonb), deliverables from bot_profiles. Drop is_orchestrator from idea_agents.
Update MCP tools. Add 5 new template tools. Remove set_orchestration_agent. Enrich get_board / get_task responses with workflow_run data inline.
Update UI. Remove agent type picker and WorkflowTemplateEditor from agent dialogs. Add "Workflows" tab to board page. Add template picker to task creation. Add template application preview dialog.
