Project Type Presets

One-click board setup — labels, workflow templates, and auto-rules seeded from a project type selection

RFC Status: Draft — companion to Workflow Orchestration v2. Depends on workflow templates and auto-rules being implemented first.

1. Problem

Workflow Orchestration v2 introduces templates, auto-rules, and database triggers that make workflows powerful. But there's a cold-start problem:

To get a working workflow pipeline, a user must:

  1. Create board labels (feature, bug, etc.)
  2. Create workflow templates with agent-assigned steps
  3. Create auto-rules mapping labels to templates
  4. Allocate agents to the idea's agent pool

That's 4 setup steps before a single workflow can auto-fire. Most users won't do this — they'll just use the board as a plain kanban and never discover workflows.

The AI Enhance Gap

The existing AI enhance button generates board tasks from an idea description, but it only creates tasks. It doesn't set up the infrastructure (labels, templates, rules) that would make those tasks flow through a structured workflow. The board starts populated but inert.

2. Solution

Add a project type selector to the AI enhance flow. When the user selects a project type, the system seeds the board with a complete, pre-wired setup:

What gets seededWithout project typeWith project type
Tasks AI-generated from description AI-generated from description (same)
Labels None Pre-configured set (e.g. feature, bug, chore)
Workflow templates None 1–2 templates with agent-assigned steps
Auto-rules None Label → template mappings with auto_run
Task labels None AI assigns labels to generated tasks based on type

Result: The user picks a project type, clicks AI enhance, and gets a fully wired board. The first task they (or Claude Code) picks up with a feature label already has a pending workflow run attached via the database trigger. Zero manual setup.

5. Data Model

Presets: Static Config, Not Database

Preset definitions are static JSON config in the codebase (e.g. src/lib/project-presets.ts), not database records. This keeps them simple, versionable, and avoids a migration every time a preset changes.

Why not store presets in the database?

  • Presets are the same for every user — no per-user customisation needed
  • Changes to presets should be code-reviewed, not admin-panel edits
  • Keeps the database focused on user-created data (their templates, their rules)
  • The preset is a one-time seed operation — after creation, the user's labels/templates/rules are independent copies they can modify

Database Changes

One column addition to the ideas table to track which preset was used (for analytics and to avoid re-seeding):

ideas (modified)
...existing cols (unchanged)
+ project_type text? (web_app, mobile_app, api, content, design, custom)

This column is nullable (existing ideas have no type) and purely informational. The seeded labels, templates, and auto-rules are independent copies — changing the preset definition later doesn't affect already-created boards.

Config Structure

src/lib/project-presets.ts
interface ProjectPreset {
  type: string
  name: string
  description: string
  icon: string
  labels: { name: string, color: string }[]
  templates: {
    name: string
    description: string
    steps: { title, agent_role, requires_approval }[]
  }[]
  autoRules: {
    labelName: string // matches a label above
    templateName: string // matches a template above
    autoRun: boolean
  }[]
}

6. Integration Points

Seeding Flow

When the user clicks "Enhance with AI & Create Board" with a project type selected:

1. Initialize board columns
Same as current flow — uses initializeBoardColumns() with user's default or system defaults.
2. Seed labels
Insert board_labels rows from the preset config. Skip any that already exist (by name) to avoid duplicates on re-runs.
3. Seed workflow templates
Insert workflow_templates rows with steps JSONB from the preset config. Templates are owned by the idea.
4. Seed auto-rules
Insert workflow_auto_rules rows mapping preset labels to preset templates. Set auto_run per the config.
5. AI generates tasks with labels
The AI prompt includes the available labels, so generated tasks come back pre-labelled. The database trigger fires on label assignment, auto-creating workflow runs for matching tasks.
6. Board is ready
User sees a fully populated board: tasks with labels, workflow runs already attached to labelled tasks, templates and auto-rules configured. Ready for Claude Code to start picking up work.

AI Prompt Enhancement

The project type also improves AI task generation. The prompt to Claude for generating board tasks gains additional context:

Added to the existing task generation prompt:

This is a web_app project. Available labels: feature, bug, chore, spike, ux.
Assign one label to each generated task. Use "feature" for new functionality,
"bug" for fixes, "chore" for setup/config, "spike" for research, "ux" for design tasks.
Tasks labelled "feature" will auto-trigger a Dev Pipeline workflow.

MCP Tool Integration

The project type should also be available via MCP for Claude Code-driven board setup:

ToolChange
create_idea Add optional project_type parameter. When provided, seeds the board with the preset after idea creation.
get_task Already returns workflow_run inline (from Workflow v2). No additional changes needed.

Edge Cases

  • Existing idea, no board yet: Project type selector shown in AI Generate dialog. Seeding happens before task generation.
  • Existing idea with board: Project type selector not shown. User already has labels/templates they've configured. Don't overwrite.
  • User picks Custom then wants presets later: They can manually create templates from the Workflows tab. No "apply preset retroactively" button needed for MVP.
  • Agent roles reference non-existent agents: Template steps use generic role names ("BA Agent", "Dev Agent"). Claude Code maps these to actual agents in the idea's agent pool, or just adopts the role name as a persona if no matching agent exists.

This feature depends on Workflow Orchestration v2 Phases 1–3 being complete (templates, lifecycle tools, auto-rules + triggers).

1 day
Phase 2 — UI Components
ProjectTypeSelector component (grid of type cards). Preset preview panel. Integration into onboarding Step 3 and AI Generate dialog.
1–2 days
Phase 3 — AI Prompt Enhancement
Modify generateBoardTasks to include project type context and available labels in the AI prompt. AI generates tasks with label assignments. Labels applied on task creation trigger auto-rule workflow runs.

Phase 4 — MCP + Testing
Add project_type parameter to create_idea MCP tool. Unit tests for seeding logic. E2E tests for onboarding flow with preset selection.
1 day

Total estimate: 3.5–5 days. Requires Workflow Orchestration v2 Phases 1–3 as a prerequisite.
