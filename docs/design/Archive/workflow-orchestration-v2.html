Design Document & Interactive Mockups — Claude Code as the orchestrator, MCP tools as the interface

RFC Status: Draft — for stakeholder review and alignment before implementation.

1. Executive Summary

Problem

The current workflow system (v1) is technically complete but practically unusable. A UX review identified 24 issues. The core problems:

  • Agent-owned workflows — templates live inside agent configuration, invisible to most users
  • Discussion-gated entry — the only way to trigger a workflow is converting a discussion, adding an opaque extra step
  • Requires orchestrator setup — users must designate an "orchestration agent" on the idea before workflows work
  • A real user tried the feature and couldn't figure out how it worked

Solution

Move workflows to where work already happens — the board. Templates are owned by the board (idea), not agents. Claude Code is the orchestrator — it reads workflow steps via MCP, assumes agent personas, executes each step, and saves output. The user's Claude Code subscription pays for AI. No server-side AI calls needed.

3. Journey 2 — Claude Code Orchestration

This is the primary workflow experience. Claude Code IS the orchestrator — it reads workflow steps via MCP, assumes each agent's persona, executes the work, and saves output. The user's Claude Code subscription pays for AI. No server-side API calls needed.

Key insight: Each workflow step = Claude Code assuming an agent persona and executing via MCP tools. MCP tools are just state management (read steps, save output, approve/reject). Claude Code does all the thinking.

4. Journey 3 — Autonomous Pipeline

The most powerful pattern: a label auto-rule triggers a full workflow pipeline. A database trigger auto-applies the template when the label is assigned. When Claude Code next reads the task, the workflow run is already attached — Claude sees it and works through each step by assuming the appropriate agent persona. Only approval gates require human input.

Trigger: A task is created and given the feature label. A Postgres trigger fires, auto-applies "Dev Pipeline", and creates the workflow run. Claude Code picks up the task, sees the pending workflow, and begins executing steps.

Key Insight

Claude Code is a single AI that switches personas for each step. It reads the agent's system prompt from bot_profiles, adopts that persona's expertise, executes the step, and saves the output via MCP. The human reviews output at natural checkpoints rather than managing each step. A 6-step workflow only needs 2 human interactions — approving design and approving final QA.

Future: Web UI. A later phase may add a read-only web UI for viewing workflow status, step outputs, and approving gates from the browser. For now, all interaction happens through Claude Code + MCP.

5. Journey 4 — Backlog Pickup (End-to-End)

The most realistic scenario: you tell Claude Code to "pick up the next thing on the backlog". The task already has a feature label. When that label was applied, a database trigger automatically applied the Dev Pipeline template — so by the time Claude reads the task, the workflow run is already there. Claude sees it in the task data and executes it.

How auto-rules fire: A Postgres trigger on board_task_labels INSERT checks workflow_auto_rules for a matching label. If auto_run = true, it automatically creates a workflow_run and populates task_workflow_steps. The workflow is already attached to the task before Claude Code ever sees it — no special instructions needed.

What Made This Automatic

Three things work together — no special Claude Code configuration needed:

  1. Auto-rule in the database — The feature label is mapped to the Dev Pipeline template with auto_run = true
  2. Database trigger on label assignment — When the label was added to the task, a Postgres trigger on board_task_labels automatically created the workflow_run and populated the steps. The workflow was already attached to the task before Claude Code ever saw it.
  3. Enriched task data from MCP tools — get_board and get_task return workflow_run data inline with the task. Claude Code sees "status": "pending" and naturally knows there's a workflow to execute. No instructions required — the data speaks for itself.

Any Claude Code session, any user, would see the pending workflow and know what to do. The system is self-describing.

Auto-Apply Trigger

A Postgres trigger on board_task_labels INSERT powers the auto-rule system:

1
Label assigned to task
Row inserted into board_task_labels
2
Trigger checks auto-rules
Looks up workflow_auto_rules for a matching label_id + idea_id with auto_run = true
3
Workflow run created automatically
Inserts workflow_run (status pending) and copies template steps into task_workflow_steps. Increments usage_count.
4
Task data enriched
MCP tools (get_board, get_task) return workflow_run data inline with the task — Claude Code sees the pending workflow without needing to query separately.

Guard: the trigger skips if the task already has an active workflow run (one run per task). If auto_run = false, the rule is available for manual application but doesn't auto-fire.

RLS Policy

All new tables use the existing is_idea_team_member() function for RLS, consistent with other board tables. Read access extends to public idea viewers (for workflow_templates and workflow_runs).

7. MCP Tool Specifications

Seven MCP tools for the full workflow lifecycle. These are state management tools — they read/write workflow data. Claude Code does all the thinking and orchestration.

ToolParametersDescription
get_workflow_templates idea_id List all workflow templates for a board. Returns template name, step count, usage stats.
create_workflow_template idea_id, name, description?, steps[] Create a new reusable template. Steps include title, description, agent_role, requires_approval.
apply_workflow_template task_id, template_id Copy template steps onto a task, creating a workflow_run. Returns run_id and step details.
get_workflow_step step_id Read step details including agent role, system prompt, description, and prior step outputs for context.
complete_workflow_step step_id, output Mark a step as completed, saving its output. Claude Code calls this after executing each step.
approve_workflow_step step_id, feedback? Approve a gate step. Advances the workflow run to the next step.
reject_workflow_step step_id, feedback, reset_to_step? Reject a gate step with mandatory feedback. Optionally reset to an earlier step number for re-work.

Error Cases

  • apply_workflow_template on a task that already has an active run → error (one active run per task)
  • complete_workflow_step on an already-completed step → no-op with current status
  • approve_workflow_step on a non-gate step → error
  • reject_workflow_step with reset_to_step greater than current step → error
  • All tools require idea team membership (enforced via RLS + MCP context)

8. Claude Code as Orchestrator

Claude Code is the orchestrator. There is no server-side AI execution — no API routes calling Claude, no resolveAiProvider(), no pg_net webhooks for resumption. The MCP tools are pure state management. Claude Code does the thinking.

How It Works

1
Read Workflow Steps
Claude Code calls get_workflow_step to read the current step's details, agent role, and prior step outputs
2
Assume Agent Persona
Load the agent's system prompt from bot_profiles and adopt that persona's expertise and perspective
3
Execute the Work
Claude Code does the actual work — analysis, design, coding, testing — using the user's subscription for AI
4
Save Output & Advance
Call complete_workflow_step with output. Move to next step.
5
Pause at Approval Gates
If next step requires_approval: pause and notify the user. Resume when user approves via Claude Code.

Why This Architecture

Simplicity. MCP tools are just database reads/writes for state management. No server-side AI calls, no webhook resumption, no API routes, no resolveAiProvider() complexity. Claude Code handles all the intelligence — and the user's existing subscription pays for it.

Self-Describing Data (No CLAUDE.md Required)

The system is designed so that Claude Code doesn't need special instructions to drive workflows. The data itself tells Claude what to do:

  • Workflow run attached to task data — get_board and get_task return workflow_run inline. When Claude sees "status": "pending" with steps and agent roles, it naturally understands there's a structured workflow to follow.
  • Step details include agent prompts — get_workflow_step returns the agent's system prompt, step description, and prior step outputs. Claude has everything it needs to assume the persona and do the work.
  • Approval gates are explicit — Steps with requires_approval: true clearly signal where Claude should pause and ask the user.
  • Auto-rules fire at the database level — A trigger on board_task_labels creates workflow runs automatically. By the time Claude reads the task, the workflow is already there.

Any Claude Code session will see a pending workflow and know to execute it — the same way it would notice a task description, labels, or comments. The workflow data is just part of the task.

Safety & Limits

  • Cost: User's Claude Code subscription — no platform AI costs
  • Idempotency: Completing an already-completed step is a no-op
  • Error handling: If Claude Code errors mid-workflow, the step state is preserved in the DB. Resuming a new session picks up where it left off.
  • One run per task: Cannot apply a new template while a run is active

9. Implementation Roadmap

Five phases, each independently deployable and testable.

Phase 1 — Data Model + Template CRUD
Migration for 3 new tables + modifications. MCP tools for template create/read/update/delete. Basic template list UI on board page.
1–2 days
Phase 2 — MCP Workflow Lifecycle Tools
MCP tools for apply_workflow_template, get_workflow_step, complete_workflow_step, approve_workflow_step, reject_workflow_step. Full state management for workflow execution.
2–3 days
Phase 3 — Auto-Rules + Database Trigger
Label-to-template mapping CRUD. MCP tools for auto-rule management. Postgres trigger on board_task_labels to auto-create workflow runs. Enrich get_board and get_task responses with inline workflow_run data.
1–2 days
Phase 4 — Testing + Hardening
Unit tests for all MCP tools and trigger logic. E2E tests for template CRUD, workflow execution, and auto-rule trigger. Edge cases: concurrent runs, mid-workflow errors, session resumption.
1–2 days
Phase 5 — Web UI for Workflow Status (Future)
Read-only web UI showing workflow status, step outputs, and progress. Browser-based approval for gate steps. This is a later phase — not needed for the core MCP-driven flow.
2–3 days

Total estimate: 5–9 days for Phases 1–4 (core functionality). Phase 5 (web UI) is a future enhancement. Each phase produces a working increment that can be reviewed and tested independently.
