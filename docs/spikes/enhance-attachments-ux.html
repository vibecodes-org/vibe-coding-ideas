UX Design · Feature Development step 2 · revised pass (Requirements v2)

Enhance-with-AI reads idea attachments
the attachment-usage indicator

The backend now feeds text attachments (md/html and text-layer PDFs) into the three idea-enhance AI calls. This doc designs the single new UI element: a passive, honest “receipt” telling the author which attached files were actually read — and a warning when none were. Builds on the Requirements v2 deliverable (card d8841bbb): PDF text extraction is now in scope for v1, and the live UI path is the streaming route src/app/api/ai/enhance/route.ts.

0Context — what exists, what changes

The enhance flow lives in src/components/ideas/enhance-dialog-shell.tsx: configure → (optional) questions → result → apply/refine. The result streams from the streaming route src/app/api/ai/enhance/route.ts (prompt build at lines 60–106, response headers at 150–152) as plain text; clarifying questions call the generateClarifyingQuestions server action directly (src/actions/ai.ts:249). Attachments render in idea-attachments-section.tsx (paperclip header, per-file rows with type-coloured icons).

  • From Requirements v2: per-file 8k / total 24k char caps, newest-first, size > 1 MB pre-skipped. Readable types: md/html and PDFs with a text layer (server-side text extraction, no OCR) — extracted PDFs behave like any other text file, including the 8k truncation cap. Scanned / image-only PDFs (no extractable text) fail extraction and are omitted as read_error; images and SVG are the only remaining unsupported_type. Byte-for-byte parity when an idea has no attachments.
  • UI scope is one element: a passive usage indicator. No file pickers, no toggles, no new phases.
  • Out of scope surfaces: the create-flow dialog (create-enhance-dialog.tsx) — no saved idea yet, so no attachments; it never renders the indicator. Task/discussion enhance are separate cards.

1The three open questions — decided

Q1 · Indicator timing: before enhance, after the result, or both?
After — actual usage from server metadata, visible from the first streamed token.

Why: a pre-enhance “will include 2 attachments” either duplicates the server’s skip logic client-side (size cap, type check, budget ordering — four ways to drift and lie) or is a guess. The server knows the truth the moment it builds the prompt (route.ts lines 60–106), which is before streaming begins — so the metadata rides the X-Attachment-Usage response header (route.ts 150–152) and the client can render it as soon as the fetch resolves, before the first chunk. Users get the pre-enhance reassurance (“it’s reading my files”) with post-enhance accuracy, from one source of truth. No second mechanism needed.

Q2 · All text attachments skipped: warn or stay silent?
Warn — an inline amber note in the result phase (not a toast, not silent).

Why: the user attached files and paid credits; silence lets them believe the files informed the result. That’s the one dishonest outcome this feature must not allow. But it’s a degraded-input note, not a failure — the enhancement itself succeeded — so it reuses the existing amber inline-banner pattern (the “output was truncated” banner already in the shell) rather than an interruptive toast. Ideas with zero attachments show nothing at all: unchanged behaviour, no noise.

Q3 · Does the clarifying-questions step need its own indicator?
No — one indicator per flow, shown in the result phase.

Why: the questions step is transient input-gathering inside the same enhance run; a banner there plus one at the result would announce the same background behaviour twice per credit-spend. The result phase is the receipt moment — where users judge what they paid for. generateClarifyingQuestions (src/actions/ai.ts:249) should still return the usage metadata (it’s the same server-side change, near-free), so if “how did the AI know that?” confusion appears, a one-line muted caption under the questions is a fast-follow — data already plumbed, no signature change.

2Anatomy & exact copy

One new component, AttachmentUsageLine, rendered as the first block of the result phase (above the existing output-truncated banner and heading chips). It is a quiet bordered row in bg-muted/20 — receipt, not celebration. Two visual modes: neutral receipt (≥1 file read) and amber warning (files attached, none read). Never colour-only: paperclip / alert-triangle icon + full sentences.

There is no PDF-specific copy anywhere; unsupported_type is images/SVG only.   • Filenames render in the UI mono style (matching attachment rows), truncate with CSS ellipsis + title attribute; full name stays in the DOM for screen readers.

4Metadata contract for Implementation

One shape, two transports. Compute it server-side at prompt-build time (the server already knows exactly what it included). Per Requirements v2, the live UI path is the streaming route src/app/api/ai/enhance/route.ts — attachment injection happens where the prompt is built (route.ts 60–106) and the header is emitted with the response headers (route.ts 150–152), which flush before the stream body.

// src/lib/ai-helpers.ts (or wherever attachment reading lands)
export interface EnhanceAttachmentUsage {
  /** Files whose content was placed in the prompt (order = prompt order, newest-first).
      Now may include PDFs — text-layer PDFs are extracted server-side (no OCR). */
  used: Array<{ id: string; name: string; truncated: boolean }>;
  /** Files that were skipped entirely */
  omitted: Array<{
    id: string;
    name: string;
    // unsupported_type = images/SVG ONLY.
    // read_error = fetch/extraction failure, INCLUDING scanned / no-text-layer PDFs.
    reason: "too_large" | "unsupported_type" | "over_budget" | "read_error";
  }>;
}
  • ids included (small delta from the Requirements sketch of {name, truncated}): they cost nothing and are required by the deferred excludeAttachmentIds opt-out — the UI can pass ids back without a second lookup.
  • Revised semantics (Requirements v2): unsupported_type now means images/SVG only. Text-layer PDFs go to used (with truncated: true when cut at 8k); scanned / image-only PDFs go to omitted with read_error — expected, not exceptional. The UI still needs omitted entries for the honesty line in mock C and the all-omitted case; “all omitted” is computed as used.length === 0 && omitted.length > 0.
  • Transport 1 — streaming route src/app/api/ai/enhance/route.ts: response header X-Attachment-Usage: encodeURIComponent(JSON.stringify(usage)) (headers are latin-1; filenames can be unicode, so URI-encode). Headers flush before the body → indicator renders at stream start. Omit the header when the idea has zero attachments. Keep the payload clamped — filenames only, no content — so the header stays well under proxy size limits.
  • Transport 2 — server action generateClarifyingQuestions (src/actions/ai.ts:249): return { questions, attachmentUsage? }. Optional field — additive, no signature break. Returned but unused by UI in v1 (Decision 3), plumbed for the fast-follow caption.
  • Client parsing is defensive: missing header, malformed JSON, or unknown reason → render nothing / fall back to “couldn’t be read”. A metadata bug must never break the enhance result.
  • Refine / regenerate / continue runs re-send attachments per Requirements, and each response carries a fresh header — the receipt always reflects the run currently on screen, replacing the previous one.

Accessibility (WCAG 2.1 AA)

  • Not colour-only: receipt = paperclip icon + sentence; omitted row = info-circle icon + “Not read:” prefix; warning = triangle icon + written title. Every state is fully legible in grayscale.
  • Contrast: receipt body uses text-muted-foreground (zinc-400 ≈ 7.9:1 on zinc-950); filenames in zinc-200. Amber banner text amber-400 on amber-500/10 ≈ 10:1. All ≥ 4.5:1.
  • Announcement: role="status" on the all-omitted banner only — announced once when the result arrives. The neutral receipt is static text discovered in reading order (announcing routine success would be noise). No mid-stream re-renders, so no live-region churn.
  • Keyboard: no interactive elements inside the indicator in v1 — nothing to trap or skip. All information is in-text, not tooltip-gated.
  • Long filenames: break-words wrapping inside the receipt (screen readers get the full name); the reason suffix always remains visible — never ellipsized away.

6Deferred (noted for later, not v1)

  • Opt-out toggle: lives in the configure phase, as a checkbox row directly below “Ask clarifying questions first (recommended)” — e.g. ☑ Include attached files (2) — with per-file exclusion via a small popover feeding excludeAttachmentIds. The metadata ids in §4 make this a UI-only change.
  • Questions-phase caption: one muted line under the questions list — “Questions informed by 2 attached files.” — using the already-returned attachmentUsage. Ship only if user confusion is observed.
  • PDF OCR: text-layer extraction shipped in v1 (Requirements v2); scanned / image-only PDFs currently fall to read_error → “couldn’t be read”, which is expected. If OCR lands later, those files simply move into the “Considered” line — no UI redesign needed.

7Design-review checklist

  • Copy never claims a skipped or truncated file was fully read; all-omitted banner states what was used.
  • Text-layer PDFs appear in the “Considered” line like any md/html file; scanned PDFs show “couldn’t be read” (read_error, neutral zinc). No copy anywhere claims PDFs are unsupported.
  • Indicator appears only when the idea has ≥1 attachment; create-flow dialog and no-attachment ideas are pixel-identical to today.
  • Receipt visible during streaming (header parsed before first chunk) and correct after Refine / Regenerate / Continue re-runs.
  • All-omitted case uses the amber banner with triangle icon + title and role="status"; partial omission stays neutral zinc.
  • No colour-only signalling; contrast ≥ 4.5:1 in dark and light themes; no tooltip-gated information.
  • 60+ character filenames wrap without pushing the “(shortened…)” / reason text out of view; dialog never scrolls horizontally.
  • Malformed or missing metadata suppresses the indicator without affecting the enhance result (no crash, no toast).
  • Server action return stays additive (attachmentUsage? optional) and includes attachment ids for the deferred opt-out.
