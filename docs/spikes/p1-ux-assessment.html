UX Assessment — No UI Surface

Step 2 (UX Design) · P1: Route in-app workflow-matching to Haiku · src/lib/workflow-matching.ts

Designer's verdict: no UI deliverable. This is a backend model-constant swap (WORKFLOW_MATCHING_MODEL: claude-sonnet-4-6 → claude-haiku-4-5-20251001). The workflow-matching adjudication runs entirely server-side. It has no view, no control, no copy. The correct UX outcome is that this change is invisible to users. Do not invent UI where none is needed.

1. There is no user-facing surface

The adjudicator (adjudicateWorkflowMatch()) is a data/logic-layer function called during auto-rule application. Its output feeds the existing workflow-suggestion experience, but the model choice behind that output is never surfaced — no model name, latency badge, or "powered by" string is shown to the user. Swapping the constant changes which API the server calls and nothing else.

2. UX non-regression requirement (the real deliverable)

Because the change is invisible by design, the sign-off is a non-regression assertion: the workflow-suggestion panel must look and behave identically before and after. Same thresholds (WORKFLOW_AI_AUTOAPPLY_THRESHOLD = 0.85), same in-flight timing, same keep/replace/remove controls, same mismatch-reason text. Each state below must render pixel- and copy-identical to today.

StateWhat the user seesMust remain
in-flight / checking "Checking…" affordance while adjudication resolves Same spinner/label, same perceived latency band — no new "slow" feeling
suggested Suggestion panel with mismatch reason + keep / replace / remove controls Same layout, same reason copy, same controls (not colour-only)
auto-applied High-confidence match (≥ 0.85) applied silently, no interruption Same silent behaviour — no new toast, banner, or confirmation
dismissed User keeps existing workflow; suggestion clears Same dismissal path and persistence

3. The one indirect risk: silent backend degradation

The adjudicator always resolves — if the model is unavailable, errors, times out, or returns invalid output, it falls back to the deterministic keyword heuristic (source: 'heuristic'). If the Haiku model id were silently rejected, every suspect case would quietly drop to the heuristic: users would get worse or no AI suggestions, with no visible error state. This is a telemetry concern, not a UI one — there is intentionally no user-facing "AI unavailable" surface here.
  • Not a UI state: do not add an error banner — degradation is meant to be graceful and invisible.
  • Observable via telemetry only: watch the source: 'ai' vs source: 'heuristic' ratio and adjudication error/timeout rates after rollout. A spike in heuristic is the real "error state."
  • Acceptance for this step: suggestion quality (which suggestions appear / auto-apply) is indistinguishable to users; only cost and latency change behind the scenes.
Sign-off: no new UI. Protect the existing workflow-suggestion UX (four states above) as a non-regression checklist; treat a rise in heuristic-fallback rate as the only "error" to watch — via telemetry, not the interface. Feeds Step 3 (Design Review / human approval).
