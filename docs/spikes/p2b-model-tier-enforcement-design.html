Feature Development · Step 2 — UX Design

P2b — Enforce Model Tiers

P2 shipped an advisory per-step model tier. P2b makes it real: a MANDATORY MODEL directive naming the exact Task-tool model parameter, resolved from a platform default plus a per-user override map (users.model_tier_map). This document designs the two user-facing surfaces — the Model tier mapping settings dialog on the own-profile page, and the “runs on” copy refresh across all five ModelTierSelect mounts — plus the directive's before/after story. Builds on the P2b Requirements deliverable (FR-10/13/14/16/17, AC-14); requirements are cited, not repeated.

00The three open questions — decided

Q1 · Placement
Own-profile settings row, next to “AI API Key” — same button-opens-dialog pattern.
The profile settings “stack” is actually a row of outline buttons that each open a dialog (profile/[id]/page.tsx:202-209). A dedicated page section would over-weight a set-once preference and break Jakob's-Law consistency with its five siblings. It sits beside AI API Key because both are personal AI-account settings. Shown in context in §01.
Q2 · Unset display
Inline in the closed trigger: “Fable (default)” — no separate caption.
Recognition over recall: the default is visible exactly where the choice is made, mirroring P2's “Auto (recommended)” inline-suffix precedent. A separate caption doubles the vertical rhythm ×3 rows and detaches the fact from the control on 375px. The “(default)” suffix also encodes set-vs-unset in text, never colour alone.
Q3 · Resolved model on steps
Tier-only on step selects & badges — agree with Requirements.
The tier is the shared, stored fact; the resolution is per-user. Printing “Frontier · Fable” on shared board/template surfaces would show different truths to different collaborators. The resolved model appears only on per-user surfaces: the mapping dialog (§02) and the tier-option glosses (§04), which read “Runs on Fable…” using the viewer's map. Revisit for P2c verification.

One deviation from Requirements: the sample gloss “Frontier — runs on Fable · your best model” is adjusted to “Runs on Fable · decisions & design” — keeping the “runs on <model>” head but retaining P2's when-to-use tail, which is the information users need at choice time (§04).

02The “Model tier mapping” dialog

Three labelled selects — one per tier, in the established cost-gradient order (Frontier → Standard → Cheap) — each offering the four models. Unset renders inline as “<default> (default)” (Q2). A footer explains the fallback substitution (FR-10) so the settings surface and the directive tell one story. Footer actions follow the api-key-settings pattern: Cancel / Save right-aligned, with Reset to defaults on the left as a staged change — it clears all three selects locally and Save commits, so there is exactly one write path and Cancel always undoes.

05The directive — before / after (the behavioural guarantee)

What the orchestrator actually receives when it claims a step with model_tier: "standard" for a user whose map is empty (standard → Sonnet, fallback → Opus). This is the invisible core of P2b; it's shown here so Design Review can verify the user-visible promise the settings dialog makes (“steps run on the tier's mapped model”) matches the machine instruction word for word.

Before — P2, appended at the end of the instruction (workflows.ts:57)
…step context, deliverables, prior outputs…

RECOMMENDED MODEL TIER for this step: `standard` — spawn this step's subagent on a `standard`-tier model if available. Advisory only; does not change any other instruction.

Two failure modes observed in P2: (1) “if available / advisory only” invites the orchestrator to skip it; (2) “standard-tier model” names no concrete parameter value, so even a willing orchestrator must guess.

After — P2b, FR-10 verbatim, placed FIRST in the instruction
MANDATORY MODEL: spawn this step's subagent with the Task tool parameter model: "sonnet". If "sonnet" is unavailable on this plan/session, use model: "opus" and state the substitution in your step output. Do not run this step inline and do not inherit your session model.

…step context, deliverables, prior outputs…
  • First position — primacy: the model choice must be made before the orchestrator starts planning the spawn, not remembered after reading the task.
  • Exact parameter value — model: "sonnet" is the literal Task-tool argument; nothing left to interpret. Resolution: platform default (frontier→fable, standard→sonnet, cheap→haiku) overridden by the user's map from §02.
  • Fallback substitution is visible to the user — “state the substitution in your step output” means a substituted run says so in the step's recorded output, which the user reads in the step-detail dialog. No silent downgrades/upgrades. Chain: haiku→sonnet, sonnet→opus, fable→opus, opus→fable.
  • Auto is untouched — model_tier = null emits no clause; the instruction is byte-for-byte identical to today (FR-10). The tier control's Auto option therefore keeps its P2 meaning exactly.
  • Ceiling, not hard enforcement — per Requirements scope, the directive is the strongest instruction we can issue without deploy-side verification; P2c owns verification.

06Mobile — 375px

The dialog is already a single column, so 375px is a spacing exercise, not a re-layout: selects and buttons grow to ≥44px (min-h-11 under pointer: coarse), the footer wraps to two rows — Reset on its own line first, then Cancel/Save full-width in a 2-col grid with Save in the thumb-nearest bottom-right position — and the fallback helper wraps to three lines without truncation.

Visible labelsEach select has a permanent <Label htmlFor> (“Frontier — decisions & design”, …) — never placeholder-as-label. The “(default)” suffix is real trigger text, not a placeholder, so it is announced.
Programmatic relationshipsAll three selects carry aria-describedby → the shared fallback helper (#fallback-help); the disabled Save carries aria-describedby → its adjacent explanation (#save-why). Dialog uses shadcn's DialogTitle/DialogDescription wiring.

Target size36px triggers on fine pointers; ≥44px triggers, options, and footer buttons under pointer: coarse. Primary action (Save) sits bottom-right adjacent to the fields it commits (Fitts).
No disabled mysteryConstraint honoured: the only disabled control (Save when unchanged) has an always-visible adjacent explanation — “Save enables when you change a tier.”
MotionSkeleton shimmer respects prefers-reduced-motion: reduce → static muted-30 blocks.

Why a dialog and not inline? The no-modals-for-inlineable-content constraint is weighed against Jakob's Law here: every sibling setting on this page (API key, notifications, board columns, MCP keys) is a dialog behind a row button. A lone inline card would be the inconsistent — and therefore harder-to-find — choice; the dialog is a short, self-contained transaction (3 fields, one Save), which is the case dialogs are for.

08Design-review checklist

For the Design Review step — each item maps to a Requirements clause or a decision above.

  • Q1: “Model Tiers” button placement in the own-profile settings row (after AI API Key) + ProfileSettingsMenu item on mobile — approve or relocate. (§01)
  • Q2: inline “Fable (default)” trigger text + “Platform default” first option with separator — approve the unset model. (§02)
  • Q3: tier-only badges on shared surfaces; resolved model only in per-user glosses and the mapping dialog — approve the split. (§00, §04)
  • FR-13/14 / AC-14: three selects, staged Reset-then-Save, defaults shown for unset, success/error toasts, unchanged-Save explanation — confirm all six states in §03 are complete and correctly copy'd.
  • FR-16 exact strings: new helper (“Steps run on the tier's mapped model — change your mapping in Profile → Model Tiers.”), three “Runs on <model> · …” glosses, Auto untouched — sign off copy table in §04, including the noted deviation from the Requirements sample gloss.
  • Gloss resolution: viewer-resolved model in glosses with static-default fallback — accept, or downgrade to static defaults for P2b if plumbing cost bites. (§04, notes)
  • FR-10 story: directive-first placement, exact parameter value, and user-visible substitution line match the settings dialog's promise verbatim. (§05)
  • Mobile: 375px dialog — 44px targets, footer wrap order (Reset / Cancel+Save), no horizontal scroll. (§06)
  • A11y: label/describedby wiring, keyboard path, contrast table, non-colour state encoding. (§07)
  • Out of scope guard: nothing here implies hard enforcement, per-idea maps, or admin maps — flag if any mockup accidentally suggests otherwise.
