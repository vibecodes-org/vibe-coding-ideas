Feature Development · Step 2 — UX Design

P2c — Capture Executed Model & Tier Adherence

P2b made the model directive mandatory; P2c closes the loop by recording what the orchestrator says it ran (executed_model) and whether that honoured the step's tier (tier_honored). Almost all of P2c is backend — schema, capture params, resolveModelTier reuse, the workflow_tier_adherence view. This document designs the small user-visible surface that telemetry deserves, and decides the three open questions from the P2c Requirements deliverable. The governing principle throughout: this is self-reported data — every surface that shows adherence must say so, and unknown must never read as failure (NULL ≠ false is load-bearing in the schema; it is load-bearing in the UI too).

00The three open questions — decided

Q1 · v1 report surface
SQL view + documented queries only. The admin “Tier adherence” card is a fast follow, not this card.
Nick's intent — “so we'll be able to check it and run reports” — is fully satisfied by the view: he operates through Claude Code with direct Supabase access, and select * from workflow_tier_adherence is a one-liner there. Shipping an admin card with the capture would guarantee it launches into a permanent empty state (zero adherence rows exist until orchestrators adopt the new params). Requirements already marked the card a follow-up; agreeing keeps this card honest. Query shapes mocked in §05 so the view's columns are designed for the card that will consume them.
Q2 · Step-card indicator
Yes — exception-only on the row, full three-state display in the step-detail dialog.
Management by exception: the row (already carrying up to six chips) gains something only when tier_honored = false — an amber suffix inside the existing tier badge (§03). Honored and unknown rows look identical to today, matching ModelTierBadge's silence-is-default philosophy. The step-detail dialog, where there is room, shows all three states with unknown rendered as a dashed, muted “Model not reported” — different icon, border style, and words from dishonored, so unknown can never be mistaken for failure (§02).
Q3 · Mismatch visibility
Both: logger.warn and an auto step comment on dishonored completion.
logger.warn lands in Vercel logs nobody reads in-product; Nick's “check it” happens on the board. fail_step already auto-posts a failure comment — reusing that exact machinery (plain comment type, no enum migration) gives free in-product visibility: the comment increments the row's existing comment-count chip and appears in the dialog's timeline (§04). Cost is one insert on a rare path. Dishonored only — honored and unknown post nothing (no noise).

Scope deviation: none pulled in, one nudged out — Q1 keeps the admin card out (agreeing with Requirements, against a maximal reading of Nick's quote); Q2/Q3 add two deliberately small in-product surfaces that reuse existing components (ModelTierBadge, step comments) rather than new ones. A follow-up board card for the admin “Tier adherence” tab section should be created at implementation time.

01Surface map — where adherence appears, and the honesty rule

Four surfaces, in descending order of detail. Every one carries the self-reported disclosure — as visible caption where there is room (dialog, comment, SQL doc), as tooltip where there isn't (badges). No surface may use verification language (“verified”, “confirmed”, “enforced”); the vocabulary is reported / honored / not honored / not reported.

02Step-detail dialog — the three-state “Execution” line

A single new element in step-detail-dialog.tsx: an Execution line rendered in the content area, above Output. It appears only when the step has a tier (Auto steps are exempt — tier_honored is NULL by design and showing anything would imply a promise Auto never made) and the step is completed or failed. Pending / in-progress steps show nothing — there is no execution to report yet.

State 1 of 3 — honored (tier_honored = true)

Fallback counts as honored — by Requirements, fallback-honored folds into true. The line always names the executed model, so a Frontier step that fell back reads “Ran on Opus · tier honored”, and the tooltip's “or that tier's allowed fallback” explains why the check is still green. The UI never recomputes exact-vs-fallback post hoc: the completing user's map at completion time isn't stored, so any recomputation could lie. Display what was stored, explain the rule.

Why unknown can never read as failure — the two states differ on every channel at once: icon (help-circle vs warning-triangle), border (dashed zinc vs solid amber), fill (none vs amber tint), headline (“Model not reported” vs “Ran on Sonnet · tier not honored”), and sub-text (“Not a failure…” says it outright). Colour is the least load-bearing of the five. Every pre-P2c completion and every orchestrator that hasn't adopted the new params lands in unknown — which is most steps for months — so this state is calm, muted, and judgment-free by construction.

03Step row — exception-only amber suffix

The compact rows in task-workflow-section.tsx and workflows-tab.tsx gain nothing for honored or unknown steps — they render exactly as today. Only tier_honored = false changes the row: ModelTierBadge grows optional executedModel / tierHonored props and, when dishonored, appends an amber triangle + “ran on <Model>” inside the existing badge (one chip, not two). Clicking the row opens the dialog with the full §02 line, per the section's existing behaviour.

Row 1 vs row 3: row 1 is honored, row 3 is unknown — and they are deliberately indistinguishable at row level. Full disclosure sentence lives in the badge's tooltip (shown above) and in aria-label: Model tier Frontier — not honored: orchestrator reported Sonnet.

04Auto comment on dishonored completion (Q3)

Mirrors fail_step's auto-posted failure comment: when complete_step computes tier_honored = false, it inserts one step comment (existing comment type — no enum migration) attributed to the completing bot, then proceeds normally. The completion itself is never blocked or failed — this is telemetry, not enforcement. logger.warn fires alongside for ops-side aggregation.

08Mobile (375px) & accessibility — WCAG 2.1 AA

No admin card in v1, so mobile work is confined to the step row and dialog. Below sm, the dishonored badge suffix drops its text and keeps the amber triangle — the row's chips already compete for 375px, and the triangle is a shape, not a colour, so the reduction stays non-colour-coded; the full wording is one tap away in the dialog, which the whole row already opens. The Execution line in the dialog wraps naturally (it is a flex row of text) and its tap targets (info tooltip trigger) get min-h-11 under coarse pointers, matching ModelTierSelect's existing treatment.

09Design-review checklist

For the Design Review step — each item maps to a decision or a Requirements clause.

  • Q1: SQL view + documented queries as the v1 report; admin “Tier adherence” section deferred as a fast-follow card consuming Query 1 + Query 2 unchanged — approve the deferral and the two query shapes. (§00, §05)
  • Q2: exception-only row suffix + full three-state Execution line in the dialog — approve the split, and specifically that honored and unknown rows are identical at row level. (§02, §03)
  • Q3: auto comment on dishonored completion (plain comment type, completing-bot attribution, completion never blocked) alongside logger.warn — approve. (§04)
  • NULL ≠ false in UI: unknown differs from dishonored on icon, border, tint, and words; “Not a failure” stated verbatim — confirm no surface lets unknown read as failure. (§02)
  • Honesty framing: canonical disclosure constant appears on every surface (dialog tooltip/sub-text, badge tooltip, comment body, SQL doc header) and no string uses verification language — audit §07 copy table.
  • Fallback display: “Ran on Opus · tier honored” for a fallback-honored Frontier step, with the tooltip's “or that tier's allowed fallback” — confirm this reads as correct, not contradictory. (§02)
  • Amber not red: dishonored uses warning amber everywhere; red remains exclusive to the Failed status — confirm. (§03)
  • States: not-applicable gating (Auto steps, pre-completion statuses show nothing), no new fetch/loading/error surface, report empty-interpretation — confirm the table in §06 is complete.
  • Mobile: icon-only suffix below sm with full text in dialog + aria — accept, or require text at all widths. (§08)
  • Out-of-scope guard: nothing here implies hard enforcement, external cross-check, backfill, or blocking completions — flag if any mockup or string suggests otherwise.
