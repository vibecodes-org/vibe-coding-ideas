Feature Development · Step 2 — UX Design (Rework)

Per-step Model Tiering

A per-step Model tier hint on workflow templates — Auto / Frontier / Standard / Cheap — that the orchestrator uses to pick which Claude model runs each step's agent — plus a per-task override on any task's still-pending steps. Cheap steps on cheap models stretch Claude usage roughly 2.5×. This document shows the full UI: the shared control, all five mounts (four template editors + the per-task step edit dialog), the read-only badge, edit-propagation rules, mobile, and accessibility.

One component, mounted identically in five places — the four template editors plus the task-view step edit dialog (§04). The helper line is always visible, not a tooltip.

06Default role → tier seeding

When platform library templates ship tier values, defaults come from the step's role using the same regex families as getRoleBadgeClasses. Seeding is a one-time fill of the select — the user can change any value; it never overwrites an explicit choice. Scope (v1): platform library templates only. A “suggest tiers” affordance for user-authored templates was considered and cut in Design Review as over-reach — user templates simply default every step to Auto.

Role familyExample rolesSeeded tierWhy
Product / Analysis Product Owner, Business Analyst, PM — plus any step gated by review / approval / decision-making tier:Frontier Judgement-heavy; errors compound downstream
Design UX Designer — plus design and design-review steps tier:Frontier Design work is judgement-heavy; a weak design compounds through every build step after it
Build Full Stack Developer, Front End Developer, Back End Developer (build steps) tier:Standard Balanced quality/cost for most implementation work
QA / Mechanical QA Engineer (verify steps), lint/format passes, changelog & docs housekeeping tier:Cheap Deterministic, checkable work — biggest usage win
Unknown / custom Any role that matches no family Auto (no badge) Never guess — let the orchestrator decide
Usage story: a typical 4-step Feature Development run seeded this way sends two steps to Frontier (requirements, UX design), one to Standard, one to Cheap — the judgement-heavy steps keep the best model while build and verify still run markedly more workflow cycles per Claude usage window than everything on Frontier.

07What happens when you edit a tier

Two edit surfaces, two blast radii. The rules are the same ones role and description edits already follow — no new mental model to learn.

  • Edit on a template step flows through to workflows already attached to tasks — but only to steps still pending, on the same propagation path role and description edits use today (propagateTemplateEdits). In-progress, completed, failed, and awaiting-approval steps are never rewritten: what ran is what history shows.
  • Edit on a task's step (the §04 dialog) is a per-task override — it changes that one task's step and nothing else. The template is never modified from the task view.
  • Overrides vs. later template edits — the simple rule: template edits propagate to pending steps exactly as today, last write wins — consistent with how role edits behave. So if you override a tier on a still-pending step and someone later edits that step on the template, the template value replaces your override. Once the step starts, nothing rewrites it either way. No merge logic, no “locked” flags — one rule, stated plainly.

The four template editors are copies of the same compact-row layout today; the tier row is added identically in each. The step-detail dialog is the one stacked-layout mount (§04). The control itself lives once in src/components/ui/model-tier-select.tsx and adapts to both layouts.

Keyboard & focus order

  • Tab order per step row: move-up → move-down → Title → Role → Description → Approval switch → Deliverables → Model tier → Delete step. The tier select is appended, so existing users' tab rhythm is unchanged.
  • Open: Enter / Space / ↓ on the trigger. Navigate: ↑↓; type-ahead by first letter (c → Cheap). Commit: Enter. Dismiss: Esc restores prior value and returns focus to the trigger (Radix Select default behaviour — no custom key handling).
  • Board badge: not focusable in the step row (whole row is the button, as today); in the step-detail header its tooltip is reachable by keyboard focus per the existing Tooltip pattern.

States

  • Default / empty: new steps start at Auto; stored as null (no migration back-fill needed; absent key = Auto). No badge anywhere while Auto.
  • Error — invalid value from API: if a template's model_tier is not one of the four values (hand-edited JSON, older client), the select renders Auto and behaves normally. On save, the normalised value is written and a single toast explains: “Model tier reset to Auto — the saved value wasn't recognised.” Never a blocking validation error; the field is advisory.
  • Loading: template editors receive steps synchronously from already-loaded template JSON, so the select never shows an indeterminate state. In the workflows sidebar detail panel, the existing step-list skeleton simply gains a third 40px pill placeholder where the tier badge will sit.
  • Read-only contexts (platform preview, read-only board): static badge only — a disabled select would imply an action that isn't available and would need adjacent explanation; the badge doesn't.
  • Partial: mixed templates (some steps tiered, some Auto) are the expected steady state — no warning, no nudge to “complete” tiering.

  • Visible label: “Model tier” is a real <Label htmlFor> in the full control and a visible 10px inline label in compact rows (same as Title/Role/Deliverables today) — never placeholder-only.
  • Programmatic wiring: trigger has aria-describedby → the helper line; listbox options expose name + gloss as the accessible name/description; selection is announced via aria-selected.
  • Badge screen-reader text: aria-label="Model tier: Standard" on the badge, so “Standard” is never announced as a bare mystery word next to the role.
  • Keyboard: fully operable (Radix Select) — see §08. No pointer-only or hover-only behaviour: the badge is always rendered, the helper is always visible, tooltips are focus-triggered too.
  • No colour-only meaning: the tier is conveyed by the word itself (badge and select value). Tier badges are deliberately monochrome — colour is reserved for role/status where it already carries redundant-coded meaning.
  • Touch: 44px minimum target on coarse pointers for the trigger and every option row (§01, §05).
Light theme: all tokens are semantic (border-border, text-muted-foreground, bg-muted/20…), so the light variant is automatic: white background, zinc-200 borders, zinc-950 text, tier badge border-zinc-300 text-zinc-700 (≈ 6.4 : 1 on white). No bespoke light styles are needed beyond the badge border token.

10Design rationale

Four options is the ceiling for a scan-and-pick decision (Hick's law) — each option carries a one-line gloss so choosing takes seconds and requires no documentation. The control deliberately mirrors the existing Role select in the same step row (Jakob's law — internal consistency): same compact height, same inline-label pattern, same popover chrome, so it reads as a native part of a form users already know. Downstream, the tier: badge on step cards means the choice is visible where its consequence plays out (recognition over recall) — nobody has to reopen the template editor to remember what a step will cost — while rendering nothing for Auto keeps the default state exactly as quiet as today. Because the whole feature is advisory, the UI never blocks, warns, or demands completeness: it is a hint you can sprinkle onto expensive workflows and ignore everywhere else.
