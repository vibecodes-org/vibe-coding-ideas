UX Assessment — No UI Surface

Step 2 (UX Design) · P2: Per-step model tiering — add model_tier to workflow steps, surfaced in claim_next_step

Designer's verdict: no UI deliverable in this slice. This is a backend/data change — a nullable model_tier (frontier / standard / cheap) added to workflow steps, threaded template → step, and echoed back in the claim_next_step MCP response so the orchestrator spawns each persona subagent on the recommended model. Per Requirements NG6, human-authoring UI is explicitly out of scope: authoring is seed-SQL + API pass-through only. The correct UX outcome is that this change is invisible to end users. No view, control, or copy changes.

1. There is no user-facing surface

All existing rows default to model_tier = null, and null means "orchestrator decides" — identical behaviour to today. The tier is a hint carried on the data layer and consumed by the orchestrating agent; it never renders. The board, the template detail view, and the workflow-suggestion panel all draw the same pixels they draw now. Do not invent UI where none is needed.

2. Non-regression checklist (the real deliverable)

Because the change is invisible by design, sign-off is a non-regression assertion. For every step where model_tier is null — which is all existing data — these surfaces must render and behave pixel- and copy-identically to today:

SurfaceComponentMust remain identical
Workflow step cards task-workflow-section.tsx Same step rows, order, status pills, claim/complete controls — no new tier badge, column, or icon
Step detail step-detail-dialog.tsx Same fields, comments, agent identity, and rework controls — no model_tier field shown or editable
Active agent / model display step + claim UI (if any) Whatever "who's running this" affordance exists today is unchanged — no model-name or tier surfaced
Template detail view workflows-tab.tsx, template dialogs Same step list and metadata — no tier picker, no tier column in the template editor
Suggestion panel workflow-suggestion-panel.tsx Same match/keep/replace copy and controls — tier is never part of a suggestion
Apply / resync template apply_workflow_template, resync Copies model_tier through silently; a null-tier template produces null-tier steps as before

Acceptance: a user with only existing (null-tier) workflows sees zero difference anywhere in the product. The only observable effect of a non-null tier is which model the orchestrator picks — a backend concern, not a rendered one.

3. Fast-follow: per-step model-tier authoring Out of scope · proposed

Future slice, not this one. Requirements NG6 deliberately deferred human authoring. Capturing the design here so the opportunity isn't lost. None of the below ships in P2. Once tiers are proven useful via seed-SQL, a follow-up could let humans set a per-step tier in the template step editor — turning an invisible backend hint into an intentional, cost-aware authoring control.

Where it lives

A single compact <select> on each template step editor row (the natural authoring home — where step name, persona/agent, and human-check already live). Not a modal; inline with the row, consistent with the rest of the template editor. Recognition over recall: the resolved tier also appears as a subtle read-only badge on the workflow step card so authored intent is visible without opening the editor.

Design rules it must honour

  • Auto = null = default: the first option is "Auto", maps to model_tier = null, preserving today's "orchestrator decides" behaviour. Authoring is purely additive.
  • Four options only (Hick's law): Auto + three abstract tiers. Abstract tiers (not raw model ids) keep the choice durable as underlying models change.
  • Inline helper, always visible: "Auto lets the orchestrator choose; cheaper tiers save your Claude usage on simpler steps." Explains the value without a tooltip-only dependency.
  • Recognition over recall: resolved tier shows as a subtle badge on the step card; a null/Auto step shows no badge, so existing boards stay visually quiet.
  • Never colour alone (WCAG 2.1 AA): the badge always carries the tier word (tier: Standard), not just a hue; the select uses text labels, not swatches.
  • No hover-only actions, no modal: the control is a persistently visible inline select.
  • Graceful degradation: unknown/legacy tier values fall back to Auto rendering — never an error state.
Sign-off: no new UI in P2. Protect the six surfaces in the non-regression checklist (all identical for null-tier steps, i.e. all existing data). The per-step tier-authoring select + card badge is a deliberate, captured fast-follow — out of scope for this slice. Feeds Step 3 (Design Review / human approval).
