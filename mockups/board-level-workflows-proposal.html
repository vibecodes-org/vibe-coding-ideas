What does the user actually experience?

Same underlying engine — ordered steps, approval gates, context chaining, agent execution. The difference is what a user has to do to get started, and how it feels day-to-day.

Board-Level Templates

The user thinks: "I want this task to follow a process"
  • "I just want to use the bug fix workflow" — Open a task, click "Apply Workflow", pick a template. Done. No prerequisites, no agent configuration, no new concepts to learn.
  • "Make all bugs follow this process" — Set one auto-rule: label "bug" triggers the Bug Fix template. Every future bug task gets the workflow automatically. No agents involved.
  • "I swapped my QA agent for a better one" — The template doesn't change. Steps define roles, not specific agents. Swap freely without rebuilding anything.
  • "I don't have agents yet" — Workflows still work. Use them as a human checklist. When you add agents later, they slot into the existing roles.
  • "I want to get started quickly" — Import a template from the library with one click. No setup, no boilerplate — start using it immediately.
  • "The agent got it wrong" — Reject a step and cascade the reset back to any earlier point. Add comments explaining what to fix. The agent picks up rework instructions automatically on retry.

Orchestrator Model

The user thinks: "I need to set up an orchestrator to manage my workers"
  • "Wait, what's an orchestrator?" — Before using workflows, users must classify agents as "orchestrator" or "worker". This is an internal implementation concept leaking into UX.
  • "Why is the workflow on the agent, not the board?" — Templates live on agent profiles. Want the same workflow on a different idea? Recreate it, or ensure that agent is allocated. Templates aren't portable.
  • "Can't I just label it as a bug?" — No automatic triggers. The orchestrator agent must be running and explicitly pick up the task. Without it, nothing happens.
  • "I don't have agents yet" — Workflows require agents. No orchestrator agent = no way to create or apply workflows. The feature is invisible to agent-free users.
  • AI step generation — Generate workflow steps from discussion content. (This is additive and works equally well with both approaches.)
  • Review loops — Agents can fail steps for rework. (Both designs support this identically.)

The question isn't technical — it's conceptual. Is a workflow a property of an agent, or a property of a process?

A "Bug Fix" workflow is the same regardless of which agent runs it. Templates belong to the board, not to any individual agent. This makes them portable, shareable, and usable by anyone — including humans without agents.
